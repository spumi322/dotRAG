# 6.2 Garbage Collection

## What the GC does

The Garbage Collector manages allocation and release of **managed heap memory** automatically. It prevents memory leaks for managed objects.

**Key rule:** GC frees managed memory. It **cannot** free unmanaged resources (file handles, DB connections, sockets) — that's your job via `IDisposable`.

---

## GC algorithm — mark-and-sweep + compaction

1. **Mark phase** — GC identifies all reachable objects (live) starting from roots (static fields, local variables, stack references)
2. **Sweep phase** — unreachable objects are reclaimed (dead)
3. **Compact phase** — surviving objects are moved together to eliminate fragmentation (Gen 0/1 only)

---

## Generations

The GC divides the heap into three generations to optimise collection frequency:

| Generation | Contents | Collection frequency | Notes |
|---|---|---|---|
| **Gen 0** | Short-lived objects (locals, temporaries) | Very frequent | Most objects die here |
| **Gen 1** | Buffer — survived Gen 0 | Moderate | Boundary between short-lived and long-lived |
| **Gen 2** | Long-lived objects (statics, caches, singletons) | Infrequent | Full GC — expensive |

Objects that survive a Gen 0 collection are promoted to Gen 1; surviving Gen 1 → Gen 2.

---

## Large Object Heap (LOH)

Objects ≥ 85,000 bytes bypass Gen 0/1 and go directly to the LOH.

- Collected with Gen 2 (full GC)
- **Not compacted by default** — the CLR avoids moving large objects for performance
- Can cause **heap fragmentation** if many LOH allocations/deallocations occur

> You can enable LOH compaction: `GCSettings.LargeObjectHeapCompactionMode = GCLargeObjectHeapCompactionMode.CompactOnce;` — but it's rarely needed.

---

## Managed vs Unmanaged Resources

| | Managed | Unmanaged |
|---|---|---|
| What | CLR-tracked objects | OS-level resources outside CLR control |
| Cleanup | GC handles automatically | You must release explicitly |
| Examples | `string`, `List<T>`, class instances | `FileStream`, `SqlConnection`, `Socket`, `HttpClient` |

## IDisposable — cleaning up unmanaged resources

Implement `IDisposable` on any class that holds unmanaged resources or wraps another `IDisposable`.

```csharp
public class TournamentExporter : IDisposable
{
    private FileStream _file;
    private bool _disposed = false;

    public TournamentExporter(string path)
    {
        _file = new FileStream(path, FileMode.Create);
    }

    public void Dispose()
    {
        Dispose(true);
        GC.SuppressFinalize(this); // skip finalizer — already cleaned up
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;

        if (disposing)
            _file?.Dispose(); // free managed IDisposable objects

        // free raw unmanaged handles here (if any)

        _disposed = true;
    }

    ~TournamentExporter() // finalizer — safety net only
    {
        Dispose(false);
    }
}
```

---

## `using` — always prefer this

`using` compiles to a `try/finally` that calls `Dispose()` automatically — even if an exception is thrown.

```csharp
// Classic block (any C# version)
using (var conn = new SqlConnection(connStr))
{
    // conn.Dispose() called here even if exception thrown
}

// using declaration (C# 8+) — less nesting, same guarantee
using var conn = new SqlConnection(connStr);
using var reader = new StreamReader(filePath);
// both disposed when method exits
```

> **Rule of thumb:** If a class implements `IDisposable`, wrap it in `using`. Common ones: `SqlConnection`, `FileStream`, `StreamReader`, `HttpClient`, `DbContext`.

---

## Dispose vs Finalize

| | `Dispose()` | Finalizer (`~Class`) |
|---|---|---|
| Called by | Developer / `using` | GC automatically |
| Timing | Deterministic | Non-deterministic |
| Performance | Fast | Slow (extra GC cycle) |
| Use for | Primary cleanup path | Safety net only |

**Finalizer penalty:** Objects with a finalizer survive Gen 0 — they're promoted to Gen 1 first, then finalized in a later cycle. This costs an extra GC pass. `GC.SuppressFinalize(this)` in `Dispose()` removes the object from the finalization queue, allowing Gen 0 collection.

**Only implement a finalizer if your class directly owns raw unmanaged handles.** If you're wrapping another `IDisposable`, no finalizer needed.

## GC.Collect() — almost always wrong

```csharp
// ❌ Almost always wrong in production
GC.Collect();

// Forces a full Gen 2 collection — disrupts generational optimisation
// GC is self-tuning; manual collection degrades performance

// ✅ Legitimate uses (rare)
// - Benchmarking: ensure stable baseline before measurements
// - After releasing a massive one-time resource (e.g., unloading a plugin)
```

## GC Modes

| Mode | Default for | Behaviour |
|---|---|---|
| **Workstation GC** | Client apps | Single GC thread, lower latency |
| **Server GC** | ASP.NET Core | One GC thread per CPU core, higher throughput |

```xml
<!-- Enable in .csproj -->
<ServerGarbageCollection>true</ServerGarbageCollection>
```

ASP.NET Core uses Server GC by default when hosted in a server environment.

## Streams and Serialization

**Streams** — always dispose:

```csharp
using var stream = new FileStream("data.json", FileMode.Open);
using var reader = new StreamReader(stream);
var content = await reader.ReadToEndAsync();
```

**Serialization** — JSON matters most:

```csharp
// System.Text.Json (default in ASP.NET Core)
var json = JsonSerializer.Serialize(tournament);
var tournament = JsonSerializer.Deserialize<Tournament>(json);
// In ASP.NET Core controllers — serialization is automatic
```

## Interview Traps

- **"Does the GC free all resources?"** — No. It frees **managed** heap objects. Unmanaged resources (file handles, DB connections) must be freed explicitly via `IDisposable` / `using`.
- **"What is the LOH?"** — Large Object Heap — objects ≥ 85KB go here, collected with Gen 2, not compacted by default. Can cause fragmentation.
- **"When should you call `GC.Collect()`?"** — Almost never in production. GC is self-tuning.
- **"Dispose vs Finalize?"** — `Dispose()` is deterministic, called by developer or `using`. Finalizer is a GC safety net, non-deterministic and slow. Call `GC.SuppressFinalize(this)` in `Dispose()` to avoid the finalizer penalty.
- **"Why does `using` matter?"** — It guarantees `Dispose()` is called even when exceptions are thrown, preventing resource leaks.
