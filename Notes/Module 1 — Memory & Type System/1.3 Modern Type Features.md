# 1.3 Modern Type Features

## Nullable\<T\> — value types that can be null

`Nullable<T>` (shorthand: `T?`) wraps a value type so it can represent the absence of a value. Uses an internal `HasValue` bool + a `Value` field.

```csharp
int? nullableNumber = null;
int? age = 42;

Console.WriteLine(age.HasValue);  // true
Console.WriteLine(age.Value);     // 42

// GetValueOrDefault — safe access
int safe = age.GetValueOrDefault(0); // 42
int fallback = nullableNumber.GetValueOrDefault(0); // 0
```

> `int?` is syntactic sugar for `Nullable<int>` — a value type wrapper with runtime presence.

## Nullable Operators

### Null-conditional (`?.`) — safe navigation

Returns `null` if the operand is null instead of throwing `NullReferenceException`.

```csharp
string? name = null;
int? length = name?.Length; // null (no exception)

// Chain
string? city = user?.Address?.City;
```

### Null-coalescing (`??`) — fallback value

```csharp
string display = userName ?? "Guest";
int value = nullableInt ?? 0;
```

### Null-coalescing assignment (`??=`) — assign only if null

```csharp
items ??= new List<string>(); // assigns only if items is null
```

### Null-forgiving (`!`) — suppress compiler warning

```csharp
string validated = input!; // "I know it's not null" — NO runtime check
```

> Use `!` only when you've already validated nullability through another path. It does nothing at runtime — it only silences the compiler.

## Nullable Reference Types (C# 8+)

NRTs are a **compile-time safety feature** for reference types — entirely different from `Nullable<T>`.

```csharp
#nullable enable

string name = "John";    // non-nullable — compiler warns if you assign null
string? nickname = null; // nullable — explicitly allowed to be null

Console.WriteLine(nickname.Length);  // ⚠️ CS8602 — possible null dereference
Console.WriteLine(nickname?.Length); // ✅ safe
```

**Key points:**
- Enabled by default in .NET 6+ (`<Nullable>enable</Nullable>` in `.csproj`)
- **Compile-time only** — no runtime effect, no boxing, no `Nullable<T>` wrapper
- `string?` is an annotation, not a new type

**Interview trap:** "What's the difference between `int?` and `string?`?"
- `int?` → `Nullable<int>` — a value type wrapper that exists at runtime
- `string?` → an annotation telling the compiler this reference may be null — no runtime difference from `string`

## Records (C# 9+)

Records are types designed for **immutable data** with **value-based equality**. The compiler auto-generates `Equals`, `GetHashCode`, `ToString`, `Deconstruct`, and a copy constructor.

```csharp
// Positional record — reference type, immutable by default
public record ProductDto(int Id, string Name, decimal Price);

var p1 = new ProductDto(1, "Widget", 9.99m);
var p2 = new ProductDto(1, "Widget", 9.99m);

p1 == p2        // true  — value equality (auto-generated)
p1.ToString()   // "ProductDto { Id = 1, Name = Widget, Price = 9.99 }"
```

### `with` expression — non-destructive mutation

Creates a copy with specific properties changed. The original is untouched.

```csharp
var discounted = p1 with { Price = 7.99m };
// ProductDto { Id = 1, Name = Widget, Price = 7.99 }
// p1 is unchanged
```

### Deconstruction

Positional records auto-generate a `Deconstruct` method:

```csharp
var (id, name, price) = p1; // deconstruct into variables
```

### record class vs record struct

| Feature | `record` (class) | `record struct` | `readonly record struct` |
|---|---|---|---|
| Memory | Heap | Stack (when local) | Stack (when local) |
| Immutable by default | ✅ (positional) | ❌ | ✅ |
| `with` expression | ✅ | ✅ | ✅ |
| Inheritance | ✅ | ❌ | ❌ |
| Value equality | ✅ | ✅ | ✅ |

### When to use records

- **DTOs** — API request/response models
- **Value objects** in DDD (e.g., `Money`, `Address`)
- Immutable configuration or event data

### When NOT to use records

- **EF Core entities** — EF relies on reference equality and change tracking; records break this
- **Mutable state** — if properties need to change frequently, use a class

## Tuples (C# 7+)

Lightweight grouping of multiple values without defining a class or struct. Backed by `System.ValueTuple` — a value type.

```csharp
// Named tuple
(int Id, string Name) product = (1, "Widget");
Console.WriteLine(product.Name); // "Widget"

// Return multiple values from a method
(decimal Min, decimal Max) GetPriceRange(List<Product> products)
    => (products.Min(p => p.Price), products.Max(p => p.Price));

var range = GetPriceRange(products);
Console.WriteLine($"{range.Min} - {range.Max}");

// Deconstruction into separate variables
var (min, max) = GetPriceRange(products);
```

> Use tuples for quick multi-value returns in **private/internal** methods. For public APIs, prefer a named type (record, class) — tuples have no semantic name.

## dynamic vs object

### `object`

Every type inherits from `object`. You can assign anything to it, but you must cast back to use it as its actual type. Errors are caught **at compile time**.

```csharp
object obj = 42;           // boxing — int stored as object
int n = (int)obj;          // explicit cast required
int bad = (string)obj;     // ❌ compile error — caught immediately
```

### `dynamic`

Type checking is **skipped at compile time** and happens at runtime. The compiler won't catch mistakes.

```csharp
dynamic x = 1;
x = "now I'm a string"; // no compile error
Console.WriteLine(x.Length); // fine — string has Length

x = 42;
Console.WriteLine(x.Length); // ❌ RuntimeBinderException — int has no Length
```

**When to use `dynamic`:** COM interop (Office APIs), JSON blobs with unknown shape, interop with dynamic languages. In normal web dev code — avoid it.

### Comparison

| | `object` | `dynamic` |
|---|---|---|
| Type checking | Compile-time | Runtime |
| Cast required | ✅ explicit | ❌ none |
| Error timing | Compile | Runtime crash |
| Use case | Store anything safely | Interop / unknown shape |

## Interview Traps

- **"What's the difference between `int?` and `string?`?"** — `int?` is `Nullable<int>` (a runtime wrapper). `string?` is a compile-time annotation only — no runtime difference from `string`.
- **"Are records immutable?"** — Positional `record class` properties are `init`-only by default. You can still define mutable properties manually. `record struct` properties are mutable by default unless you use `readonly record struct`.
- **"Can records be used as EF Core entities?"** — Generally no. EF Core relies on reference equality and change tracking, which records undermine with value-based equality.
- **"Can you store any type in both `object` and `dynamic`?"** — Yes. The difference is when the type is checked: compile time (`object`) vs runtime (`dynamic`).
