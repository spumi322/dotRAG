# 1.1 Memory Architecture

## Stack

Used for static memory allocation. Managed automatically by the runtime (LIFO — Last In, First Out).

**What lives here:**
- Local variables
- Method parameters
- Return addresses / stack frames

**Characteristics:**
- Fast access — contiguous, predictable addresses
- Limited size (OS-defined; excessive recursion → `StackOverflowException`)
- Each thread has its own stack
- Memory is reclaimed automatically when the method returns

## Heap

Used for dynamic memory allocation. Managed by the **Garbage Collector**.

**What lives here:**
- All reference type instances (`class`, `string`, `array`, `List<T>`, etc.)
- Value type fields when they're part of a class (they're embedded in the class object on the heap)

**Characteristics:**
- Larger and more flexible than the stack
- Non-contiguous — accessing it requires pointer dereferencing (slower)
- Survives beyond the scope of a single method call
- GC reclaims unreachable objects

## Memory Layout Diagram

```
          STACK                        HEAP
┌─────────────────────┐      ┌────────────────────────────┐
│  int x = 5          │      │  Tournament { Name="ESL" } │
│  int _id (in Order) │ ───► │  Order { _id = ... }       │
│  ref to tournament  │ ───► │  Tournament { ... }        │
│  ref to order       │ ───► │  Order { ... }             │
└─────────────────────┘      └────────────────────────────┘

Value type as field → lives ON the heap INSIDE its containing object
Value type as local variable → lives ON the stack
```

## Value Types vs Reference Types

### Value types

Store the actual data directly. Copied on assignment and when passed to a method.

Examples: `int`, `double`, `bool`, `decimal`, `char`, `struct`, `enum`

### Reference types

Store a reference (pointer) to where the data lives on the heap. Assignment copies the reference, not the object.

Examples: `class`, `string`, `array`, `List<T>`, all user-defined classes

### Where they live

The type alone does not determine location — **context does**.

```csharp
class Order              // heap-allocated (reference type)
{
    private int _id;     // int lives on the HEAP — it's a field of a class
}

void Process()
{
    int x = 5;               // stack — local variable
    var order = new Order(); // reference on stack, Order object on heap
}
```

### Assignment behaviour

**Value type — creates an independent copy:**

```csharp
int a = 10;
int b = a;  // b is a separate copy
b = 99;
Console.WriteLine(a); // 10 — unaffected
```

**Reference type — both variables point to the same object:**

```csharp
var t1 = new Tournament { Name = "ESL" };
var t2 = t1;        // t2 points to the same object, not a copy
t2.Name = "BLAST";
Console.WriteLine(t1.Name); // "BLAST"
```

### Passing to methods

**Value type — a copy goes in, original is untouched:**

```csharp
void AddTen(int number)
{
    number += 10; // modifies the local copy only
}

int x = 5;
AddTen(x);
Console.WriteLine(x); // still 5
```

**Reference type — the reference is copied, but both point to the same object:**

```csharp
void Rename(Tournament t)
{
    t.Name = "New Name"; // modifies the actual object on the heap
}

var tournament = new Tournament { Name = "Old Name" };
Rename(tournament);
Console.WriteLine(tournament.Name); // "New Name"
```

### Other differences

- Value types inherit from `System.ValueType`, reference types from `System.Object`
- Value types are always **sealed** — you cannot inherit from them
- Reference types are cleaned up by the **Garbage Collector**; value types are reclaimed when they go out of scope

## Boxing and Unboxing

**Boxing** converts a value type to a reference type (wraps it in `System.Object` and places it on the heap).
**Unboxing** extracts the value type back from the heap object.

```csharp
int num = 42;             // value type, lives on stack
object obj = num;         // BOXING → wraps in System.Object, allocates on heap
int unwrapped = (int)obj; // UNBOXING → extracts int from heap object
```

### What happens during boxing

1. Heap allocation — a new object is allocated
2. Value copy — the value type's data is copied into the heap object
3. Reference returned — you now have a reference to the heap object

The value exists in **two places**: original stack + heap copy.

### Performance cost

Boxing is expensive:
- Every box allocates a new heap object → GC pressure
- Unboxing requires runtime type verification
- Value is copied twice

```csharp
// ❌ Performance disaster — 100,000 heap allocations
ArrayList list = new ArrayList();
for (int i = 0; i < 100_000; i++)
    list.Add(i); // each int gets boxed

// ✅ No boxing — List<T> stores int directly
List<int> list = new List<int>();
for (int i = 0; i < 100_000; i++)
    list.Add(i);
```

### Common boxing scenarios

```csharp
// 1. Non-generic collections
ArrayList arr = new ArrayList();
arr.Add(5);           // int → object (BOXING)
int x = (int)arr[0];  // object → int (UNBOXING)

// 2. Passing value type to method expecting object
void Print(object obj) => Console.WriteLine(obj);
Print(10); // BOXING

// 3. Interface assignment on structs
struct Point : IComparable { ... }
IComparable comp = new Point(); // BOXING

// 4. Nullable boxing — CLR boxes the underlying value, not Nullable<T>
int? nullable = 42;
object boxed = nullable;
Console.WriteLine(boxed.GetType()); // System.Int32 (not Nullable<Int32>!)

int? nullValue = null;
object boxedNull = nullValue;
Console.WriteLine(boxedNull == null); // True
```

### Unboxing type mismatch

```csharp
object boxed = 123;
int correct = (int)boxed;      // ✅ Works
double wrong = (double)boxed;  // ❌ InvalidCastException at runtime
```

## Interview Traps

- **"Where are value types stored?"** — On the stack _when local variables_. If they're fields of a class, they're on the heap. Never just say "stack."
- **"Is string a value or reference type?"** — Reference type. But it behaves like a value type because it's immutable — every modification creates a new string object.
- **"What happens when you pass a class to a method and reassign it inside?"** — Reassigning the parameter only changes the local reference. The caller's variable still points to the original object. Mutating properties of the object does affect the caller.
- **"Boxing copies a value type from stack to heap — correct?"** — Imprecise. Boxing **wraps** the value in a `System.Object` on the heap. The framing "stack to heap" misses the key point (allocation + copy into wrapper).
- **"Boxing is automatic so it's fine, right?"** — No. It's a performance cost. Boxing in tight loops is a known bottleneck; always use generic collections instead.
