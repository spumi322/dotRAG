# 10.2 JavaScript & TypeScript

## JavaScript

---

## The Event Loop — single-threaded async

JavaScript runs on **one thread**. It handles async work (HTTP calls, timers, events) without blocking because of the **event loop**.

```
Call stack        — executes synchronous code (LIFO)
Browser/Node API  — handles async operations (setTimeout, fetch, DOM events)
Task queues       — hold callbacks when async work completes
Event loop        — if call stack is empty, moves next item from queue to stack
```

**Two queues, different priorities:**

- **Microtask queue** (higher priority) — `Promise.then/catch/finally`, `queueMicrotask`. Drained completely before any macrotask runs.
- **Macrotask queue** (lower priority) — `setTimeout`, `setInterval`, I/O, UI rendering.

```js
console.log("1");
setTimeout(() => console.log("2"), 0);
Promise.resolve().then(() => console.log("3"));
console.log("4");

// Output: 1, 4, 3, 2
// sync first (1, 4) → microtasks (3) → macrotasks (2)
```

> **Why this matters:** Heavy computation on the call stack freezes the UI. `setTimeout(..., 0)` doesn't run immediately — it queues after the current stack clears and all microtasks drain.

## `var` / `let` / `const`

| | `var` | `let` | `const` |
|---|---|---|---|
| Scope | Function | Block | Block |
| Hoisting | Hoisted, initialized to `undefined` | Hoisted, Temporal Dead Zone | Hoisted, Temporal Dead Zone |
| Reassignable | ✅ | ✅ | ❌ |

**Always use `const` by default, `let` only when reassignment is needed. Never use `var`.**

**Hoisting and the Temporal Dead Zone (TDZ):**

JavaScript processes declarations before running code — this is called **hoisting**.

```js
console.log(x); // undefined (not an error) — var is hoisted + initialized to undefined
var x = 5;

console.log(y); // ReferenceError: Cannot access 'y' before initialization
let y = 5;
```

- `var` → hoisted AND initialized to `undefined`. Readable before declaration, just returns `undefined`.
- `let` / `const` → hoisted BUT **not** initialized. The period from top-of-scope to the declaration line is the **Temporal Dead Zone** — access throws a `ReferenceError`.
- Function declarations → fully hoisted, callable anywhere in scope.
- Function expressions (`const fn = () => {}`) → not hoisted; same rules as `const`.

> "TDZ is why `let` and `const` can't be used before their declaration even though they are technically hoisted."

**Classic `var` closure trap:**

```js
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 0);
}
// Output: 3, 3, 3 — not 0, 1, 2
// `var` is function-scoped, all callbacks share the same `i`

// Fix: use `let` — creates a new binding per iteration
for (let i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 0);
}
// Output: 0, 1, 2
```

## `==` vs `===`

`==` checks equality with **type coercion** — JavaScript converts types before comparing. `===` checks value AND type, no conversion.

```js
0 == "0"           // true  (coercion)
0 === "0"          // false (different types)
null == undefined  // true  (special case)
null === undefined // false
```

**Always use `===`.** `==` is a source of subtle bugs and an interview trap.

## Closures

A closure is a function that retains access to variables from the scope where it was created, even after that scope has finished executing.

```js
function makeCounter() {
    let count = 0;
    return function() {
        count++;
        return count;
    };
}
const counter = makeCounter();
counter(); // 1
counter(); // 2 — `count` is kept alive by the closure
```

Used for: data privacy, factory functions, callbacks that access surrounding context.

## `this` keyword

`this` refers to the **execution context** — its value depends on how the function is called, not where it's defined.

```js
const user = {
    name: "Alex",
    greet() { console.log(this.name); }
};
user.greet();           // "Alex" — method call: this = user

const fn = user.greet;
fn();                   // undefined (strict) / global — lost context when detached
```

**Arrow functions have no own `this`** — they inherit `this` lexically from the enclosing scope. This is why arrow functions are used for class methods passed as callbacks:

```js
class Timer {
    count = 0;
    start() {
        setInterval(() => {
            this.count++; // ✅ `this` is the Timer instance (arrow captures enclosing `this`)
        }, 1000);
    }
}
```

Using a regular function in `setInterval` would lose `this`.

**`call` / `apply` / `bind`** — explicitly control `this`:

```js
function introduce(greeting) {
    console.log(`${greeting}, I'm ${this.name}`);
}

const bob = { name: "Bob" };

introduce.call(bob, "Hi");          // "Hi, I'm Bob" — args spread
introduce.apply(bob, ["Hello"]);    // "Hello, I'm Bob" — args as array
const boundFn = introduce.bind(bob);
boundFn("Hey");                     // "Hey, I'm Bob" — returns new bound function
```

> "`call` and `apply` invoke immediately. `bind` returns a new function with `this` permanently locked."

## Prototypes

JavaScript has no classes like C#. The `class` keyword (ES6) is **syntactic sugar over prototypes**. Every object has a hidden `[[Prototype]]` link forming a chain — when a property isn't found on an object, JS walks up this chain.

`extends` wires up the prototype chain. Saying "JS has classes like C#" is wrong — the mechanism is prototype-based.

---

## Event Bubbling

When an event fires on an element, it first triggers handlers on the target, then **bubbles up** through ancestors to the root.

```
Click on <button> inside <div>:
  button handler fires → div handler fires → body fires → document fires
```

`event.stopPropagation()` stops the bubble. **Event delegation** — attach one handler to a parent instead of many handlers to children. Efficient for dynamic lists.

## Promises and async/await

A Promise represents the eventual result of an async operation. Three states: **pending → fulfilled or rejected**. Once settled, it cannot change.

`async/await` is syntactic sugar over Promises — `await` suspends the function (not the thread) until the Promise settles.

```js
// Equivalent
fetch("/api/tournaments").then(r => r.json()).then(data => console.log(data));

async function load() {
    const data = await fetch("/api/tournaments").then(r => r.json());
    console.log(data);
}
```

**Key combinators:**

| Method | Resolves when | Rejects when |
|---|---|---|
| `Promise.all([p1, p2])` | ALL resolve | ANY rejects |
| `Promise.allSettled([p1, p2])` | ALL settle (never rejects) | — |
| `Promise.race([p1, p2])` | First settles (either way) | First settles rejected |

> **`fetch` trap:** `fetch` does NOT reject on HTTP 404/500 — only on network errors. Always check `response.ok` before processing.

**Error handling with async/await:**

```js
async function loadUser(id) {
    try {
        const res = await fetch(`/api/users/${id}`);
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return await res.json();
    } catch (err) {
        console.error("Failed:", err.message);
    }
}
```

Common mistake — forgetting `await` inside `try/catch`:
```js
try {
    fetch("/api/data"); // ❌ no await — Promise is not awaited, catch never fires
} catch (e) { }
```

For independent parallel calls with partial failure tolerance: `Promise.allSettled` over `Promise.all` + try/catch.

## `Map` vs `Object` / `Set`

`Object` is JavaScript's default key-value store, but it has limitations:

```js
const obj = {};
obj[1]    = "one";   // key coerced to string "1"
obj[true] = "yes";   // key coerced to string "true"
```

`Map` is a purpose-built key-value collection:

| | `Object` | `Map` |
|---|---|---|
| Key types | Strings + Symbols only | Any value (objects, functions, primitives) |
| Size | `Object.keys(obj).length` | `map.size` |
| Iteration order | Not fully guaranteed for non-string keys | Insertion order, always |
| Prototype pollution risk | ✅ (inherits from `Object.prototype`) | None |

```js
const map = new Map();
map.set("name", "Summer Cup");
map.set(42, "answer");
console.log(map.size); // 2
```

**Use `Map`** for dynamic key-value stores where keys aren't known ahead of time.
**Use `Object`** for structured records with known, named properties.

**`Set`** — collection of unique values, O(1) lookup:

```js
const ids = new Set([1, 2, 2, 3]);
console.log([...ids]); // [1, 2, 3] — duplicates removed
ids.has(2);            // true
```

Use `Set` for deduplication or membership tests instead of `.indexOf()` on arrays.

## What TypeScript is

TypeScript is a **statically-typed superset of JavaScript**. All valid JS is valid TS. The type system is checked at **compile time** and completely **erased at runtime** — zero performance overhead.

---

## Structural typing

C# uses **nominal typing**: two types are compatible only if they explicitly declare a relationship. TypeScript uses **structural typing**: compatibility is based on **shape** — same properties and types — regardless of name.

```ts
interface Cat   { name: string; meow(): void; }
interface Robot { name: string; meow(): void; }

const r: Robot = { name: "RoboCat", meow() {} };
const c: Cat = r; // ✅ Same shape → compatible, even though unrelated types
```

---

## `any` vs `unknown` vs `never`

| Type | Accepts | Must narrow before use |
|---|---|---|
| `any` | Anything | ❌ No — disables type checking |
| `unknown` | Anything | ✅ Yes — the safe alternative to `any` |
| `never` | Nothing | N/A — represents impossible values |

```ts
// unknown forces narrowing
function process(value: unknown) {
    if (typeof value === "string") {
        console.log(value.toUpperCase()); // safe — narrowed to string
    }
}

// never — exhaustive switch check
type Shape = "circle" | "square";
function area(s: Shape): number {
    switch (s) {
        case "circle": return Math.PI;
        case "square": return 1;
        default: const _: never = s; // compile error if new shape added without handling
    }
}
```

## Interface vs Type alias

```ts
// Interface — objects and class contracts
interface Tournament { id: number; name: string; }
interface Tournament { status: string; } // declaration merging — auto-merged

// Type alias — unions, intersections, primitives, tuples
type Id = number | string;
type Result<T> = { data: T; error: null } | { data: null; error: string };
```

**Intersection types** — combine multiple types into one. The value must satisfy all of them:

```ts
type Timestamps = { createdAt: Date; updatedAt: Date };
type Tournament  = { id: number; name: string };
type TournamentRecord = Tournament & Timestamps; // must have all four properties
```

Union = "one of these shapes". Intersection = "all of these shapes combined."

**Rule of thumb:** `interface` for objects/APIs. `type` for everything else (unions, intersections, primitives, tuples).

## Type narrowing (type guards)

TypeScript narrows broad types to specific ones through control flow:

```ts
// typeof
if (typeof value === "string") { /* narrowed to string */ }

// instanceof
if (error instanceof NotFoundException) { /* narrowed */ }

// in operator
if ("name" in obj) { /* obj has name property */ }

// Custom type predicate
function isTournament(x: Tournament | Match): x is Tournament {
    return "teams" in x;
}
```

## Enums

Enums create a named set of constants:

```ts
// Numeric enum — values default to 0, 1, 2...
enum Direction { Up, Down, Left, Right }
Direction.Up // 0

// String enum — explicit values, no reverse lookup surprises
enum Status {
    Pending  = "PENDING",
    Active   = "ACTIVE",
    Archived = "ARCHIVED",
}
```

**String enums are safer** — numeric enums allow reverse lookup (`Direction[0]` returns `"Up"`) which can surprise you. String enums are also readable in JSON/logs.

**`const enum`** — inlined at compile time; no runtime object generated. Smallest output, but can't be iterated or accessed dynamically.

```ts
const enum Color { Red, Green, Blue }
let c = Color.Red; // compiled to: let c = 0;
```

## `readonly` and tuples

**`readonly`** — prevents property mutation after construction:

```ts
interface Config {
    readonly apiUrl: string;
    readonly timeout: number;
}

const cfg: Config = { apiUrl: "/api", timeout: 5000 };
cfg.apiUrl = "/other"; // ❌ compile error

// Utility type version:
type FrozenConfig = Readonly<Config>; // all props readonly
```

Note: `readonly` is shallow — nested objects can still be mutated.

**Tuples** — fixed-length arrays where each position has a known type:

```ts
type Point = [number, number];
type Result<T> = [T, null] | [null, Error]; // Result-pattern without throwing

const [x, y]: Point = [10, 20];

// Practical: React useState-style
function useState<T>(init: T): [T, (v: T) => void] { ... }
const [count, setCount] = useState(0);
```

## `tsconfig.json`

The central config file for any TypeScript project. Marks the directory as a TS project root and controls compilation.

```json
{
  "compilerOptions": {
    "strict": true,            // enables strictNullChecks, noImplicitAny, and more
    "target": "ES2020",        // output JS version
    "moduleResolution": "Bundler", // how imports are resolved (Node, Bundler, NodeNext)
    "paths": {
      "@app/*": ["./src/app/*"] // import aliases — avoids ../../../
    }
  }
}
```

**Key settings:**
- `"strict": true` — the most important setting. Enables a bundle of strict checks including `strictNullChecks` (null/undefined aren't assignable to other types) and `noImplicitAny` (variables can't silently be `any`). Always enable this.
- `"target"` — what JS version is emitted. Angular projects typically use `ES2020` or `ESNext`.
- `"moduleResolution"` — affects how `import` paths are resolved. Angular uses `Bundler`.

In Angular: `tsconfig.json` is the base; `tsconfig.app.json` extends it for app code, `tsconfig.spec.json` for tests.

## Utility types

Built-in generic types for common transformations:

```ts
Partial<T>       // all properties optional — useful for PATCH DTOs
Required<T>      // all properties required
Readonly<T>      // all properties readonly
Pick<T, K>       // keep only specified keys
Omit<T, K>       // remove specified keys

interface Tournament { id: number; name: string; status: string; }
type TournamentUpdate  = Partial<Tournament>;           // all optional
type TournamentSummary = Pick<Tournament, "id"|"name">; // subset
type CreateInput       = Omit<Tournament, "id">;        // no id (server assigns)
```

## Interview Traps

- **"What is the event loop?"** — Single-threaded. Call stack runs synchronous code. Async callbacks go to queues. Microtask queue (Promises) drains before macrotask queue (setTimeout). Event loop moves items from queue to stack when stack is empty.
- **"`var` loop trap?"** — `var` is function-scoped; all loop callbacks close over the same variable. Output is the final value for all callbacks. `let` creates a new binding per iteration.
- **"What is hoisting? What's the TDZ?"** — Declarations are processed before code runs. `var` is hoisted and initialized to `undefined` — readable before declaration. `let`/`const` are hoisted but not initialized — accessing them before declaration throws a `ReferenceError`. That window is the Temporal Dead Zone.
- **"Explain closures."** — Inner function retains access to outer scope after outer function returns. Variables are kept alive as long as something references them.
- **"`this` in arrow function vs regular function?"** — Regular functions: `this` depends on the call site (lost when detached from an object). Arrow functions: `this` is inherited lexically — it's always the `this` of the enclosing scope at the time of definition. Arrow functions are used in class callbacks for exactly this reason.
- **"Structural vs nominal typing?"** — TypeScript checks shape compatibility (same properties/types). C# checks declared type hierarchy. Two unrelated TS types with the same shape are compatible.
- **"`any` vs `unknown`?"** — Both accept any value. `unknown` forces you to narrow the type before using it — it's type-safe. `any` opts out of type checking entirely.
- **"`Promise.all` vs `Promise.allSettled`?"** — `all` rejects immediately if any promise rejects. `allSettled` always waits for all and never rejects — gives you a result for each promise.
- **"When would you use `Map` over `Object`?"** — When keys aren't strings/symbols, when you need guaranteed insertion-order iteration, or when key count is dynamic. `Object` for named, known properties. `Set` when you just need unique values.
- **"What does `tsconfig` `strict: true` enable?"** — A bundle including `strictNullChecks` (null/undefined are separate types) and `noImplicitAny` (implicit any is a compile error). Always enable it.
- **"String enum vs numeric enum?"** — String enums are explicit and readable in logs/JSON. Numeric enums have reverse lookup which can cause surprises. Prefer string enums for anything user-facing.
