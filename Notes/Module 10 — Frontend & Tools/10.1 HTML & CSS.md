# 10.1 HTML & CSS

## HTML — Structure and Meaning

HTML defines the **structure and meaning** of web content. It is not a programming language — it's a markup language. HTML tells the browser _what_ the content is, CSS tells it _how it looks_, JavaScript tells it _how it behaves_. That separation of concerns is the first thing interviewers test.

---

## Semantic HTML

Semantic elements describe _what_ the content is. Non-semantic elements describe nothing.

```
<nav>        → navigation
<main>       → primary content of the page
<article>    → self-contained piece of content
<section>    → thematic group of related content
<aside>      → supplementary content (sidebar)
<header>     → header for a page or section
<footer>     → footer for a page or section

<div>        → nothing — generic block container
<span>       → nothing — generic inline wrapper
```

**Why semantics matter:**

1. **Accessibility.** Screen readers navigate by landmarks. `<nav>` announces itself as navigation — users can jump straight to it. `<div class="nav">` is invisible to them. A `<button>` is keyboard-focusable and responds to Enter/Space automatically; a `<div onclick>` requires manual `tabindex`, keyboard listeners, and ARIA — and you'll still miss something.

2. **SEO.** Search engines weight content inside semantic elements (`<h1>`, `<article>`, `<main>`) more than content in anonymous `<div>`s.

3. **Maintainability.** `<nav>` is self-documenting. `<div class="nav-wrapper-outer-container">` is not.

> "Use the native element."

---

## The DOM

When the browser receives HTML, it parses it and builds a **DOM (Document Object Model)** — an in-memory tree of objects. JavaScript never touches the raw HTML file; it reads and modifies the page through the DOM.

```js
document.getElementById("title").textContent = "Hello"; // modifies a DOM node
```

When React, Angular, or any framework "updates the UI", it modifies DOM nodes. The browser re-renders whatever changed.

**Why DOM size matters:** every DOM node costs memory. Deep nesting makes traversal slower.

---

## Script loading: `defer` vs `async`

By default a `<script>` tag stops HTML parsing, downloads the script, executes it, then resumes. This blocks rendering.

```
Normal:  HTML parsing → STOP → download → execute → resume parsing
async:   HTML parsing continues while downloading → execute immediately when ready
defer:   HTML parsing continues while downloading → execute AFTER full parse, in order
```

| | `async` | `defer` |
|---|---|---|
| Blocks parsing? | Only during execution | Never |
| Execution order | First to finish | Document order |
| Use for | Independent 3rd-party scripts | App scripts |

---

## SSR vs CSR

**Client-Side Rendering (CSR)** — server sends a minimal HTML shell (`<div id="app"></div>`) and a JS bundle. JavaScript builds all the HTML in the browser.
- ✅ Fast navigation after initial load, rich interactivity
- ❌ Slow initial load (blank page until JS runs), poor SEO by default

**Server-Side Rendering (SSR)** — server generates complete, populated HTML for every request.
- ✅ Fast first load (content visible immediately), great SEO
- ❌ Every navigation is a full page reload, more server load

Modern frameworks (Next.js, Nuxt, Angular Universal) combine both: SSR the first page load, CSR for navigation. This is called **hydration**.

---

## Accessibility

Two rules that cover 80% of what's asked:

1. **Use native interactive elements.** Always use `<button>` for clickable actions, `<a>` for navigation, `<input>` for form fields. Never use `<div onclick>` to fake interactive behavior — native elements are keyboard-focusable and announce themselves to screen readers for free.

2. **Every image needs `alt`.** Descriptive images: `alt="Tournament bracket showing semifinals"`. Decorative images: `alt=""` (empty string — screen readers skip it). Missing `alt` entirely causes the screen reader to read the filename.

## Specificity

When two CSS rules target the same element, the browser picks the winner by **specificity score**:

```
Inline style   (style="...")      — highest priority
ID             (#id)              — very strong
Class          (.class), :hover   — medium
Element        (p, div)           — weakest
```

`!important` overrides everything — avoid it. It breaks the predictable resolution chain and makes debugging a nightmare.

> "If a style isn't applying, check whether a more specific selector is overriding it."

## Box model

Every element is a box with four layers (inside out):

```
+-------------------------------+  ← margin (transparent, pushes others away)
|  +-------------------------+  |
|  |  border                 |  |
|  |  +-------------------+  |  |
|  |  |  padding          |  |  |
|  |  |  +-------------+  |  |  |
|  |  |  |   content   |  |  |  |
|  |  |  +-------------+  |  |  |
|  |  +-------------------+  |  |
|  +-------------------------+  |
+-------------------------------+
```

**Content** — actual text or image. **Padding** — space inside the border (inherits background color). **Border** — visible edge. **Margin** — transparent space outside (pushes other elements away).

**The problem:** default `box-sizing: content-box` adds padding and border on top of the declared `width`. A `width: 200px` element with `padding: 20px` and `border: 1px` is actually 242px wide.

**The fix every modern project uses:**
```css
*, *::before, *::after {
    box-sizing: border-box;
}
```

Now `width: 200px` includes padding and border. Predictable. Angular projects apply this by default.

> "Padding and margin difference: padding is inside the border (part of the element, has background). Margin is outside — transparent, pushes others away. Vertical margins between elements **collapse** — the larger one wins, they don't add up."

## Flexbox vs Grid

Two modern layout systems. Interviewers ask you to choose between them.

**Flexbox** — **1D layout** (row or column). Content decides how much space each item takes.

```css
.navbar {
    display: flex;
    justify-content: space-between; /* spread items */
    align-items: center;            /* vertically center */
}
```

**Grid** — **2D layout** (rows AND columns). You define the structure; content fills it.

```css
.card-grid {
    display: grid;
    grid-template-columns: repeat(3, 1fr); /* 3 equal columns */
    gap: 20px;
}
```

> **Mental model:** Flex = content decides layout. Grid = you decide layout, content fills it.

They compose: use Grid for page structure, Flex inside each card to align content.

## Positioning and stacking contexts

| Value | Flow | Relative to |
|---|---|---|
| `relative` | Stays in flow | Normal position — reference for absolute children |
| `absolute` | Removed from flow | Nearest positioned ancestor |
| `fixed` | Removed from flow | Viewport (doesn't scroll) |
| `sticky` | In flow until threshold | Viewport after threshold |

**Why `z-index: 9999` sometimes doesn't work:**

`z-index` only stacks within the same **stacking context**. Things that create a new context: `position` + any `z-index`, `opacity < 1`, `transform`, `filter`.

If a parent has `opacity: 0.99`, it creates a stacking context — your modal inside it can't stack above elements outside that context, regardless of z-index.

## Responsive design

Media queries apply styles conditionally based on screen properties:

```css
/* Mobile-first: base styles for small screens */
.card { width: 100%; }

/* Add complexity for larger screens */
@media (min-width: 768px) {
    .card { width: 50%; }
}
```

**Mobile-first** = write base styles for small screens, add complexity with `min-width` queries. This is the current standard.

## CSS custom properties (variables)

```css
:root {
    --primary: #4caf50;
    --bg-dark: #1a1a2e;
}

.card {
    background: var(--bg-dark);
    border-color: var(--primary);
}
```

Unlike Sass variables (compiled away), CSS variables exist at runtime — JavaScript can read and write them. This is how dark mode works: override the variables on a `.dark-theme` class and the whole UI updates.

## Interview Traps

- **"Semantic HTML — why use it?"** — Accessibility (screen readers navigate landmarks, native buttons are keyboard-friendly), SEO, maintainability.
- **"What is the DOM?"** — Browser's in-memory tree of the HTML document. JavaScript interacts with the page through it — never the HTML file directly.
- **"Box model?"** — Content + padding + border + margin. Use `box-sizing: border-box` so declared width includes padding and border.
- **"Flexbox vs Grid?"** — Flex is 1D (row or column), content-driven. Grid is 2D (rows + columns), structure-driven. Use flex for component internals, grid for page layout.
- **"Why doesn't `z-index: 9999` work?"** — `z-index` stacks within the same stacking context. A parent with `opacity`, `transform`, or `position + z-index` creates an isolated context — children can't escape it.
- **"`display: none` vs `visibility: hidden`?"** — `none` removes from layout (takes no space). `hidden` is invisible but still occupies space.
- **"`em` vs `rem`?"** — `rem` is relative to the root element — predictable. `em` compounds through nesting — causes bugs. Default to `rem`.
