# 9.2 Integration Testing

## What integration testing tests

Integration tests verify that multiple components work correctly together — across real HTTP, real DI container, real middleware, and optionally a real database. They catch bugs that unit tests miss: routing misconfigurations, missing middleware, incorrect DI registrations.

```
Unit test:    [Service] ← mock repo, mock clock
Integration:  HTTP client → [Controller → Service → DbContext → Test DB]
```

---

## WebApplicationFactory\<T\>

`WebApplicationFactory<T>` spins up your full ASP.NET Core application in-process for tests. No real network port is opened — it uses a `HttpClient` backed by an in-memory transport.

```csharp
// NuGet: Microsoft.AspNetCore.Mvc.Testing

public class TournamentsApiTests : IClassFixture<WebApplicationFactory<Program>>
{
    private readonly HttpClient _client;

    public TournamentsApiTests(WebApplicationFactory<Program> factory)
    {
        _client = factory.CreateClient();
    }

    [Fact]
    public async Task GetTournament_Exists_Returns200WithData()
    {
        // Act
        var response = await _client.GetAsync("/api/tournaments/1");

        // Assert
        response.EnsureSuccessStatusCode();                             // 2xx
        var json = await response.Content.ReadAsStringAsync();
        var tournament = JsonSerializer.Deserialize<TournamentDto>(json);
        Assert.Equal("Spring Cup", tournament!.Name);
    }
}
```

> `IClassFixture<T>` shares one factory instance across all tests in the class. `IAsyncLifetime` lets you seed data before each test and clean up after.

### Customising the factory

Override services for testing — replace real DB with test DB, swap out external HTTP clients:

```csharp
var factory = new WebApplicationFactory<Program>()
    .WithWebHostBuilder(builder =>
    {
        builder.ConfigureServices(services =>
        {
            // Remove real DbContext registration
            var descriptor = services.Single(d => d.ServiceType == typeof(DbContextOptions<AppDbContext>));
            services.Remove(descriptor);

            // Add test database (SQLite in-memory or SQL Server LocalDB)
            services.AddDbContext<AppDbContext>(o =>
                o.UseInMemoryDatabase("TestDb"));

            // Replace external HTTP client with a fake
            services.AddSingleton<IEmailService, FakeEmailService>();
        });
    });
```

## In-memory vs real DB for tests

| Approach | Speed | Fidelity | SQL features |
|---|---|---|---|
| **EF In-Memory provider** | ✅ Very fast | ❌ Low (no SQL, no constraints) | ❌ None |
| **SQLite (in-memory mode)** | ✅ Fast | ✅ Medium (SQL, FKs, most queries) | ⚠️ SQL Server syntax may differ |
| **SQL Server LocalDB / TestContainers** | ⚠️ Slower | ✅ High (production-equivalent) | ✅ Full |

**Recommendation for most projects:**
- Unit tests → mock everything, no DB
- Integration tests → SQLite in-memory (fast, real SQL)
- Full fidelity critical paths → TestContainers (real SQL Server in Docker)

```csharp
// SQLite in-memory for integration tests
services.AddDbContext<AppDbContext>(o =>
    o.UseSqlite("DataSource=:memory:"));
```

> **EF In-Memory provider pitfall:** It doesn't enforce FK constraints, doesn't run triggers, and handles certain queries differently from SQL Server. Use SQLite or TestContainers for anything involving real query translation.

## CQRS pattern

CQRS (Command Query Responsibility Segregation) separates read operations (queries) from write operations (commands). Common in .NET shops.

```
Command — changes state, no return value (or just an ID)
  e.g., RegisterTeamCommand, CancelTournamentCommand

Query — reads state, no side effects
  e.g., GetTournamentQuery, ListTeamsQuery
```

**Why:** Read models can be optimized differently from write models. Queries can use projections/AsNoTracking; commands go through full validation and domain logic.

```csharp
// Command
public record RegisterTeamCommand(string Name, string City) : IRequest<TeamDto>;

// Handler
public class RegisterTeamHandler : IRequestHandler<RegisterTeamCommand, TeamDto>
{
    public async Task<TeamDto> Handle(RegisterTeamCommand cmd, CancellationToken ct)
    {
        var team = new Team { Name = cmd.Name, City = cmd.City };
        _repo.Add(team);
        await _unitOfWork.SaveChangesAsync(ct);
        return new TeamDto(team.Id, team.Name);
    }
}
```

## MediatR

MediatR is the most common CQRS library in .NET. It acts as an in-process mediator — controllers send commands/queries to handlers without direct dependencies between them.

```csharp
// NuGet: MediatR

// Registration
builder.Services.AddMediatR(cfg => cfg.RegisterServicesFromAssembly(typeof(Program).Assembly));

// Controller — no service dependency, only IMediator
[ApiController]
[Route("api/teams")]
public class TeamsController(IMediator mediator) : ControllerBase
{
    [HttpPost]
    public async Task<IActionResult> Register(
        RegisterTeamCommand cmd,
        CancellationToken ct)
    {
        var result = await mediator.Send(cmd, ct);
        return CreatedAtAction(nameof(Get), new { id = result.Id }, result);
    }

    [HttpGet("{id}")]
    public async Task<IActionResult> Get(int id, CancellationToken ct)
    {
        var result = await mediator.Send(new GetTeamQuery(id), ct);
        return result is null ? NotFound() : Ok(result);
    }
}
```

**Pipeline behaviours** — MediatR supports middleware-like pipeline behaviours that wrap all handlers:

```csharp
// Validation behaviour — runs before every handler
public class ValidationBehaviour<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : IRequest<TResponse>
{
    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken ct)
    {
        // validate request (FluentValidation, etc.)
        return await next(); // continue to handler
    }
}
```

> **Why MediatR?** It decouples controllers from service classes, makes adding cross-cutting concerns (logging, validation, caching) easy via pipeline behaviours, and aligns naturally with CQRS.

## Interview Traps

- **"Unit vs Integration test — what does each catch?"** — Unit tests catch logic bugs in isolation; integration tests catch wiring bugs (wrong routing, missing DI registrations, incorrect middleware order, DB query translation errors).
- **"What is WebApplicationFactory?"** — An in-process test host that starts your full ASP.NET Core app without a real network port. Provides an `HttpClient` backed by in-memory transport for fast, realistic endpoint tests.
- **"In-memory DB vs real DB in tests?"** — EF in-memory provider is fast but doesn't enforce constraints or translate all queries correctly. SQLite is a better default for integration tests. TestContainers gives full SQL Server fidelity.
- **"What is CQRS?"** — Command Query Responsibility Segregation: separates write operations (commands, change state) from read operations (queries, return data). Allows read and write models to be optimized independently.
- **"What is MediatR?"** — An in-process mediator. Controllers send commands/queries; MediatR routes them to the right handler. Pipeline behaviours add cross-cutting concerns (validation, logging) without touching handlers.
