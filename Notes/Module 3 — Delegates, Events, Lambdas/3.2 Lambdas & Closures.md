# 3.2 Lambdas & Closures

## Lambda expressions

A lambda is a shorthand for an anonymous method. Syntax: `parameters => body`.

```csharp
// Expression lambda (single expression, no braces)
Func<int, int> square = x => x * x;
square(5); // 25

// Statement lambda (block body)
Func<int, int, int> add = (a, b) =>
{
    int sum = a + b;
    return sum;
};

// No parameters
Action greet = () => Console.WriteLine("Hello");

// Multiple parameters
Func<string, string, string> combine = (a, b) => a + " " + b;
```

### Lambdas in LINQ

```csharp
var activeUsers = users
    .Where(u => u.IsActive)
    .OrderBy(u => u.Name)
    .Select(u => new UserDto(u.Id, u.Name));
```

### Passing lambdas as arguments

```csharp
public void Process(IEnumerable<Order> orders, Func<Order, bool> filter)
{
    var filtered = orders.Where(filter);
    // ...
}

// Call site
Process(orders, o => o.Total > 1000);
```

## Closures

A closure is a lambda (or anonymous method) that **captures variables from its enclosing scope**. The delegate holds a reference to the outer variable — not a copy of its value at the time the lambda was created.

```csharp
int multiplier = 3;
Func<int, int> multiply = x => x * multiplier;

Console.WriteLine(multiply(5));  // 15

multiplier = 10;
Console.WriteLine(multiply(5));  // 50 — uses the CURRENT value, not the original
```

The variable is shared between the outer scope and the lambda.

---

## The variable capture trap

The most common closure pitfall: capturing a loop variable. The lambda captures the **variable itself**, not its value at each iteration.

```csharp
// ❌ Classic trap — all lambdas capture the SAME i variable
var actions = new List<Action>();
for (int i = 0; i < 3; i++)
{
    actions.Add(() => Console.WriteLine(i));
}

foreach (var action in actions)
    action(); // Prints: 3, 3, 3 — not 0, 1, 2
// When the lambdas run, i = 3 (the final value after the loop)
```

**Fix: capture a local copy**

```csharp
// ✅ Each iteration captures its own local variable
var actions = new List<Action>();
for (int i = 0; i < 3; i++)
{
    int captured = i;       // new variable per iteration
    actions.Add(() => Console.WriteLine(captured));
}

foreach (var action in actions)
    action(); // Prints: 0, 1, 2
```

> **`foreach` in modern C#:** Since C# 5, `foreach` creates a fresh loop variable per iteration, so the trap does not apply to `foreach`. It only applies to `for` loops with a shared counter variable.

---

## What the compiler does

When a lambda captures an outer variable, the compiler generates a **hidden closure class** that holds the captured variable as a field. The lambda becomes a method on that class.

```csharp
// Your code
int x = 5;
Func<int> get = () => x;

// What the compiler generates (conceptually)
class ClosureClass
{
    public int x;
    public int get() => x;
}

var closure = new ClosureClass { x = 5 };
Func<int> get = closure.get;
```

This is why closures can modify the captured variable — both the outer scope and the lambda share the same field on the closure object.

---

## Closures and mutation

Closures can both **read** and **modify** captured variables:

```csharp
int count = 0;
Action increment = () => count++;
Action reset = () => count = 0;

increment(); // count = 1
increment(); // count = 2
Console.WriteLine(count); // 2
reset();
Console.WriteLine(count); // 0
```

This is useful for callbacks and stateful delegates, but can lead to unexpected shared state bugs in concurrent code.

---

## Interview Traps

- **"What does a closure capture — the value or the variable?"** — The variable (by reference). Mutations to the outer variable after the lambda is defined are visible inside the lambda.
- **"What does this print: `for (int i = 0; i < 3; i++) { actions.Add(() => Console.WriteLine(i)); }`?"** — `3, 3, 3`. All three lambdas share the same `i`, which is 3 after the loop ends. Fix by copying `i` to a local variable inside the loop body.
- **"Does `foreach` have the same problem?"** — Not in C# 5+. The compiler creates a new copy of the iteration variable per iteration in `foreach`. Safe to capture.
- **"Why can a lambda modify an outer variable?"** — Because the compiler wraps the captured variable in a hidden class instance shared by both the outer scope and the lambda. It's the same field, not a copy.
