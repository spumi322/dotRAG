# 10.3 Angular

---

## What Angular is

A frontend framework that keeps your data and your UI in sync automatically. You change a value in code — the screen updates. The user clicks something — the code updates. That's the core contract.

It's a **Single Page Application** framework — the browser loads once, Angular handles all navigation internally without hitting the server again.

---

## Components

A component is a self-contained piece of UI. It owns its data, its logic, and its template (the HTML it renders).

**Why:** breaking the UI into components means each piece is isolated, reusable, and testable — same reason you break C# logic into classes instead of one giant method.

**Where:** everything visible is a component. Nav bar, form, data table — each is its own component, composed together to build the full page.

```typescript
// tournament-header.component.ts
@Component({
  selector: 'app-tournament-header',
  templateUrl: './tournament-header.component.html',
  styleUrls: ['./tournament-header.component.css']
})
export class TournamentHeaderComponent { ... }
```

Used in a parent template like any HTML element:
```html
<app-tournament-header [tournament]="tournament" (startGroups)="onStartGroups()">
</app-tournament-header>
```

---

## Parent and Child Components

A parent component contains child components in its template — a tournament page might hold a standings table, a fixtures list, and a bracket as separate children. This is **not** inheritance. There's no `Child : Parent`, no shared behaviour, no overriding. It's composition: one component wraps others.

**Parent → Child (`@Input`)** — parent passes data down into the child. The child declares what it expects, the parent supplies it. Think of it as a method argument.

```typescript
// tournament-header.component.ts
export class TournamentHeaderComponent {
  @Input() tournament: Tournament | null = null;
  @Input() tournamentState: TournamentStateDTO | null = null;
  @Input() isReloading: boolean = false;
}
```

```typescript
// matches.component.ts — typed collection, required field
export class MatchesComponent {
  @Input() matches: Match[] = [];
  @Input() teams: Team[] = [];
  @Input() isGroupFinished: boolean = false;
  @Input() tournamentId!: number;  // ! = required, no default
}
```

**Child → Parent (`@Output` + `EventEmitter`)** — child raises an event, parent listens and reacts. The child never calls the parent directly — same pattern and same reason as a C# event. The child stays reusable because it has no knowledge of what's above it.

```typescript
// tournament-header.component.ts
@Output() startGroups   = new EventEmitter<void>();
@Output() startTournament = new EventEmitter<void>();
@Output() startBracket  = new EventEmitter<void>();

onStartGroups(): void {
  this.startGroups.emit();  // fires event upward, parent reacts
}
```

```typescript
// matches.component.ts — typed payload in the event
@Output() matchFinished = new EventEmitter<MatchFinishedIds>();

// inside updateMatchScore():
this.matchFinished.emit(matchFinishedData);
```

Parent template wires up the event:
```html
<!-- tournament-header.component.html -->
<p-button (onClick)="onStartGroups()" label="Start Groups"
          [disabled]="!tournament || isReloading">
</p-button>
```

**Siblings / distant components** have no template relationship, so `@Input`/`@Output` don't apply. Shared state goes into a service instead — which is partly why services exist.

## Data Binding

How component data gets connected to the template. Four modes, all variations of the same idea:

|Syntax|Direction|Use for|
|---|---|---|
|`{{ value }}`|Component → screen|Displaying a value|
|`[property]="value"`|Component → screen|Setting an element attribute|
|`(event)="method()"`|Screen → component|Reacting to user input|
|`[(ngModel)]="value"`|Both ways|Form inputs|

**Why it matters in interviews:** know the difference between one-way (predictable, single direction) and two-way (convenient but harder to trace where a change came from).

```html
<!-- matches.component.html -->

<!-- {{ }} interpolation -->
{{ getTeamName(match.teamAId) }}
{{ match.teamAWins }} - {{ match.teamBWins }}

<!-- [property] binding -->
<p-button [disabled]="!tournament || (tournament?.teams?.length || 0) < 2 || isReloading">
</p-button>
<td [class.score-updating]="isUpdating[match.id]">

<!-- (event) binding -->
<td (click)="!match.winnerId && !isUpdating[match.id] && updateMatchScore(match.id, match.teamAId)">

<!-- [(ngModel)] two-way binding — login.component.html -->
<input type="email"    [(ngModel)]="email"    name="email" />
<input type="password" [(ngModel)]="password" name="password" />
```

Under the hood, Angular runs **change detection** after every event, async operation, or timer to check whether any component data changed and update the DOM accordingly. By default it checks every component in the tree. `OnPush` is an opt-in strategy that tells Angular to skip a component unless its `@Input` references changed — a performance optimisation worth knowing exists, even if you haven't tuned it yourself.

## Directives

Instructions you attach to HTML elements to tell Angular what to do with them.

**Structural** — add or remove elements from the DOM:

- `*ngIf` — renders the element only if the condition is true
- `*ngFor` — renders the element once per item in a collection, like a foreach in HTML

**Attribute** — change how an element looks or behaves without touching the DOM structure: `[ngClass]`, `[ngStyle]`.

```html
<!-- *ngIf — tournament-header.component.html -->
<div class="tournament-header" *ngIf="tournament">

<p-button *ngIf="tournamentState?.currentStatus === TournamentStatus.Setup
                 && tournament?.format === Format.GroupsAndBracket"
          label="Start Groups">
</p-button>
```

```html
<!-- *ngFor with trackBy — matches.component.html -->
<!-- trackBy avoids destroying/recreating DOM nodes when the list changes -->
<tr *ngFor="let match of matches; trackBy: trackByMatchId">
```

```typescript
// matches.component.ts
trackByMatchId(index: number, match: Match): number {
  return match.id;
}
```

```html
<!-- [ngClass] object syntax — keys are class names, values are booleans -->
<!-- matches.component.html -->
<td [ngClass]="{
      'winner':         match.winnerId === match.teamAId,
      'loser':          match.loserId  === match.teamAId,
      'clickable-team': !match.winnerId && !isUpdating[match.id] && !isGroupFinished
    }">
```

```html
<!-- [ngClass] string expression — tournament-state-banner.component.html -->
<div class="state-banner" [ngClass]="'state-' + tournamentState.currentStatus">
  <i class="pi" [ngClass]="tournamentState.isTransitionState ? 'pi-spin pi-spinner' : 'pi-info-circle'"></i>
</div>
```

```html
<!-- ng-container — group directives without adding a DOM element -->
<!-- create-tournament.component.html -->
<ng-container *ngIf="format !== null">
  <div class="field-divider"></div>
  <!-- max teams, groups etc. only shown once format is selected -->
</ng-container>
```

## Services and Dependency Injection

A service is a class that holds logic or data shared across multiple components — API calls, business logic, shared state. Components should only handle display and user input; the real work lives in services.

DI is how Angular delivers a service to whoever needs it, without manual instantiation — identical mental model to ASP.NET Core DI.

**Where you provide it controls the lifetime:**

- `providedIn: 'root'` → one singleton for the whole app (`AddSingleton`)
- Provided at component level → each component gets its own instance (`AddTransient`)

```typescript
// tournament.service.ts — singleton, available app-wide
@Injectable({
  providedIn: 'root'
})
export class TournamentService {
  private apiUrl = `${environment.apiUrl}/tournaments`;
  constructor(private http: HttpClient) { }
}
```

```typescript
// create-tournament.component.ts — constructor injection
constructor(
  private fb: FormBuilder,
  private tournamentService: TournamentService,
  private configService: TournamentConfigService,
  private router: Router,
  private messageService: MessageService
) {}
```

## Lifecycle Hooks

Methods Angular calls at specific moments of a component's life. Three matter at junior level:

**`ngOnInit`** — runs once after Angular finishes setting up the component. Fetch data here, not in the constructor. The constructor runs before Angular wires up inputs — it's for DI only.

```typescript
// navbar.component.ts — subscribe to shared service streams
ngOnInit(): void {
  this.userSubscription = this.authService.currentUser$.subscribe(user => {
    this.currentUser = user;
    this.setupUserMenu();
  });

  this.tournamentSubscription = this.tournamentContext.tournament$.subscribe(tournament => {
    this.currentTournament = tournament;
  });
}
```

```typescript
// create-tournament.component.ts — build a reactive form
ngOnInit(): void {
  this.form = this.fb.group({
    name:   ['', [Validators.required, Validators.minLength(4), Validators.maxLength(100)]],
    format: [null, Validators.required],
    maxTeams: [null],
  });

  this.form.get('format')!.valueChanges
    .pipe(takeUntil(this.destroy$))
    .subscribe(() => this.onFormatChange());
}
```

**`ngOnChanges`** — runs whenever the parent passes new input data into this component. Use it to react to those changes.

```typescript
// bracket.component.ts
ngOnChanges(changes: SimpleChanges) {
  // firstChange is true on initial render — skip that
  if (changes['tournament'] && !changes['tournament'].firstChange && this.viewInitialized) {
    this.loadBracket();
  }
}
```

**`ngOnDestroy`** — runs before the component is removed. Cancel subscriptions here or you'll leak memory.

```typescript
// matches.component.ts — the standard TO2 pattern
private destroy$ = new Subject<void>();

ngOnDestroy(): void {
  this.destroy$.next();
  this.destroy$.complete();
}
```

## RxJS and Observables

Angular's way of handling async data. The key distinction:

- **Promise** — delivers one value, once, can't be cancelled
- **Observable** — emits values over time, cancellable, composable

Angular uses Observables everywhere (HTTP, routing, forms) because they're cancellable — if the user navigates away mid-request, Angular can abort it. A Promise keeps running regardless.

You get values out by **subscribing**. Your responsibility is to unsubscribe when done — if you don't, the subscription keeps the component alive in memory after it's destroyed (memory leak).

The `async` pipe in templates subscribes and unsubscribes automatically — prefer it over manual subscription for display scenarios.

**Three operators to know:** `map` (transform values), `filter` (gate values), `switchMap` (cancel the previous inner Observable when a new value arrives — essential for search inputs).

```typescript
// tournament-context.service.ts — BehaviorSubject holds current value,
// replays it to new subscribers; exposed as read-only Observable
private tournamentSubject = new BehaviorSubject<Tournament | null>(null);
private groupsSubject     = new BehaviorSubject<Standing[]>([]);

tournament$ = this.tournamentSubject.asObservable();
groups$     = this.groupsSubject.asObservable();

setTournament(tournament: Tournament | null): void {
  this.tournamentSubject.next(tournament);  // push new value to all subscribers
}
```

```typescript
// tournament-context.service.ts — Subject for action triggers (no initial value)
private startGroupsSubject = new Subject<void>();
startGroups$ = this.startGroupsSubject.asObservable();

triggerStartGroups(): void { this.startGroupsSubject.next(); }
```

```typescript
// tournament.service.ts — map + switchMap
// switchMap cancels the previous inner Observable when a new value arrives
getTournamentWithTeams(id: number): Observable<Tournament> {
  return this.getTournament(id).pipe(
    switchMap(tournament =>
      this.getAllTeamsByTournamentId(tournament.id).pipe(
        map(teams => ({ ...tournament, teams }))
      )
    )
  );
}
```

```typescript
// app.component.ts — filter: gate values by condition
this.router.events
  .pipe(
    filter(event => event instanceof NavigationEnd),
    takeUntil(this.destroy$)
  )
  .subscribe((event: any) => {
    this.showNavbar = !['/', '/login', '/register'].includes(event.url);
  });
```

```typescript
// bracket.component.ts — takeUntil: auto-unsubscribe on destroy
// Every component in TO2 uses this pattern
private destroy$ = new Subject<void>();

ngOnInit() {
  this.tournamentContext.bracket$
    .pipe(takeUntil(this.destroy$))
    .subscribe(bracket => {
      if (bracket?.matches) {
        this.matches = bracket.matches;
        if (this.viewInitialized) this.renderBracket();
      }
    });
}

ngOnDestroy(): void {
  this.destroy$.next();
  this.destroy$.complete();
}
```

```typescript
// matches.component.ts — subscribe with next/error
this.matchService.setGameResult(gameResult)
  .pipe(takeUntil(this.destroy$))
  .subscribe({
    next: (result) => {
      if (result.matchFinished) {
        match.winnerId = result.matchWinnerId;
        this.matchFinished.emit({ winnerId: result.matchWinnerId, ... });
      }
      this.isUpdating[matchId] = false;
    },
    error: (err) => {
      match.teamAWins = previousTeamAWins;  // rollback optimistic update
      match.teamBWins = previousTeamBWins;
      this.isUpdating[matchId] = false;
    }
  });
```

```html
<!-- tournament-list.component.html — async pipe: no manual subscribe/unsubscribe -->
<div *ngIf="groupedTournaments$ | async as tournaments">
  <section *ngIf="tournaments.upcoming.length">...</section>
</div>
```

## HttpClient

Angular's built-in HTTP service. Returns Observables, not Promises. Call it inside a service, consume the result in a component or template.

**Interceptors** sit in the middle of every request and response — same mental model as ASP.NET Core middleware. Common uses: attaching JWT tokens to every outgoing request, handling 401s globally, centralised error logging.

```typescript
// tournament.service.ts
@Injectable({ providedIn: 'root' })
export class TournamentService {
  private apiUrl = `${environment.apiUrl}/tournaments`;
  constructor(private http: HttpClient) { }

  getAllTournaments(): Observable<Tournament[]> {
    return this.http.get<Tournament[]>(`${this.apiUrl}/all`);
  }

  createTournament(request: CreateTournamentRequest): Observable<Tournament> {
    return this.http.post<Tournament>(this.apiUrl, request);
  }

  startTournament(tournamentId: number) {
    return this.http.put(`${this.apiUrl}/${tournamentId}/start`, {});
  }
}
```

```typescript
// auth.interceptor.ts — attaches JWT to every request, retries on 401
@Injectable()
export class AuthInterceptor implements HttpInterceptor {
  private isRefreshing = false;
  private refreshTokenSubject = new BehaviorSubject<string | null>(null);

  constructor(private authService: AuthService, private router: Router) {}

  intercept(request: HttpRequest<unknown>, next: HttpHandler): Observable<HttpEvent<unknown>> {
    const isAuthEndpoint = request.url.includes('/auth/login') ||
                           request.url.includes('/auth/register') ||
                           request.url.includes('/auth/refresh');

    if (!isAuthEndpoint) request = this.addToken(request);

    return next.handle(request).pipe(
      catchError((error: HttpErrorResponse) => {
        if (error.status === 401 && !isAuthEndpoint) return this.handle401(request, next);
        return throwError(() => error);
      })
    );
  }

  private addToken(request: HttpRequest<unknown>): HttpRequest<unknown> {
    const token = this.authService.getAccessToken();
    if (!token) return request;
    return request.clone({ setHeaders: { Authorization: `Bearer ${token}` } });
  }

  private handle401(request: HttpRequest<unknown>, next: HttpHandler): Observable<HttpEvent<unknown>> {
    if (this.isRefreshing) {
      return this.refreshTokenSubject.pipe(
        filter(token => token !== null), take(1),
        switchMap(() => next.handle(this.addToken(request)))
      );
    }
    this.isRefreshing = true;
    this.refreshTokenSubject.next(null);

    return this.authService.refreshAccessToken().pipe(
      switchMap(response => {
        this.isRefreshing = false;
        this.refreshTokenSubject.next(response.accessToken);
        return next.handle(this.addToken(request));
      }),
      catchError(err => {
        this.isRefreshing = false;
        this.authService.logout();
        this.router.navigate(['/login'], { queryParams: { returnUrl: this.router.url } });
        return throwError(() => err);
      })
    );
  }
}
```

Registered in `app.module.ts`:
```typescript
providers: [
  { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor, multi: true }
]
```

## Routing

The router watches the URL and swaps which component is displayed — without a server round-trip. `<router-outlet>` is the slot in the template where the current route's component renders.

**Route guards** (`CanActivate`) run before a route loads and decide whether navigation is allowed. Direct frontend equivalent of `[Authorize]` on a controller.

**Important:** a route guard is UX, not real security. It can be bypassed. The backend must always enforce its own authorization.

```typescript
// app-routing.module.ts
const routes: Routes = [
  { path: '',                  component: LandingComponent },
  { path: 'login',             component: LoginComponent },
  { path: 'tournaments',       component: TournamentListComponent,    canActivate: [AuthGuard] },
  { path: 'tournament/:id',    component: TournamentDetailsComponent, canActivate: [AuthGuard] },
  { path: 'create-tournament', component: CreateTournamentComponent,  canActivate: [AuthGuard] },
];
```

```html
<!-- app.component.html -->
<header *ngIf="showNavbar"><app-navbar></app-navbar></header>
<main [class.with-navbar]="showNavbar">
  <router-outlet></router-outlet>
</main>
```

```typescript
// auth.guard.ts — attempts token refresh before redirecting to login
@Injectable({ providedIn: 'root' })
export class AuthGuard implements CanActivate {
  constructor(private authService: AuthService, private router: Router) {}

  canActivate(route: ActivatedRouteSnapshot, state: RouterStateSnapshot) {
    if (this.authService.isAuthenticated()) return true;

    if (this.authService.getRefreshToken()) {
      return this.authService.refreshAccessToken().pipe(
        map(() => true),
        catchError(() => of(this.loginRedirect(state.url)))
      );
    }

    return this.loginRedirect(state.url);
  }

  private loginRedirect(returnUrl: string): UrlTree {
    return this.router.createUrlTree(['/login'], { queryParams: { returnUrl } });
  }
}
```

```typescript
// programmatic navigation — login.component.ts
this.authService.login(this.email, this.password).subscribe({
  next: () => this.router.navigate(['/tournaments'])
});
```

## Forms

Two approaches:

**Template-driven** — form structure defined in HTML using `ngModel`. Quick to write, suitable for simple forms.

```html
<!-- login.component.html -->
<form (ngSubmit)="onSubmit()">
  <input type="email"    [(ngModel)]="email"    name="email"    required />
  <input type="password" [(ngModel)]="password" name="password" required />

  <div *ngIf="errorMessage" class="custom-error-message">{{ errorMessage }}</div>

  <button type="submit" [disabled]="isLoading">
    <span *ngIf="!isLoading">Sign In</span>
    <span *ngIf="isLoading">Signing In...</span>
  </button>
</form>
```

```typescript
// login.component.ts — no FormBuilder, plain properties bound via ngModel
export class LoginComponent {
  email = '';
  password = '';
  isLoading = false;
  errorMessage = '';

  onSubmit(): void {
    if (!this.email || !this.password) {
      this.errorMessage = 'Please fill in all fields';
      return;
    }
    // call service...
  }
}
```

**Reactive** — form model built explicitly in TypeScript, connected to the template. More code, but fully testable, easier to validate, better for dynamic or complex forms.

```typescript
// create-tournament.component.ts
ngOnInit(): void {
  this.form = this.fb.group({
    name:              ['', [Validators.required, Validators.minLength(4), Validators.maxLength(100)]],
    description:       ['', [Validators.maxLength(250)]],
    format:            [null, Validators.required],
    maxTeams:          [null],
    numberOfGroups:    [null],
    advancingPerGroup: [null],
  });

  this.form.get('format')!.valueChanges
    .pipe(takeUntil(this.destroy$))
    .subscribe(() => this.onFormatChange());
}
```

```html
<!-- create-tournament.component.html -->
<form [formGroup]="form" (ngSubmit)="submit()">

  <input formControlName="name"
         [class.invalid]="form.get('name')?.invalid && form.get('name')?.touched"
         placeholder="Tournament name" />

  <!-- Radio buttons bound to enum values -->
  <input type="radio" formControlName="format" [value]="Format.BracketOnly" />
  <input type="radio" formControlName="format" [value]="Format.GroupsAndBracket" />

  <!-- Conditional field — shown only for group formats -->
  <select formControlName="numberOfGroups"
          *ngIf="format === Format.GroupsOnly || format === Format.GroupsAndBracket">
    <option *ngFor="let n of numberOfGroupsOptions" [value]="n">{{ n }}</option>
  </select>

</form>
```

Custom validator — a function that returns `null` (valid) or an errors object (invalid):

```typescript
// team-management-card.component.ts
duplicateTeamNamesValidator(): ValidatorFn {
  return (control: AbstractControl): ValidationErrors | null => {
    const value = control.value;
    if (!value || value.trim() === '') return null;

    const names = value.split(',')
      .map((n: string) => n.trim().toLowerCase())
      .filter((n: string) => n.length > 0);

    const unique = new Set(names);
    return unique.size !== names.length ? { duplicateTeamNames: true } : null;
  };
}

// wired into FormBuilder:
this.bulkAddForm = this.fb.group({
  teamNames: ['', [Validators.required, this.duplicateTeamNamesValidator()]]
});
```

Default to reactive for anything non-trivial — it keeps logic in TypeScript, out of the template.

## Interview Traps

- **"Component vs directive?"** — A component is a directive with a template. Directives modify existing elements; components create new UI blocks.
- **"Why `ngOnInit` not the constructor?"** — Constructor runs before Angular wires inputs. Use it only for DI. `ngOnInit` is where the component is actually ready.
- **"Observable vs Promise?"** — Promise resolves once, not cancellable. Observable emits over time, cancellable. Angular prefers Observables for HTTP.
- **"Forget to unsubscribe?"** — Memory leak. Subscription holds a reference to the component, keeping it alive after destruction.
- **"What is `providedIn: 'root'`?"** — Registers the service as an app-wide singleton. One instance shared everywhere.
- **"Template-driven vs reactive?"** — Template-driven for simple forms. Reactive for anything you need to test or that has real complexity.
- **"Is a route guard real security?"** — No. It's client-side UX. Always enforce authorization on the backend.
