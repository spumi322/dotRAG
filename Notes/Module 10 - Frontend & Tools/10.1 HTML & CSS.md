# 10.1 HTML & CSS

## HTML — Structure and Meaning

HTML defines the **structure and meaning** of web content. It is not a programming language — it's a markup language. HTML tells the browser _what_ the content is, CSS tells it _how it looks_, JavaScript tells it _how it behaves_. 

---

## Semantic HTML

HTML defines **structure and meaning**. CSS handles appearance, JavaScript handles behavior.

Semantic elements describe _what_ the content is:

```
<nav>      → navigation
<main>     → primary page content
<article>  → self-contained content
<section>  → thematic group
<header>   → page or section header
<footer>   → page or section footer

<div>      → generic block (no meaning)
<span>     → generic inline (no meaning)
```

**Why it matters:**

1. **Accessibility** — screen readers (assistive software for visually impaired users) navigate by landmarks. `<nav>` announces itself; `<div>` is invisible to them. Use descriptive images: `alt="..."`, so the screen readers do not read filenames.
2. **SEO** — search engines weight content inside semantic elements over anonymous `<div>`s.
3. **Maintainability** — `<nav>` is self-documenting; `<div class="nav-wrapper-outer">` is not.

> Rule of thumb: use the native element. Reach for `<div>` only when no semantic element fits.

## The DOM

When the browser receives HTML, it parses it and builds a **DOM (Document Object Model)** — an in-memory tree of objects. JavaScript never touches the raw HTML file; it reads and modifies the page through the DOM.

```js
document.getElementById("title").textContent = "Hello"; // modifies a DOM node
```

When React, Angular, or any framework "updates the UI", it modifies DOM nodes. The browser re-renders whatever changed.

**Why DOM size matters:** every DOM node costs memory. Deep nesting makes traversal slower.

## `localStorage` vs `sessionStorage` vs Cookies

Three ways to store data in the browser — different lifetimes and purposes.

**`localStorage`** — persists until explicitly cleared. Use for non-sensitive user preferences: dark mode, language, UI settings.

**`sessionStorage`** — wiped when the tab closes. Use for temporary per-tab state: wizard step progress, unsaved form data.

**Cookies** — configurable lifetime, and critically: the browser sends them automatically on every HTTP request to the same domain. Use for authentication — the server sets the cookie on login, the browser attaches it to every subsequent request without any frontend code.

---

## Which to use when

|Scenario|Use|
|---|---|
|Auth token|Cookie (`HttpOnly` + `Secure` + `SameSite=Strict`)|
|Dark mode preference|`localStorage`|
|Multi-step form progress|`sessionStorage`|
|Anything sensitive|Cookie with `HttpOnly` — JS can't read it|

---

## Why cookies for auth, not `localStorage`?

`localStorage` is readable by any JavaScript on the page. An XSS(Cross-Site Scripting) — attacker injects malicious JavaScript into your page) attack can steal your token instantly.

`HttpOnly` cookies are invisible to JavaScript — only the browser and server touch them. XSS is neutralized.

```
HttpOnly   → JS cannot read the cookie (XSS protection)
Secure     → only sent over HTTPS
SameSite=Strict → only sent on same-site requests (CSRF protection)
```

In ASP.NET Core, calling `SignInAsync()` sets the auth cookie for you — you configure the flags, you don't write cookie code manually.

## Specificity

When two CSS rules target the same element, the browser picks the winner by **specificity score**:

```
Inline style   (style="...")      — highest priority
ID             (#id)              — very strong
Class          (.class), :hover   — medium
Element        (p, div)           — weakest
```

`!important` overrides everything — avoid it. It breaks the predictable resolution chain and makes debugging a nightmare.

> "If a style isn't applying, check whether a more specific selector is overriding it."

## Box model

Every element is a box with four layers (inside out):

```
+-------------------------------+  ← margin (transparent, pushes others away)
|  +-------------------------+  |
|  |  border                 |  |
|  |  +-------------------+  |  |
|  |  |  padding          |  |  |
|  |  |  +-------------+  |  |  |
|  |  |  |   content   |  |  |  |
|  |  |  +-------------+  |  |  |
|  |  +-------------------+  |  |
|  +-------------------------+  |
+-------------------------------+
```

**Content** — actual text or image. **Padding** — space inside the border (inherits background color). **Border** — visible edge. **Margin** — transparent space outside (pushes other elements away).

**The problem:** default `box-sizing: content-box` adds padding and border on top of the declared `width`. A `width: 200px` element with `padding: 20px` and `border: 1px` is actually 242px wide.

**The fix every modern project uses:**
```css
*, *::before, *::after {
    box-sizing: border-box;
}
```

Now `width: 200px` includes padding and border. Predictable. Angular projects apply this by default.

> "Padding and margin difference: padding is inside the border (part of the element, has background). Margin is outside — transparent, pushes others away. Vertical margins between elements **collapse** — the larger one wins, they don't add up."

## Flexbox vs Grid

Two modern layout systems. Interviewers ask you to choose between them.

**Flexbox** — **1D layout** (row or column). Content decides how much space each item takes.

```css
.navbar {
    display: flex;
    justify-content: space-between; /* spread items */
    align-items: center;            /* vertically center */
}
```

**Grid** — **2D layout** (rows AND columns). You define the structure; content fills it.

```css
.card-grid {
    display: grid;
    grid-template-columns: repeat(3, 1fr); /* 3 equal columns */
    gap: 20px;
}
```

> **Mental model:** Flex = content decides layout. Grid = you decide layout, content fills it.

They compose: use Grid for page structure, Flex inside each card to align content.

## Responsive Design & Mobile Web

Responsive design means the layout adapts to different screen sizes instead of breaking or scrolling unexpectedly.

**Media queries** — apply CSS conditionally based on screen width:

```css
/* Mobile-first: base styles for small screens */
.card { width: 100%; }

/* Adapt for larger screens */
@media (min-width: 768px) {
    .card { width: 50%; }
}
```

**Mobile-first** = write base styles for small screens, layer complexity up with `min-width`. Current industry standard. In practice, component libraries (PrimeNG, Bootstrap) handle responsive breakpoints via grid classes — you rarely write media queries by hand.

If an Angular app targets mobile browser users, the architecture doesn't change — same components, same routing. What changes:

- **Viewport meta tag** — without it, mobile browsers render the desktop version scaled down tiny:
```html
  <meta name="viewport" content="width=device-width, initial-scale=1">
```

- **Touch targets** — buttons need to be large enough to tap (~48px minimum)
- **No hover states** — `:hover` is unreliable on touch devices, don't rely on it for critical UI
- **Performance** — mobile CPUs and networks are weaker; keep bundle size small, use lazy loading

> **Testing:** Chrome/Edge DevTools → `F12` → device toolbar icon to simulate any screen size. Test on Chrome and Safari at minimum — Safari (iOS) has its own rendering engine and occasionally handles CSS differently.

## Interview Traps

- **"Semantic HTML — why use it?"** — Accessibility (screen readers navigate landmarks, native buttons are keyboard-friendly), SEO, maintainability.
- **"What is the DOM?"** — Browser's in-memory tree of the HTML document. JavaScript interacts with the page through it — never the HTML file directly.
- **"Box model?"** — Content + padding + border + margin. Use `box-sizing: border-box` so declared width includes padding and border.
- **"Flexbox vs Grid?"** — Flex is 1D (row or column), content-driven. Grid is 2D (rows + columns), structure-driven. Use flex for component internals, grid for page layout.
- **"`display: none` vs `visibility: hidden` vs `opacity: 0`?"** — `none` removes from layout (takes no space, hidden from screen readers). `hidden` is invisible but still occupies space. `opacity: 0` is invisible but still takes space and receives events.
- **"`localStorage` vs cookies for auth tokens?"** — Always cookies: set `HttpOnly` so JavaScript can't read them (XSS safe) + `Secure` + `SameSite=Strict`. `localStorage` is accessible to any script on the page.
