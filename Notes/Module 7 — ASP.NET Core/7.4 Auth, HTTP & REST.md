# 7.4 Auth, HTTP & REST

## HTTP Fundamentals

HTTP (Hypertext Transfer Protocol) is a stateless, request-response protocol. The client sends a request, the server returns a response. Each request is independent — the server stores no session state between requests.

### HTTP methods

| Method | Purpose | Idempotent | Safe | Has Body |
|---|---|---|---|---|
| `GET` | Retrieve a resource | ✅ | ✅ | ❌ |
| `POST` | Create a resource | ❌ | ❌ | ✅ |
| `PUT` | Replace a resource entirely | ✅ | ❌ | ✅ |
| `PATCH` | Partially update a resource | ⚠️ Not guaranteed | ❌ | ✅ |
| `DELETE` | Remove a resource | ✅ | ❌ | ❌ |
| `OPTIONS` | Discover allowed methods (CORS preflight) | ✅ | ✅ | ❌ |

**Idempotent** = calling it N times produces the same result as once. **Safe** = does not modify server state.

### Status codes

**2xx — Success:**
- `200 OK` — general success
- `201 Created` — resource created (POST), include `Location` header
- `204 No Content` — success, no body (DELETE / PUT)

**3xx — Redirection:**
- `301 Moved Permanently`, `304 Not Modified`

**4xx — Client Error:**
- `400 Bad Request` — invalid input / validation failure
- `401 Unauthorized` — not authenticated (misleading name — means "unauthenticated")
- `403 Forbidden` — authenticated but not authorized
- `404 Not Found` — resource doesn't exist
- `409 Conflict` — state conflict (e.g., duplicate create)
- `422 Unprocessable Entity` — semantically invalid request
- `429 Too Many Requests` — client exceeded rate limit

**5xx — Server Error:**
- `500 Internal Server Error` — unhandled exception
- `502 Bad Gateway`, `503 Service Unavailable`

### Important headers

| Header | Purpose |
|---|---|
| `Content-Type` | Format of the request/response body (`application/json`) |
| `Accept` | Client's preferred response format |
| `Authorization` | Credentials (`Bearer <token>`) |
| `Cache-Control` | Caching directives |
| `Location` | URI of newly created resource (201 responses) |
| `X-Correlation-Id` | Request tracing across services (custom) |

### HTTPS

HTTPS = HTTP + TLS. All production APIs must use HTTPS. ASP.NET Core enforces redirection via `app.UseHttpsRedirection()`.

### CORS

When a browser makes a cross-origin request (different domain/port), the server must include `Access-Control-Allow-Origin` headers. The browser sends a **preflight** `OPTIONS` request first for non-simple requests (custom headers, PUT/DELETE).

```csharp
builder.Services.AddCors(o => o.AddPolicy("AllowFrontend", p =>
    p.WithOrigins("https://myapp.com")
     .AllowAnyMethod()
     .AllowAnyHeader()));

app.UseCors("AllowFrontend");
```

### ProblemDetails (RFC 9457)

Standardized error response format. ASP.NET Core returns this automatically when using `[ApiController]`.

```json
{
  "type": "https://tools.ietf.org/html/rfc7231#section-6.5.4",
  "title": "Not Found",
  "status": 404,
  "detail": "Tournament with id 42 was not found."
}
```

## REST

REST (Representational State Transfer) is an architectural style for APIs. It uses HTTP as transport and treats everything as a **resource** identified by a URL.

### Core constraints

1. **Stateless** — each request contains all info needed; no server-side session
2. **Client-Server** — separation of concerns between UI and data storage
3. **Cacheable** — responses define if they can be cached
4. **Uniform Interface** — consistent URL structure, standard HTTP methods
5. **Layered System** — client doesn't know if it talks directly to the server or through proxies

### Resource naming conventions

- Use **nouns**, not verbs: `/api/products` not `/api/getProducts`
- Use **plural**: `/api/orders` not `/api/order`
- Use **hierarchy** for relations: `/api/orders/42/items`
- Use **query parameters** for filtering/sorting: `/api/products?category=electronics&sort=price`

| Action | Method | Endpoint | Response |
|---|---|---|---|
| List all | GET | `/api/products` | 200 + array |
| Get one | GET | `/api/products/5` | 200 or 404 |
| Create | POST | `/api/products` | 201 + Location header |
| Replace | PUT | `/api/products/5` | 200 or 204 |
| Partial update | PATCH | `/api/products/5` | 200 or 204 |
| Delete | DELETE | `/api/products/5` | 204 or 404 |

### API versioning strategies

1. **URL path** (most common): `/api/v1/products` — simple, explicit
2. **Query string**: `/api/products?api-version=1.0`
3. **Header**: `X-Api-Version: 1` or `Accept: application/vnd.myapi.v1+json`

## Authentication & Authorization

### Authentication — who are you?

Verifies the identity of the user. Common schemes for APIs:
- **JWT Bearer** — dominant for APIs (stateless, scalable)
- **Cookie-based** — for MVC/Razor Pages (server-side sessions, automatic in browser)
- **OAuth 2.0 / OpenID Connect** — external identity providers (Google, Azure AD)

### JWT setup

```csharp
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(o =>
    {
        o.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidIssuer = builder.Configuration["Jwt:Issuer"],
            ValidateAudience = true,
            ValidAudience = builder.Configuration["Jwt:Audience"],
            ValidateLifetime = true,
            IssuerSigningKey = new SymmetricSecurityKey(
                Encoding.UTF8.GetBytes(builder.Configuration["Jwt:Key"]!))
        };
    });

app.UseAuthentication(); // must come BEFORE UseAuthorization
app.UseAuthorization();
```

**What `UseAuthentication()` does per request:**
1. Extracts token from `Authorization: Bearer <token>` header
2. Validates signature using `IssuerSigningKey`
3. Validates issuer, audience, and expiry
4. If all pass → builds `ClaimsPrincipal`, sets it on `HttpContext.User`

### Authorization — what can you do?

Determines what an authenticated user is allowed to do.

```csharp
// Policy-based authorization (preferred over role strings)
builder.Services.AddAuthorization(o =>
    o.AddPolicy("AdminOnly", p => p.RequireClaim("role", "admin")));

// Apply to endpoint
[Authorize(Policy = "AdminOnly")]
[HttpDelete("{id}")]
public IActionResult Delete(int id) => ...;

// Role-based (simpler, less flexible)
[Authorize(Roles = "Admin,Moderator")]
public IActionResult AdminAction() => ...;

// Allow anonymous (override controller-level [Authorize])
[AllowAnonymous]
[HttpGet]
public IActionResult PublicEndpoint() => ...;
```

## Interview Traps

- **"401 vs 403?"** — 401 = not authenticated (you haven't proven who you are). 403 = authenticated but not authorized (we know who you are, you just don't have permission). The name "Unauthorized" is misleading — 401 means unauthenticated.
- **"PUT vs PATCH?"** — PUT replaces the entire resource (idempotent). PATCH modifies specific fields. Two identical PUTs produce the same result; two POSTs create two resources.
- **"What is idempotency?"** — An operation is idempotent if calling it N times produces the same result as 1 call. GET, PUT, DELETE are idempotent. POST is not.
- **"What makes an API RESTful?"** — Stateless, resource-based URLs (nouns + hierarchy), standard HTTP methods, proper status codes, uniform interface.
- **"JWT vs Cookie auth — when to choose?"** — JWT for APIs consumed by SPAs or mobile apps (stateless, easy to pass in headers). Cookies for server-rendered MVC apps (automatic in browser, CSRF protection needed).
- **"Authentication middleware order?"** — `UseAuthentication()` must come before `UseAuthorization()`. Wrong order → authorization checks run before identity is established → every request appears anonymous.
