# 7.3 Transactions & Security

## ACID Properties

ACID guarantees that database transactions are processed reliably.

```
A ── All or nothing          → ROLLBACK on failure
C ── Rules always enforced   → Constraints never violated
I ── Transactions isolated   → No dirty reads mid-flight
D ── Committed = permanent   → Survives crashes
```

### A — Atomicity

> "All or nothing."

Either ALL steps commit, or NONE do. A failure mid-transaction leaves the database as if nothing happened.

```
RECORD MATCH RESULT
  Step 1: Save match score      ✅
  Step 2: Update standings      ❌ FAIL
  ──────────────────────────────────
  Result: ROLLBACK — both undone. Match score: unchanged.
```

### C — Consistency

> "Valid state in, valid state out."

Every transaction must bring the DB from one **valid** state to another. Integrity constraints (NOT NULL, CHECK, FK) are never violated — the transaction is rejected if they would be.

### I — Isolation

> "Concurrent transactions don't see each other's work."

Transactions execute as if they were sequential, even when running concurrently. How much they see of each other's in-progress work is controlled by the **isolation level**.

### D — Durability

> "Committed = permanent."

Once committed, a transaction survives any failure — power loss, crash, restart. Data is written to disk (or write-ahead log) before the COMMIT returns.

## Transactions

A transaction is a unit of work where all operations succeed together or none of them do.

```sql
BEGIN TRANSACTION;

    UPDATE Matches SET HomeScore = 3, AwayScore = 1 WHERE Id = 1;
    UPDATE Standings SET Points = Points + 3 WHERE TeamId = 5;

COMMIT;  -- both updates are permanently saved
```

```sql
BEGIN TRANSACTION;

    UPDATE Matches SET HomeScore = 3, AwayScore = 1 WHERE Id = 1;
    -- something goes wrong
ROLLBACK;  -- first UPDATE is undone, score restored
```

```sql
-- SAVEPOINT: partial rollback within a transaction
BEGIN TRANSACTION;

    INSERT INTO Matches (TournamentId, Round) VALUES (1, 1);
    SAVE TRANSACTION AfterInsert;

    UPDATE Standings SET Points = Points + 3 WHERE TeamId = 5;

    -- roll back only the UPDATE, keep the INSERT
    ROLLBACK TRANSACTION AfterInsert;

COMMIT;
```

## Isolation Levels

Define how much one transaction can see the uncommitted work of another. Higher isolation = safer data, lower concurrency.

- **Dirty read** — reading uncommitted data from another transaction that might be rolled back

> **Junior level:** Know what isolation levels are and the default (`Read Committed`). Understanding dirty reads is the key concept to internalize.

## Stored Procedures

A named, saved SQL block that can be executed repeatedly. Lives in the database, not the application.

```sql
-- Create
CREATE PROCEDURE GetMatchesByTournament
    @TournamentId INT
AS
BEGIN
    SELECT * FROM Matches WHERE TournamentId = @TournamentId;
END;

-- Execute
EXEC GetMatchesByTournament @TournamentId = 42;
```

### Stored Procedure vs Function

| | Stored Procedure | Function |
|---|---|---|
| Returns | Optional / OUTPUT params | Must return a value |
| Used in SELECT | ❌ No | ✅ Yes |
| Can modify data | ✅ Yes | ❌ Generally no |
| Called with | `EXEC` | Inside a query |

```sql
-- Function: used inline in queries
CREATE FUNCTION CalculateGroupPoints (@Wins INT, @Draws INT)
RETURNS INT
AS
BEGIN
    RETURN (@Wins * 3) + @Draws;
END;

SELECT TeamId, dbo.CalculateGroupPoints(Wins, Draws) AS TotalPoints FROM Standings;
```

**As a .NET developer** — you'll rarely write stored procedures. EF Core handles most data access. What to know:
- What they are and why they exist (reusability, encapsulation, DB-level permissions)
- How to call one from EF Core: `context.Matches.FromSqlRaw("EXEC GetMatchesByTournament @TournamentId = {0}", id).ToListAsync()`
- Main downside: **business logic hidden in the database** — hard to version-control, test, and debug

## Triggers

A trigger automatically executes in response to a data modification event on a table.

```sql
-- Log every deletion from Orders
CREATE TRIGGER trg_MatchDeleted
ON Matches
AFTER DELETE
AS
BEGIN
    INSERT INTO AuditLog (Event, MatchId, DeletedAt)
    SELECT 'MATCH_DELETED', deleted.Id, GETDATE()
    FROM deleted;  -- 'deleted' is a special pseudo-table inside triggers
END;
```

**Events:** `INSERT`, `UPDATE`, `DELETE`
**Timings:** `BEFORE` (validation) / `AFTER` (auditing, cascading)

**As a .NET developer** — almost never write triggers. Know:
- What they are — automatic DB-side reactions to data changes
- Common use case — audit logging
- Why avoided in modern apps — same problems as stored procedures: hidden logic, untestable, hard to debug
- Modern equivalent: EF Core `SaveChangesInterceptor` for audit logging

## Database Security

### Authentication vs Authorization

- **Authentication** — verifies identity (username + password, certificates, AD integration)
- **Authorization** — controls what authenticated users can do (GRANT/REVOKE specific privileges)

**Principle of least privilege:** Grant only the permissions each user/role actually needs.

```sql
-- Grant read-only access
GRANT SELECT ON Tournaments TO AppReadUser;

-- Revoke write access
REVOKE INSERT, UPDATE, DELETE ON Tournaments FROM AppReadUser;
```

### SQL Injection

SQL injection inserts malicious SQL into input fields, potentially exposing or destroying data.

```sql
-- ❌ Vulnerable — string concatenation
var sql = "SELECT * FROM Users WHERE Name = '" + userInput + "'";
-- Input: ' OR '1'='1  → returns all users

-- ✅ Safe — parameterized query
SELECT * FROM Users WHERE Name = @Name;  -- @Name is never interpreted as SQL
```

**Prevention:**
1. **Parameterized queries** — always use parameters, never concatenate user input into SQL
2. **ORM (EF Core)** — generates parameterized SQL automatically from LINQ
3. **Stored procedures** — with parameterized parameters (not dynamic SQL inside the proc)
4. **Input validation** — reject unexpected characters at the application boundary
5. **Principle of least privilege** — even if injected, the DB user can't DROP TABLE if it lacks that permission

```csharp
// EF Core — parameterized automatically
var users = await _context.Users
    .Where(u => u.Name == userInput) // safe — EF parameterizes this
    .ToListAsync();

// Raw SQL — use interpolated (parameterized) not Raw (concatenated)
var users = await _context.Users
    .FromSqlInterpolated($"SELECT * FROM Users WHERE Name = {userInput}") // ✅ safe
    .ToListAsync();
```

### Row-Level Security (RLS)

RLS controls access to individual rows based on user permissions via security policies. Prevents users from viewing rows they don't own — enforced at the database level, not the application.

## Interview Traps

- **"What are ACID properties?"** — Atomicity (all or nothing), Consistency (valid state in/out), Isolation (concurrent transactions don't interfere), Durability (committed data survives crashes).
- **"What is a dirty read?"** — Reading uncommitted data from another transaction. If that transaction rolls back, you've read data that never "existed." Prevented by Read Committed and above.
- **"Default isolation level in SQL Server?"** — Read Committed. Prevents dirty reads but allows non-repeatable reads and phantom reads.
- **"Stored procedure advantages?"** — Reusability, encapsulation, can grant execute permission without granting table access. Downside: logic in the DB is hard to test and version-control.
- **"How do you prevent SQL injection?"** — Parameterized queries. Never concatenate user input into SQL strings. ORMs like EF Core parameterize automatically.
- **"Trigger vs EF Core interceptor?"** — Triggers live in the database (hard to test, version, debug). EF Core `SaveChangesInterceptor` handles the same use cases (audit logging, cascading) in testable C# code.
