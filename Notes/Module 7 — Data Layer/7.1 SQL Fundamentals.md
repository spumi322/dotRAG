# 7.1 SQL Fundamentals

## SQL command categories

SQL (Structured Query Language) is the standard language for managing and querying relational databases. Commands are grouped into five categories:

| Category | Full name | Commands | Purpose |
|---|---|---|---|
| **DDL** | Data Definition Language | `CREATE`, `ALTER`, `DROP` | Define/modify schema structure |
| **DML** | Data Manipulation Language | `SELECT`, `INSERT`, `UPDATE`, `DELETE` | Manipulate data in tables |
| **DQL** | Data Query Language | `SELECT` | Retrieve data (subset of DML) |
| **DCL** | Data Control Language | `GRANT`, `REVOKE` | Control access and permissions |
| **TCL** | Transaction Control Language | `COMMIT`, `ROLLBACK`, `SAVEPOINT` | Manage transactions |

```sql
-- DDL
CREATE TABLE Products (Id INT PRIMARY KEY, Name NVARCHAR(100), Price DECIMAL(10,2));
ALTER TABLE Products ADD CategoryId INT;
DROP TABLE Products;

-- DML
INSERT INTO Products (Name, Price) VALUES ('Widget', 9.99);
UPDATE Products SET Price = 12.99 WHERE Id = 1;
DELETE FROM Products WHERE Id = 1;

-- DCL
GRANT SELECT ON Products TO ReadOnlyUser;
REVOKE INSERT ON Products FROM ReadOnlyUser;

-- TCL
BEGIN TRANSACTION;
    UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 1;
    UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 2;
COMMIT;
```

## SQL data types

| Category | Common types | Notes |
|---|---|---|
| **Integer** | `INT`, `BIGINT`, `SMALLINT`, `TINYINT` | `INT` = 4 bytes, maps to `int` in C# |
| **Decimal** | `DECIMAL(p,s)`, `NUMERIC`, `MONEY` | Use `DECIMAL` for currency ‚Äî exact |
| **Float** | `FLOAT`, `REAL` | Approximate ‚Äî avoid for money |
| **String** | `NVARCHAR(n)`, `VARCHAR(n)`, `CHAR(n)` | `NVARCHAR` = Unicode (use this); `VARCHAR` = ASCII |
| **Date/Time** | `DATE`, `DATETIME2`, `DATETIMEOFFSET` | `DATETIME2` > `DATETIME` (more precision); `DATETIMEOFFSET` includes timezone |
| **Boolean** | `BIT` | 0/1, maps to `bool` in C# |
| **GUID** | `UNIQUEIDENTIFIER` | Maps to `Guid` in C# |
| **Binary** | `VARBINARY(MAX)` | Blobs, file storage |

> **`NVARCHAR` vs `VARCHAR`:** Prefer `NVARCHAR` for user-facing text ‚Äî it stores Unicode and handles international characters. `VARCHAR` is ASCII-only.

## Constraints

Constraints enforce data integrity at the database level:

| Constraint | Purpose | Example |
|---|---|---|
| `PRIMARY KEY` | Unique row identifier, not null | `Id INT PRIMARY KEY` |
| `FOREIGN KEY` | Links to another table's PK | `REFERENCES Orders(Id)` |
| `UNIQUE` | No duplicate values allowed | `UNIQUE (Email)` |
| `NOT NULL` | Column must have a value | `Name NVARCHAR(100) NOT NULL` |
| `CHECK` | Restricts allowed values | `CHECK (Price > 0)` |
| `DEFAULT` | Value when not provided | `DEFAULT GETDATE()` |
| `CASCADE` | Action when referenced row is updated/deleted | `ON DELETE CASCADE` |

```sql
CREATE TABLE Orders (
    Id         INT             PRIMARY KEY,
    CustomerId INT             NOT NULL REFERENCES Customers(Id) ON DELETE CASCADE,
    Amount     DECIMAL(10,2)   NOT NULL CHECK (Amount > 0),
    Status     NVARCHAR(20)    NOT NULL DEFAULT 'Pending',
    Email      NVARCHAR(200)   UNIQUE
);
```

## Normalization

Normalization reduces data redundancy and prevents anomalies. Each normal form builds on the previous one.

**The three anomalies normalization solves:**
- **Insert anomaly** ‚Äî can't add data without unrelated data existing first
- **Update anomaly** ‚Äî changing one fact requires updating multiple rows
- **Delete anomaly** ‚Äî deleting one record accidentally removes other facts

### 1NF ‚Äî First Normal Form

**Rule:** Every column holds a single atomic value. No repeating groups. Every row is unique (has a PK).

**Violation:** `Teams` column holds multiple values ‚Üí `"Alpha FC, Beta United"` in one cell.
**Fix:** Split into separate columns (`HomeTeam`, `AwayTeam`) or a separate table.

### 2NF ‚Äî Second Normal Form

**Rule:** Must be in 1NF. Every non-key column must depend on the **whole** primary key ‚Äî no partial dependencies.

> Only relevant when the PK is **composite** (multiple columns). Single-column PK ‚Üí 2NF is automatically satisfied.

**Violation:** Table with PK `(MatchId, TournamentId)` where `TournamentName` depends only on `TournamentId` (partial dependency).
**Fix:** Move `TournamentName` to a separate `Tournaments` table.

### 3NF ‚Äî Third Normal Form

**Rule:** Must be in 2NF. No **transitive dependencies** ‚Äî non-key columns must not depend on other non-key columns.

**Violation:** `Teams` table with `TeamId ‚Üí CityId ‚Üí CityName`. `CityName` depends on `CityId`, not the PK.
**Fix:** Extract `City` to its own table; `Teams` holds `CityId` FK.

### Quick reference

| Form | Requirement | Catches |
|---|---|---|
| 1NF | Atomic values, unique rows, PK exists | Multi-value columns |
| 2NF | 1NF + no partial dependencies | Depends on part of composite PK |
| 3NF | 2NF + no transitive dependencies | Non-key depends on non-key |

### Denormalization

Sometimes deliberate denormalization improves read performance:
- Pre-computed aggregates (e.g., store `StandingPoints` instead of summing match results every query)
- Reporting tables / materialized views

> Denormalization is a trade-off: write complexity for read performance. Target 3NF by default.

## Interview Traps

- **"What are the SQL command categories?"** ‚Äî DDL (schema definition), DML (data manipulation), DCL (access control), TCL (transactions). DQL is sometimes listed separately for SELECT.
- **"NVARCHAR vs VARCHAR?"** ‚Äî `NVARCHAR` stores Unicode (use for any user-facing text). `VARCHAR` is ASCII-only. In SQL Server, `NVARCHAR(n)` takes 2 bytes per character.
- **"Does 2NF matter with a single-column PK?"** ‚Äî No. Partial dependency requires a composite PK ‚Äî a single-column PK can't have partial dependencies.
- **"What's a transitive dependency?"** ‚Äî A ‚Üí B ‚Üí C where B is not a key. Fix: move B ‚Üí C to its own table.
- **"PRIMARY KEY vs UNIQUE constraint?"** ‚Äî PK = unique + not null + only one per table. UNIQUE = unique but allows null + multiple per table.
