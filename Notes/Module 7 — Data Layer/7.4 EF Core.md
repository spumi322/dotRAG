# 8.4 EF Core

## What is an ORM?

ORM (Object-Relational Mapper) bridges the gap between relational databases and object-oriented code. Instead of writing raw SQL, you work with C# classes and the ORM generates SQL.

```csharp
// Without ORM ‚Äî manual, verbose, error-prone
var cmd = new SqlCommand("SELECT * FROM Products WHERE Id = @id", conn);
cmd.Parameters.AddWithValue("@id", id);
var reader = cmd.ExecuteReader();
// manually map every column to a property...

// With EF Core ‚Äî one line
var product = await _context.Products.FindAsync(id);
```

**Advantages:** Productivity, LINQ support, migrations, change tracking, parameterized queries by default.
**Disadvantages:** Performance overhead on complex queries, less control than raw SQL for advanced scenarios.

## DbContext

`DbContext` is the central class in EF Core. It represents a **session with the database** and is responsible for:

- Managing the database connection
- Exposing `DbSet<T>` collections (one per entity)
- Tracking changes to entities
- Translating LINQ queries to SQL
- Executing queries and saving data via `SaveChanges()`
- Managing transactions (implicit per `SaveChanges()`)

```csharp
public class AppDbContext : DbContext
{
    public DbSet<Product> Products { get; set; }
    public DbSet<Order> Orders { get; set; }

    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // Fluent API configuration ‚Äî relationships, constraints, indexes
        modelBuilder.Entity<Order>()
            .HasMany(o => o.Items)
            .WithOne(i => i.Order)
            .HasForeignKey(i => i.OrderId);
    }
}
```

> `DbContext` must be registered as **Scoped** in ASP.NET Core ‚Äî one instance per HTTP request. Never register it as Singleton (captive dependency problem ‚Äî see [[6.3 Dependency Injection]]).

## Code-First Migrations

Migrations evolve the database schema incrementally as your model changes, without losing data. EF Core compares your current model to a stored snapshot and generates migration files.

```bash
dotnet ef migrations add AddProductCategory   # create migration file
dotnet ef database update                     # apply pending migrations to DB
dotnet ef migrations remove                   # undo last migration (if not applied to DB)
dotnet ef migrations script --idempotent      # generate SQL script for production deployment
```

Each migration has `Up()` (apply) and `Down()` (rollback):

```csharp
public partial class AddProductCategory : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.AddColumn<string>(name: "Category", table: "Products", nullable: true);
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropColumn(name: "Category", table: "Products");
    }
}
```

`__EFMigrationsHistory` table tracks which migrations have been applied.

**Best practices:**
- Always review generated migration code before applying
- Use `--idempotent` for production deployments (safe to re-run)
- Never edit the model snapshot manually
- Name migrations descriptively: `AddOrderStatus`, not `Migration3`

## Change Tracking

EF Core tracks every entity it loads so it knows what to persist on `SaveChanges()`.

**How it works (Snapshot tracking ‚Äî default):**
1. Entity is loaded ‚Üí EF takes a snapshot of original values
2. You modify properties
3. `SaveChanges()` ‚Üí EF compares current values to snapshot ‚Üí generates INSERT/UPDATE/DELETE

```csharp
var product = await _context.Products.FindAsync(id); // tracked, snapshot taken
product.Price = 19.99m;                               // EF detects the change
await _context.SaveChangesAsync();                    // UPDATE Products SET Price = 19.99 WHERE Id = @id
```

**Entity states:**

| State | Meaning |
|---|---|
| `Added` | New entity, will INSERT |
| `Modified` | Existing entity with changes, will UPDATE |
| `Deleted` | Marked for removal, will DELETE |
| `Unchanged` | Loaded, no changes |
| `Detached` | Not tracked by this context |

### AsNoTracking ‚Äî read-only performance

For read-only queries, disable tracking ‚Äî no snapshot is taken, less memory and CPU:

```csharp
// ‚úÖ Use for GET endpoints, reports, any query you won't modify
var products = await _context.Products
    .AsNoTracking()
    .Where(p => p.IsActive)
    .ToListAsync();
```

## Loading Related Data

### Eager loading ‚Äî preferred

Loads related data in one query with a JOIN:

```csharp
var orders = await _context.Orders
    .Include(o => o.Items)
    .ThenInclude(i => i.Product)
    .ToListAsync();
```

### Explicit loading ‚Äî on demand

Load a specific navigation property for an already-loaded entity:

```csharp
await _context.Entry(order).Collection(o => o.Items).LoadAsync();
```

### Lazy loading ‚Äî avoid in APIs

Automatically loads navigation properties on first access. Requires proxy configuration.

```csharp
// ‚öÝÔ∏è Avoid ‚Äî causes N+1 when accessed in a loop
var orders = await _context.Orders.ToListAsync();
foreach (var o in orders)
    Console.WriteLine(o.Items.Count); // triggers a separate query per order
```

---

## N+1 Query Problem

The most common EF Core performance mistake: loading a collection, then accessing a navigation property inside a loop.

```csharp
// ‚ùå N+1: 1 query for orders + N queries for items (one per order)
var orders = await _context.Orders.ToListAsync();
foreach (var o in orders)
    Console.WriteLine(o.Items.Count); // each access hits the database

// ‚úÖ Fixed: single query with JOIN via Include
var orders = await _context.Orders
    .Include(o => o.Items)
    .ToListAsync();
```

## Projections ‚Äî Select Only What You Need

Projections load only the columns needed for the response, reducing data transfer and memory:

```csharp
// ‚ùå Loads all columns into memory
var products = await _context.Products.ToListAsync();

// ‚úÖ Loads only Id, Name, Price ‚Äî generates SELECT Id, Name, Price FROM Products
var dtos = await _context.Products
    .Where(p => p.IsActive)
    .Select(p => new ProductDto(p.Id, p.Name, p.Price))
    .ToListAsync();
```

> Projections with `.Select()` also bypass change tracking automatically ‚Äî you don't need `.AsNoTracking()`.

## Transactions in EF Core

`SaveChangesAsync()` wraps all pending changes in a **single implicit transaction** automatically.

```csharp
// Implicit transaction ‚Äî covers all pending changes in one SaveChanges call
_context.Orders.Add(order);
_context.Invoices.Add(invoice);
await _context.SaveChangesAsync(); // both are committed or both rolled back
```

For multi-step operations requiring multiple `SaveChanges` calls in one atomic unit:

```csharp
// Explicit transaction
await using var tx = await _context.Database.BeginTransactionAsync();
try
{
    _context.Orders.Add(order);
    await _context.SaveChangesAsync();

    _context.Invoices.Add(invoice);
    await _context.SaveChangesAsync();

    await tx.CommitAsync();
}
catch
{
    await tx.RollbackAsync();
    throw;
}
```

## Optimistic Concurrency

**The problem:** two users load the same entity, both modify it, last write wins and silently overwrites the first change.

**The solution:** attach a `RowVersion` timestamp to the entity. EF includes it in every `UPDATE WHERE` clause ‚Äî if the row changed since you loaded it, the update affects 0 rows and EF throws `DbUpdateConcurrencyException`.

```csharp
public class Tournament
{
    public long Id { get; set; }
    public string Name { get; set; }

    [Timestamp]
    public byte[] RowVersion { get; set; }  // auto-incremented by SQL Server on every update
}
```

EF generates:
```sql
UPDATE Tournaments SET Name = @name
WHERE Id = @id AND RowVersion = @originalRowVersion  -- fails if someone else updated it
```

```csharp
try
{
    await _context.SaveChangesAsync();
}
catch (DbUpdateConcurrencyException)
{
    throw new ConflictException("The record was modified by another user. Please retry.");
}
```

> "Optimistic concurrency assumes conflicts are rare ‚Äî it doesn't lock the row, but detects conflicts at save time using a rowversion token. If a conflict occurs, EF throws `DbUpdateConcurrencyException`."

## Raw SQL (Escape Hatch)

For complex queries that EF Core can't express efficiently:

```csharp
// FromSqlInterpolated ‚Äî parameterized, safe from SQL injection
var products = await _context.Products
    .FromSqlInterpolated($"SELECT * FROM Products WHERE Category = {category}")
    .ToListAsync();

// Call a stored procedure
var orders = await _context.Orders
    .FromSqlRaw("EXEC GetOrdersByCustomer @CustomerId = {0}", customerId)
    .ToListAsync();
```

> Always use `FromSqlInterpolated` (not `FromSqlRaw` with string concatenation) to stay safe from SQL injection.

## Interview Traps

- **"What is DbContext?"** ‚Äî Represents a session with the database: manages connection, tracks entity changes, translates LINQ to SQL, wraps SaveChanges in a transaction.
- **"What is change tracking?"** ‚Äî EF snapshots entity values on load. On `SaveChanges()`, it diffs current vs snapshot to generate the minimal INSERT/UPDATE/DELETE. Use `AsNoTracking()` to skip it for read-only queries.
- **"What is the N+1 problem?"** ‚Äî Loading a collection, then accessing a navigation property in a loop triggers one query per item. Fix: use `Include()` to load it in a single JOIN query.
- **"Eager vs Lazy loading?"** ‚Äî Eager (`Include`) loads related data upfront in one query. Lazy loads on property access ‚Äî bad in APIs, causes N+1. Prefer eager loading.
- **"When do you use an explicit transaction?"** ‚Äî When you need multiple `SaveChanges` calls to succeed or fail together as one atomic unit.
- **"What is optimistic concurrency?"** ‚Äî No row locking. A rowversion stamp is added to the UPDATE WHERE clause ‚Äî if it doesn't match, the update fails and `DbUpdateConcurrencyException` is thrown. Assumes conflicts are rare.
