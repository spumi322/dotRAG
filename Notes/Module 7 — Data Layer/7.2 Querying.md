# 7.2 Querying

## SQL Joins

Joins combine rows from two or more tables based on a related column.

```
INNER JOIN          LEFT JOIN           RIGHT JOIN          FULL JOIN
  L ∩ R              L + (L ∩ R)         R + (L ∩ R)         L ∪ R

 [L | R]           [██| R]             [L |██]             [██|██]
     ↑                  ↑                   ↑                   ↑
 only overlap      all left,           all right,          everything,
                   nulls for           nulls for           nulls where
                   no match            no match            no match
```

### INNER JOIN

Returns only rows with a match in **both** tables.

```sql
-- Teams registered to at least one tournament
SELECT t.Name, tr.Name AS Tournament
FROM Teams t
INNER JOIN TournamentTeams tt ON t.Id = tt.TeamId
INNER JOIN Tournaments tr     ON tt.TournamentId = tr.Id;
```

> If a team has no tournament registrations, it won't appear. Most common join — use when both sides must have matching rows.

### LEFT JOIN

Returns **all rows from the left table**. Right side is `NULL` where no match exists.

```sql
-- All teams, including those not in any tournament
SELECT t.Name, tr.Name AS Tournament
FROM Teams t
LEFT JOIN TournamentTeams tt ON t.Id = tt.TeamId
LEFT JOIN Tournaments tr     ON tt.TournamentId = tr.Id;

-- Find records with no match: WHERE tr.Id IS NULL
```

### RIGHT JOIN / FULL OUTER JOIN

- **RIGHT JOIN:** All rows from right table. Rarely used — swap tables and use LEFT JOIN instead.
- **FULL OUTER JOIN:** All rows from both tables, `NULL` on the side without a match. Useful for reconciliation audits. Not supported in MySQL (simulate with `LEFT JOIN UNION RIGHT JOIN`).

### SELF JOIN

A table joined to itself. Useful for hierarchical or comparative data.

```sql
-- Find matches where the same team plays both home and away
SELECT m1.Id
FROM Matches m1
INNER JOIN Matches m2 ON m1.HomeTeamId = m2.AwayTeamId
                      AND m1.AwayTeamId = m2.HomeTeamId
                      AND m1.Id <> m2.Id;
```

## Aggregate Functions

Aggregate functions operate on a set of rows and return a single value.

```sql
SELECT
    COUNT(*)           AS TotalOrders,      -- count all rows
    COUNT(Amount)      AS NonNullAmounts,   -- count non-null values
    SUM(Amount)        AS TotalRevenue,
    AVG(Amount)        AS AverageOrder,
    MIN(Amount)        AS SmallestOrder,
    MAX(Amount)        AS LargestOrder
FROM Orders;
```

> `COUNT(*)` counts all rows including NULLs. `COUNT(column)` skips NULL values.

## GROUP BY and HAVING

`GROUP BY` groups rows sharing the same value so aggregate functions can be applied per group.

```sql
-- Orders per customer
SELECT CustomerId, COUNT(*) AS OrderCount
FROM Orders
GROUP BY CustomerId;

-- High-value customers only (filter after aggregation)
SELECT CustomerId, SUM(Amount) AS Total
FROM Orders
WHERE Status = 'Shipped'        -- 1. filter individual rows
GROUP BY CustomerId             -- 2. group
HAVING SUM(Amount) > 1000;      -- 3. filter groups by aggregate
```

### WHERE vs HAVING

| | WHERE | HAVING |
|---|---|---|
| Filters | Individual rows | Groups (after aggregation) |
| Runs | Before GROUP BY | After GROUP BY |
| Can use aggregates | ❌ No | ✅ Yes |

> Rule: if you can write it in WHERE, do — it reduces rows before grouping (cheaper).

### Query execution order

```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
```

This is why you **cannot use a SELECT alias in HAVING** — HAVING runs before SELECT is evaluated.

```sql
-- ❌ Invalid — alias 'OrderCount' not yet defined at HAVING stage
HAVING OrderCount > 5;

-- ✅ Correct
HAVING COUNT(*) > 5;
```

## Views

A view is a virtual table based on a SELECT query. It simplifies complex queries and provides an abstraction layer.

```sql
-- Create
CREATE VIEW CustomerOrderSummary AS
SELECT CustomerId, COUNT(OrderId) AS OrderCount, SUM(Amount) AS Total
FROM Orders
GROUP BY CustomerId;

-- Use like a table
SELECT * FROM CustomerOrderSummary WHERE OrderCount > 5;
```

**Use cases:**
- Encapsulate complex joins/aggregations behind a simple name
- Security — expose only specific columns/rows to certain roles
- Simplify reporting queries
- Shield applications from schema changes (schema migration layer)

> Views don't store data — they re-execute the query each time. Use **indexed views** (materialized views) in SQL Server for performance-critical aggregations.

## Indexes

An index is a data structure that speeds up row lookups at the cost of extra storage and slower writes.

### Clustered vs Non-Clustered

| | Clustered | Non-Clustered |
|---|---|---|
| Stores | Actual data rows, sorted by key | Separate structure with pointers to rows |
| Per table | **Only one** | Many (up to ~999 in SQL Server) |
| Default | Primary key | All other indexes |
| Speed | Fastest for range scans | Slightly slower (extra pointer lookup) |

```sql
-- PK creates a clustered index automatically
CREATE TABLE Orders (OrderId INT PRIMARY KEY, CustomerId INT, Amount DECIMAL);

-- Non-clustered: separate lookup structure
CREATE INDEX IX_Orders_CustomerId ON Orders (CustomerId);
```

> "A clustered index determines the physical storage order of the table — there can only be one. A non-clustered index is a separate structure storing key + pointer to the row."

### When to index

Index columns that appear in:
- `WHERE` clauses (filtering)
- `JOIN` conditions
- `ORDER BY` / `GROUP BY`

### Index trade-offs

- ✅ Speed up reads
- ❌ Slow down writes (every INSERT/UPDATE/DELETE must update the index)
- ❌ Consume disk space

> Don't index every column. Index based on actual query patterns from profiling.

### Other index types

- **Composite** — multiple columns: `CREATE INDEX IX ON Orders (CustomerId, Status)`. Column order matters — most selective first.
- **Unique** — enforces uniqueness like a constraint
- **Covering** — includes all columns a query needs so it never touches the main table (`INCLUDE` clause in SQL Server)

## Interview Traps

- **"INNER vs LEFT — when to use which?"** — INNER when both sides must have a match; LEFT when you need all rows from one side regardless of whether there's a match.
- **"Why avoid RIGHT JOIN?"** — Equivalent to LEFT JOIN with tables swapped. LEFT is the convention — easier to read and reason about.
- **"WHERE vs HAVING?"** — WHERE filters rows before grouping; HAVING filters groups after aggregation. HAVING can reference aggregates, WHERE cannot.
- **"Clustered vs non-clustered index?"** — Clustered = physical row order, only one per table (PK). Non-clustered = separate pointer structure, many allowed.
- **"When should you NOT add an index?"** — Small tables (full scan is faster), columns with very low cardinality (few distinct values), write-heavy tables where index maintenance cost outweighs read benefit.
- **"NULL in JOIN column?"** — `NULL != NULL` in SQL — rows with NULL keys never match in any JOIN type.
