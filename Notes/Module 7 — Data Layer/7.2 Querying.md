# 7.2 Querying

## SQL Joins

Joins combine rows from two or more tables based on a related column.

```
INNER JOIN          LEFT JOIN           FULL OUTER JOIN
 [L | R]           [██| R]             [██|██]
     ↑                  ↑                   ↑
 only overlap      all left,           everything,
                   nulls for           nulls where
                   no match            no match
```

### INNER JOIN

Returns only rows with a match in **both** tables. Most common join.

```sql
-- Teams registered to at least one tournament
SELECT t.Name, tr.Name AS Tournament
FROM Teams t
INNER JOIN TournamentTeams tt ON t.Id = tt.TeamId
INNER JOIN Tournaments tr     ON tt.TournamentId = tr.Id;
```

If a team has no registrations, it won't appear. Use when both sides must have matching rows.

### LEFT JOIN

Returns **all rows from the left table**. Right side is `NULL` where no match exists.

```sql
-- All teams, including those not in any tournament
SELECT t.Name, tr.Name AS Tournament
FROM Teams t
LEFT JOIN TournamentTeams tt ON t.Id = tt.TeamId
LEFT JOIN Tournaments tr     ON tt.TournamentId = tr.Id;

-- Find unregistered teams: filter for NULL on the right side
SELECT t.Name
FROM Teams t
LEFT JOIN TournamentTeams tt ON t.Id = tt.TeamId
WHERE tt.TeamId IS NULL;
```

### RIGHT JOIN / FULL OUTER JOIN

- **RIGHT JOIN** — all rows from right table. Rarely used in practice — just swap the tables and use LEFT JOIN instead.
- **FULL OUTER JOIN** — all rows from both sides, `NULL` where no match. Useful for reconciliation (finding mismatches between two datasets).

## Aggregate Functions

Operate on a set of rows and return a single value.

```sql
SELECT
    COUNT(*)        AS TotalMatches,     -- counts all rows including NULLs
    COUNT(Score)    AS ScoredMatches,    -- counts only non-NULL values
    SUM(Score)      AS TotalGoals,
    AVG(Score)      AS AvgGoals,
    MIN(Score)      AS LowestScore,
    MAX(Score)      AS HighestScore
FROM Matches;
```

> `COUNT(*)` vs `COUNT(column)` — the most common trap. `COUNT(*)` counts every row; `COUNT(column)` skips NULLs.

## GROUP BY and HAVING

`GROUP BY` collapses rows sharing the same value so aggregates apply per group.

```sql
-- Goals per team
SELECT TeamId, SUM(Score) AS TotalGoals
FROM Matches
GROUP BY TeamId;

-- Only teams with more than 10 total goals
SELECT TeamId, SUM(Score) AS TotalGoals
FROM Matches
WHERE Status = 'Completed'          -- 1. filter individual rows first
GROUP BY TeamId                     -- 2. group
HAVING SUM(Score) > 10;             -- 3. filter groups by aggregate result
```

### WHERE vs HAVING

||WHERE|HAVING|
|---|---|---|
|Filters|Individual rows|Groups (after aggregation)|
|Runs|Before GROUP BY|After GROUP BY|
|Can use aggregates|❌ No|✅ Yes|

Rule: if you can express it in WHERE, do — it reduces rows before grouping, which is cheaper.

### Query execution order

```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
```

This is why you can't reference a SELECT alias in HAVING — HAVING runs before SELECT is evaluated.

```sql
-- ❌ Invalid — 'TotalGoals' alias doesn't exist yet at HAVING stage
HAVING TotalGoals > 10;

-- ✅ Correct — repeat the aggregate expression
HAVING SUM(Score) > 10;
```

## Indexes

An index is a data structure that speeds up row lookups at the cost of extra storage and slower writes.

### Clustered vs Non-Clustered

||Clustered|Non-Clustered|
|---|---|---|
|Stores|Actual data rows, sorted by key|Separate structure with pointers to rows|
|Per table|**Only one**|Many|
|Default|Primary key|All other indexes|
|Best for|Range scans, PK lookups|Filtering/joining on non-PK columns|

```sql
-- PK creates a clustered index automatically
CREATE TABLE Matches (
    Id       INT PRIMARY KEY,   -- clustered index created here
    TeamId   INT,
    Score    INT
);

-- Non-clustered: separate structure for fast TeamId lookups
CREATE INDEX IX_Matches_TeamId ON Matches (TeamId);
```

### When to add an index

Index columns that appear frequently in:

- `WHERE` filters
- `JOIN` conditions
- `ORDER BY` / `GROUP BY`

### Index trade-offs

Adding an index is not free:

- ✅ Speeds up reads
- ❌ Slows down writes — every INSERT / UPDATE / DELETE must update the index too
- ❌ Consumes storage

Don't index every column. Profile first, index based on actual query patterns.

## Interview Traps

- **"INNER vs LEFT JOIN — when to use which?"** — INNER when both sides must have a match. LEFT when you need all rows from one side regardless of whether the other side matches (e.g. "all teams, even those without matches").
- **"Why avoid RIGHT JOIN?"** — It's equivalent to LEFT JOIN with tables swapped. Convention is LEFT — easier to read and reason about.
- **"WHERE vs HAVING?"** — WHERE filters rows before grouping; HAVING filters groups after aggregation. HAVING can reference aggregates, WHERE cannot.
- **"`COUNT(*)` vs `COUNT(column)`?"** — `COUNT(*)` counts all rows. `COUNT(column)` skips NULLs. They can return different results on the same table.
- **"Clustered vs non-clustered index?"** — Clustered determines the physical row order — only one per table, defaults to the PK. Non-clustered is a separate pointer structure — you can have many.
- **"When should you NOT add an index?"** — Small tables (full scan is faster), low-cardinality columns (e.g. a boolean flag), write-heavy tables where maintenance cost outweighs read benefit.
- **"NULL in a JOIN condition?"** — `NULL != NULL` in SQL. Rows with NULL keys never match in any join type.
- **"What's a subquery?"** — A query nested inside another query. Usually in a WHERE or FROM clause. In most cases EF Core handles these for you via LINQ, but knowing they exist matters for reading query plans or writing raw SQL.
