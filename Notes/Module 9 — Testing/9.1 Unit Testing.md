# 9.1 Unit Testing

## Testing pyramid — scope and purpose

| Level | Scope | Speed | Cost | Tools |
|---|---|---|---|---|
| **Unit** | Single class/method in isolation | Very fast | Low | xUnit, Moq |
| **Integration** | Multiple components wired together | Moderate | Medium | WebApplicationFactory |
| **E2E** | Full system through UI/HTTP | Slow | High | Playwright, Selenium |

**Unit tests** — isolate the class under test. All external dependencies (DB, HTTP, time) are replaced with fakes/mocks.
**Integration tests** — test how components work together (controller → service → real DB, or HTTP endpoint → response).
**E2E tests** — simulate real user interactions in a browser.

> Aim for many fast unit tests, fewer integration tests, very few E2E tests. Expensive tests slow CI and are fragile.

## xUnit basics

xUnit is the default test framework for .NET. Tests are discovered by the test runner based on attributes.

```csharp
// [Fact] — a single test case with no parameters
public class CalculatorTests
{
    [Fact]
    public void Add_TwoPositiveNumbers_ReturnsSum()
    {
        // Arrange
        var calc = new Calculator();

        // Act
        var result = calc.Add(2, 3);

        // Assert
        Assert.Equal(5, result);
    }
}
```

### [Theory] and [InlineData] — parameterized tests

`[Theory]` runs the same test with multiple input sets. Each `[InlineData]` becomes a separate test case.

```csharp
[Theory]
[InlineData(2,  3,  5)]
[InlineData(0,  0,  0)]
[InlineData(-1, 1,  0)]
[InlineData(-2, -3, -5)]
public void Add_VariousInputs_ReturnsCorrectSum(int a, int b, int expected)
{
    var calc = new Calculator();
    var result = calc.Add(a, b);
    Assert.Equal(expected, result);
}
```

**Other data sources:**
```csharp
[MemberData(nameof(AddTestCases))]  // data from a static property/method
[ClassData(typeof(AddTestData))]    // data from a class implementing IEnumerable
```

## AAA pattern — Arrange / Act / Assert

The universal structure for readable, maintainable tests:

```
Arrange — set up inputs, dependencies, and expected values
Act     — call the method under test (one action per test)
Assert  — verify the outcome
```

```csharp
[Fact]
public async Task RegisterTeam_ValidRequest_ReturnsCreatedTeam()
{
    // Arrange
    var mockRepo = new Mock<ITeamRepository>();
    var service  = new TeamService(mockRepo.Object);
    var request  = new RegisterTeamRequest("Alpha FC", "Budapest");

    mockRepo
        .Setup(r => r.AddAsync(It.IsAny<Team>(), default))
        .ReturnsAsync(new Team { Id = 1, Name = "Alpha FC" });

    // Act
    var result = await service.RegisterAsync(request);

    // Assert
    Assert.Equal("Alpha FC", result.Name);
    Assert.Equal(1, result.Id);
}
```

## Test naming convention

**Pattern:** `MethodName_Scenario_ExpectedResult`

```
Add_TwoPositiveNumbers_ReturnsSum
GetTournament_TournamentNotFound_ThrowsNotFoundException
RegisterTeam_DuplicateName_ReturnsBadRequest
SaveChanges_DbThrows_RollsBackTransaction
```

A well-named test is self-documenting — it tells you what broke without reading the body.

## Moq — mocking dependencies

Moq creates fake implementations of interfaces. Use it to isolate the class under test from its dependencies.

### Setup — define behaviour

```csharp
var mockRepo = new Mock<ITeamRepository>();

// Setup a return value
mockRepo
    .Setup(r => r.GetByIdAsync(1, default))
    .ReturnsAsync(new Team { Id = 1, Name = "Alpha FC" });

// Setup with any argument (argument matcher)
mockRepo
    .Setup(r => r.GetByIdAsync(It.IsAny<int>(), It.IsAny<CancellationToken>()))
    .ReturnsAsync(new Team { Id = 99 });

// Setup to throw
mockRepo
    .Setup(r => r.GetByIdAsync(999, default))
    .ThrowsAsync(new NotFoundException("Team not found"));
```

### ReturnsAsync vs Returns

```csharp
// For async methods (Task<T>)
.ReturnsAsync(value)

// For sync methods
.Returns(value)

// For void async (Task)
.Returns(Task.CompletedTask)
```

### Verify — assert that a method was called

```csharp
// Verify called exactly once with specific args
mockRepo.Verify(r => r.AddAsync(It.IsAny<Team>(), default), Times.Once);

// Verify never called
mockRepo.Verify(r => r.DeleteAsync(It.IsAny<int>(), default), Times.Never);

// Verify called N times
mockRepo.Verify(r => r.SaveAsync(default), Times.Exactly(2));
```

### Full example — service test with mock

```csharp
public class TournamentServiceTests
{
    private readonly Mock<ITournamentRepository> _mockRepo;
    private readonly TournamentService _service;

    public TournamentServiceTests()
    {
        _mockRepo = new Mock<ITournamentRepository>();
        _service  = new TournamentService(_mockRepo.Object);
    }

    [Fact]
    public async Task GetTournamentAsync_Exists_ReturnsTournament()
    {
        // Arrange
        var tournament = new Tournament { Id = 1, Name = "Spring Cup" };
        _mockRepo
            .Setup(r => r.GetByIdAsync(1, default))
            .ReturnsAsync(tournament);

        // Act
        var result = await _service.GetTournamentAsync(1);

        // Assert
        Assert.Equal("Spring Cup", result.Name);
        _mockRepo.Verify(r => r.GetByIdAsync(1, default), Times.Once);
    }

    [Fact]
    public async Task GetTournamentAsync_NotFound_ThrowsNotFoundException()
    {
        // Arrange
        _mockRepo
            .Setup(r => r.GetByIdAsync(999, default))
            .ReturnsAsync((Tournament?)null);

        // Act & Assert — exception thrown
        await Assert.ThrowsAsync<NotFoundException>(
            () => _service.GetTournamentAsync(999));
    }
}
```

## Common xUnit assertions

```csharp
Assert.Equal(expected, actual);             // value equality
Assert.NotEqual(expected, actual);
Assert.True(condition);
Assert.False(condition);
Assert.Null(value);
Assert.NotNull(value);
Assert.Empty(collection);
Assert.Contains(item, collection);
Assert.IsType<Tournament>(result);          // exact type
Assert.IsAssignableFrom<ITournament>(obj);  // type or derived
await Assert.ThrowsAsync<NotFoundException>(() => sut.GetAsync(999));
```

## Interview Traps

- **"Unit vs Integration test?"** — Unit tests isolate one class, replacing all dependencies with mocks. Integration tests wire multiple real components together. Unit tests are faster and pinpoint failures precisely; integration tests catch component interaction bugs.
- **"What is AAA?"** — Arrange (set up), Act (invoke the method under test), Assert (verify outcome). One `Act` per test keeps tests focused and failure messages clear.
- **"Why mock dependencies?"** — To isolate the class under test. Tests don't need a real database, HTTP client, or email server — mocks let you define exact behaviour and assert exactly which calls were made.
- **"`[Fact]` vs `[Theory]`?"** — `[Fact]` is a single test case. `[Theory]` is parameterized — the same test logic runs with each `[InlineData]` / `[MemberData]` row, reducing duplication.
- **"Moq `Setup` vs `Verify`?"** — `Setup` defines what the mock returns when called. `Verify` asserts the mock was called in a specific way. Use both: setup to control the scenario, verify to assert expected interactions.
