# 5.4 Exceptions

## What exceptions are

An exception is an event that disrupts normal program flow, signalling that something unexpected happened. Handled with `try`/`catch`/`finally`.

```csharp
try
{
    int divisor = 0;
    int result = 10 / divisor; // throws DivideByZeroException
}
catch (DivideByZeroException ex)
{
    Console.WriteLine("Error: " + ex.Message);
}
finally
{
    Console.WriteLine("Always runs ‚Äî cleanup goes here");
}
```

---

## `finally` ‚Äî guaranteed execution

`finally` runs whether or not an exception is thrown ‚Äî even if the `try` block returns early. Use it for cleanup that must always happen.

```csharp
StreamReader? reader = null;
try
{
    reader = new StreamReader("file.txt");
    return reader.ReadToEnd(); // returns here ‚Äî finally still runs
}
catch (FileNotFoundException ex)
{
    Console.WriteLine(ex.Message);
    return string.Empty;
}
finally
{
    reader?.Dispose(); // always cleans up
}
```

> Prefer `using` / `using var` over manual `finally` for `IDisposable` cleanup ‚Äî same guarantee, less code.

---

## `throw` vs `throw ex` ‚Äî the #1 rethrow trap

```csharp
// ‚ùå throw ex ‚Äî RESETS the stack trace
catch (Exception ex)
{
    Log(ex);
    throw ex; // stack trace now points to THIS line ‚Äî you lose the original call stack
}

// ‚úÖ throw ‚Äî PRESERVES the original stack trace
catch (Exception ex)
{
    Log(ex);
    throw; // rethrows the original exception with original stack trace intact
}
```

> **Always use bare `throw`** (no exception variable) when rethrowing. `throw ex` makes debugging much harder because you lose where the exception actually originated.

---

## Rethrowing as a different type

When you catch one exception and want to surface a different, higher-level exception:

```csharp
catch (SqlException ex)
{
    // Wrap ‚Äî original exception is preserved as InnerException
    throw new DataAccessException("Failed to load tournament", ex);
}
```

The `InnerException` lets you keep the original cause available for logging and debugging.

---

## Exception filters (`when`)

Run additional logic before deciding whether to catch:

```csharp
try
{
    // code that might throw
}
catch (HttpRequestException ex) when (ex.StatusCode == HttpStatusCode.NotFound)
{
    // only catches 404 HttpRequestExceptions
}
catch (HttpRequestException ex)
{
    // catches all other HttpRequestExceptions
}
```

`when` evaluates before the catch block runs. If `when` returns false, the exception propagates as if the catch didn't exist ‚Äî the stack trace is NOT reset.

## Custom exceptions

```csharp
// ‚úÖ Correct custom exception structure
public class TournamentNotFoundException : Exception
{
    public long TournamentId { get; }

    public TournamentNotFoundException(long id)
        : base($"Tournament {id} not found.")
    {
        TournamentId = id;
    }

    // Recommended: also provide message + inner exception constructor
    public TournamentNotFoundException(long id, Exception inner)
        : base($"Tournament {id} not found.", inner)
    {
        TournamentId = id;
    }
}

// Usage
throw new TournamentNotFoundException(tournamentId);
```

**Best practices:**
- Inherit from `Exception` directly
- Name them `*Exception`
- Provide at minimum: message constructor + message+inner constructor
- Add domain-specific properties (`TournamentId`) for programmatic handling in middleware

## How exceptions propagate

Exceptions travel up the call stack until caught or until they reach the top (crashing the process).

```
ASP.NET pipeline          ‚Üê GlobalExceptionHandler catches here
  ‚Üì no catch
TournamentController
  ‚Üì no catch
RegisterTeam()
  ‚Üì no catch
ValidateSlot()            ‚Üê throws ValidationException
```

In ASP.NET Core, global exception handling middleware catches unhandled exceptions at the pipeline level and returns structured `ProblemDetails` responses.

## Common exception types to know

| Exception | Thrown when |
|---|---|
| `ArgumentNullException` | Null passed where not allowed |
| `ArgumentOutOfRangeException` | Argument outside valid range |
| `InvalidOperationException` | Method call invalid for current state |
| `NullReferenceException` | Accessing member on null reference |
| `NotImplementedException` | Method stub not yet implemented |
| `NotSupportedException` | Operation not supported by this type |
| `KeyNotFoundException` | Dictionary key not found |
| `TimeoutException` | Operation exceeded time limit |

## Interview Traps

- **"`throw` vs `throw ex`?"** ‚Äî `throw` preserves the original stack trace. `throw ex` resets it to the current line. Always use bare `throw` when rethrowing.
- **"When does `finally` NOT run?"** ‚Äî When the process is killed (`Environment.FailFast()`), a `StackOverflowException`, or an unhandled fatal CLR error. In normal flow, `finally` always runs.
- **"Can you catch multiple exception types in one block?"** ‚Äî Yes, with `catch (ExA or ExB ex)` syntax (C# 6+), or by catching separate blocks.
- **"Difference between `Exception` and `ApplicationException`?"** ‚Äî `ApplicationException` is an older, unused marker class. There is no meaningful distinction today. Derive from `Exception` directly.
- **"Why use `when` in catch?"** ‚Äî More specific handling without fully entering the catch block. Crucially, if `when` returns false, the stack trace is not corrupted ‚Äî the exception propagates cleanly.
