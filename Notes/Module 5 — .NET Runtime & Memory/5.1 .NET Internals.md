# 5.1 .NET Internals

## Source to Native: the pipeline

```
C# / F# / VB.NET source code
        ↓ Roslyn compiler
   CIL bytecode + metadata
   (packaged into an Assembly .dll/.exe)
        ↓ CLR loads the Assembly
   JIT compiler translates CIL → native
   (method by method, on first call)
        ↓
   Native machine code executes
```

---

## CLR — Common Language Runtime

The runtime environment that manages execution of all .NET code. Acts as a layer between the OS and the application.

**Responsibilities:**
- **JIT compilation** — translates CIL to native code for the current platform
- **Memory management** — allocates heap memory for objects; hosts the Garbage Collector
- **Exception handling** — redirects execution to appropriate catch blocks
- **Thread management** — manages thread pool, synchronisation
- **Type safety** — enforces type rules at runtime via the CTS

All code running under the CLR is called **managed code**. Code that runs outside the CLR (C++, Win32 API calls) is **unmanaged code**.

---

## CIL — Common Intermediate Language

The intermediate bytecode that all .NET languages compile to.

- C#, F#, VB.NET all compile to the same CIL
- CIL is **platform-independent** — it describes operations abstractly
- The CLR's JIT compiler translates CIL to **platform-specific native code** just before execution
- This is why .NET code can run on Windows, macOS, and Linux from the same compiled assembly

```
C# source → CIL (cross-platform) → native machine code (platform-specific, at runtime)
```

---

## JIT — Just-In-Time Compilation

The CLR compiles CIL to native code **on the first call** of each method:

1. First time a method is called → JIT compiles it → native code is cached
2. Subsequent calls → cached native code runs directly (no JIT overhead)
3. JIT can apply platform-specific optimisations unavailable at compile time

**AOT (Ahead-of-Time):** .NET 7+ supports Native AOT — compiles directly to native without JIT, for smaller binaries and faster startup (no warmup cost). Trade-off: less runtime optimisation flexibility.

---

## CTS — Common Type System

Defines a standard for all type definitions across .NET languages. Ensures that `int` in C# is the same thing as `Integer` in VB.NET — enabling cross-language interoperability. Every type ultimately derives from `System.Object`.

---

## BCL — Base Class Library

The foundational library of pre-built types and APIs included with the CLR:

- Collections (`List<T>`, `Dictionary<K,V>`)
- File I/O (`FileStream`, `StreamReader`)
- Networking (`HttpClient`)
- Threading (`Task`, `SemaphoreSlim`)
- String/DateTime/Math utilities

You use the BCL constantly without thinking about it.

---

## Assemblies

An assembly (`.dll` or `.exe`) is the unit of deployment in .NET. It contains:

- **CIL bytecode** — compiled code
- **Metadata** — type information (names, members, attributes)
- **Manifest** — assembly identity (name, version, dependencies)

An assembly may contain one or more namespaces. The CLR loads assemblies on demand.

## .NET Framework vs .NET Core / .NET 5+

| | .NET Framework | .NET Core → .NET 5+ |
|---|---|---|
| Platform | Windows only | Cross-platform (Win/macOS/Linux) |
| Open source | ❌ | ✅ |
| Performance | Moderate | Higher (rewritten runtime) |
| Deployment | Installed on machine | Self-contained possible |
| Status | Legacy (4.8 is final) | Active (current: .NET 9/10) |
| Desktop (WinForms/WPF) | ✅ | ✅ (added in .NET 3+) |

> **Interview answer:** ".NET Framework is legacy Windows-only. .NET Core was the cross-platform rewrite. Since .NET 5 they unified under a single brand. We're on .NET 9/10 now."

## Reflection

Reflection lets you inspect and interact with types **at runtime** — read class names, list properties, create instances, invoke methods — without knowing the type at compile time.

```csharp
Type type = typeof(Tournament);

// Inspect members
var props = type.GetProperties();
var methods = type.GetMethods();

// Invoke dynamically
MethodInfo method = type.GetMethod("Cancel");
method.Invoke(obj, null);

// Read attributes
var attr = type.GetCustomAttribute<AuditableAttribute>();
```

**How frameworks use reflection internally:**
- **ASP.NET Core model binding** — reads properties to populate controller parameters
- **EF Core** — reads properties to map entities to columns
- **AutoMapper** — maps property values by name
- **xUnit** — discovers `[Fact]` / `[Theory]` methods

In day-to-day web dev: you use libraries that use reflection internally. You rarely write reflection code directly.

> Reflection is expensive — avoid it in hot paths. Frameworks cache reflection results to mitigate this.

## Interview Traps

- **"What is the CLR?"** — The runtime that manages execution of .NET code: JIT compilation, GC, type safety, exception handling, thread management.
- **"What is CIL and why does it exist?"** — The intermediate bytecode that all .NET languages compile to. Makes .NET cross-language and cross-platform — the JIT translates it to native code for the current OS/CPU.
- **"What's the difference between .NET Framework and .NET Core?"** — Framework is legacy Windows-only. Core is the cross-platform rewrite. Since .NET 5 they merged into a single "modern .NET."
- **"How does reflection work?"** — It reads type metadata from the assembly at runtime, allowing you to inspect and interact with types without compile-time knowledge of them.
- **"What's managed vs unmanaged code?"** — Managed = runs under CLR control (memory managed by GC). Unmanaged = runs outside CLR (C++, OS APIs — you manage memory manually).
