# 5.3 Async & Concurrency

## Multi-threading vs Async — the core distinction

| | Multi-threading | Async/await |
|---|---|---|
| Goal | **Parallelism** — do multiple things simultaneously on multiple cores | **Scalability** — don't block threads while waiting for I/O |
| Use case | CPU-bound work (calculations, image processing) | I/O-bound work (DB queries, HTTP calls, file reads) |
| Mechanism | `Thread`, `Task.Run`, `Parallel.ForEach` | `async`/`await`, `Task<T>` |

```csharp
// CPU-bound → use threads
Parallel.ForEach(images, img => ProcessImage(img));

// I/O-bound → use async/await (thread is FREE while waiting)
var data = await _httpClient.GetStringAsync(url);
```

---

## async / await basics

`async` marks a method as asynchronous. `await` suspends the method without blocking the thread until the awaited operation completes. The compiler transforms the method into a state machine.

```csharp
public async Task<Tournament?> GetTournamentAsync(long id)
{
    // Thread is NOT blocked while the DB query runs
    var tournament = await _context.Tournaments.FindAsync(id);
    return tournament;
}
```

- `async` alone does not make a method asynchronous — it just enables `await` inside it
- A method returning `Task` without awaiting anything is still synchronous in practice

---

## Task\<T\> and ValueTask\<T\>

**`Task<T>`** — represents an in-progress or completed operation. The standard return type for async methods.

```csharp
public async Task<string> GetNameAsync() { ... }
Task<string> task = GetNameAsync();
string name = await task;
```

**`ValueTask<T>`** — avoids heap allocation when the result is already available (synchronous path). Use when a method **frequently completes synchronously** (e.g., cache hit).

```csharp
public ValueTask<Product?> GetProductAsync(int id)
{
    if (_cache.TryGetValue(id, out var product))
        return ValueTask.FromResult(product); // no Task allocation

    return new ValueTask<Product?>(LoadFromDbAsync(id));
}
```

> `ValueTask` can only be awaited **once** and must not be stored. When in doubt, use `Task<T>`.

## Common async/await pitfalls

### 1. Deadlock with `.Result` / `.Wait()`

Blocking on async code in a synchronization context causes a deadlock: the thread waits for the task, but the task needs that thread to continue.

```csharp
// ❌ DEADLOCK in ASP.NET controller
public IActionResult GetProduct()
{
    var product = _productService.GetProductAsync(1).Result; // blocks thread
    return Ok(product);
}

// ✅ Async all the way down
public async Task<IActionResult> GetProduct()
{
    var product = await _productService.GetProductAsync(1);
    return Ok(product);
}
```

> **Rule: async all the way down.** Once you go async, every caller must be async too. One synchronous `.Result` in the chain can cause a deadlock.

### 2. `async void` — don't use it

`async void` methods cannot be awaited. Unhandled exceptions crash the process, not just the method.

```csharp
// ❌ Crashes the process if it throws — exception not observable
async void DoWork() { await Task.Delay(1000); throw new Exception("💥"); }

// ✅ Use async Task — caller can await and handle exceptions
async Task DoWorkAsync() { ... }
```

**Only legitimate exception:** WinForms/WPF event handlers (required by the framework event signature).

### 3. Fire-and-forget — swallowed exceptions

```csharp
// ❌ Exception is silently lost — no logs, no alerts
public void SendWelcomeEmail(User user)
{
    _ = _emailService.SendAsync(user.Email, "Welcome!");
}

// ✅ If truly fire-and-forget, always handle the error
public void SendWelcomeEmail(User user)
{
    _ = Task.Run(async () =>
    {
        try { await _emailService.SendAsync(user.Email, "Welcome!"); }
        catch (Exception ex) { _logger.LogError(ex, "Email send failed"); }
    });
}
```

### 4. `ConfigureAwait(false)` — library code

In **library/service code**, use `ConfigureAwait(false)` to avoid capturing the synchronization context:

```csharp
// In a reusable library or service class
var data = await _httpClient.GetStringAsync(url).ConfigureAwait(false);
```

In **ASP.NET Core** (which has no synchronization context), it has no functional effect but is still good practice in shared libraries that might be used from WinForms/WPF.

## Thread pool

The CLR maintains a pool of pre-allocated threads. `async/await` and `Task.Run` dispatch work through the thread pool — no manual thread creation needed in most cases.

```csharp
// Dispatch CPU-bound work to thread pool
await Task.Run(() => HeavyCalculation());
```

## Locks and synchronisation

When multiple threads share mutable state, race conditions occur. Synchronise with:

```csharp
// lock — mutual exclusion for a block of code
private readonly object _sync = new();
lock (_sync) { _counter++; }

// Interlocked — atomic operations on primitives (faster than lock)
Interlocked.Increment(ref _counter);
Interlocked.CompareExchange(ref _value, newValue, expected);
```

**Deadlock** — two threads each wait for the other to release a lock. Prevention:
- Acquire locks in a consistent order everywhere
- Use `SemaphoreSlim` with timeout instead of blocking `lock`
- Minimise the scope of lock blocks

## SemaphoreSlim

Limits the number of concurrent accesses to a resource. Unlike `lock`, it's async-friendly.

```csharp
private readonly SemaphoreSlim _semaphore = new(1, 1); // max 1 at a time

public async Task DoProtectedWorkAsync()
{
    await _semaphore.WaitAsync();           // async wait — doesn't block the thread
    try
    {
        await SomeExclusiveOperationAsync();
    }
    finally
    {
        _semaphore.Release();               // always release in finally
    }
}
```

```csharp
// Rate-limiting: allow up to 5 concurrent callers
private readonly SemaphoreSlim _throttle = new(5, 5);
```

> Use `SemaphoreSlim` over `Semaphore` for async code — `Semaphore` blocks threads, `SemaphoreSlim` awaits.

## CancellationToken — cooperative cancellation

Signals a running async operation to stop gracefully. Three components:

```csharp
// 1. CancellationTokenSource — the controller
var cts = new CancellationTokenSource();

// 2. CancellationToken — the signal passed down
CancellationToken token = cts.Token;

// 3. Cancel from outside
cts.Cancel();
```

### Checking cancellation

```csharp
// Option A: throws OperationCanceledException (preferred — propagates cleanly)
token.ThrowIfCancellationRequested();

// Option B: check and handle manually
if (token.IsCancellationRequested) return;
```

### Propagation pattern — always pass the token down

```csharp
public async Task ProcessTournamentAsync(long id, CancellationToken cancellationToken = default)
{
    var tournament = await _repo.GetByIdAsync(id, cancellationToken);
    await _unitOfWork.SaveChangesAsync(cancellationToken);
}
```

`default` = "no cancellation" — safe for internal callers that don't supply one.

### In ASP.NET Core — automatic

```csharp
[HttpGet("{id}")]
public async Task<IActionResult> GetTournament(long id, CancellationToken cancellationToken)
{
    // If client disconnects, token fires → DB query cancelled → no wasted work
    var result = await _service.GetAsync(id, cancellationToken);
    return Ok(result);
}
```

ASP.NET Core binds `CancellationToken` parameters from `HttpContext.RequestAborted` automatically.

### Timeout and linked tokens

```csharp
// Auto-cancel after 5 seconds
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5));
await DoWorkAsync(cts.Token);

// Cancel if either user aborts OR timeout fires
using var combined = CancellationTokenSource.CreateLinkedTokenSource(
    httpContext.RequestAborted, timeoutToken);
await DoWorkAsync(combined.Token);
```

## Parallel programming

### When to use what

| Problem | Solution |
|---|---|
| I/O-bound (DB, HTTP, file) | `async/await` |
| CPU-bound on a collection | `Parallel.ForEach` or `Task.Run` |
| Multiple independent async ops | `Task.WhenAll` |

```csharp
// Parallel.ForEach — CPU-bound
Parallel.ForEach(images, new ParallelOptions { CancellationToken = token },
    img => ProcessImage(img));

// Task.WhenAll — concurrent async I/O
var tasks = ids.Select(id => _repo.GetByIdAsync(id, token));
var results = await Task.WhenAll(tasks);
```

## Interview Traps

- **"Deadlock from `.Result`?"** — The thread calling `.Result` blocks waiting for the task. The task continuation needs the same thread (sync context). Neither can proceed — deadlock. Fix: async all the way.
- **"async void vs async Task?"** — `async void` exceptions crash the process and can't be caught by callers. `async Task` exceptions propagate normally when awaited.
- **"`CancellationToken` — what is it?"** — Cooperative cancellation. The caller signals intent to cancel; the callee checks `ThrowIfCancellationRequested()` and stops gracefully. Nothing is forcibly killed.
- **"`ThrowIfCancellationRequested` vs `IsCancellationRequested`?"** — `ThrowIfCancellationRequested` throws `OperationCanceledException` — the standard way that propagates cleanly. `IsCancellationRequested` is for when you need to do cleanup before stopping.
- **"`Parallel.ForEach` vs `Task.WhenAll`?"** — `Parallel.ForEach` = CPU-bound, threads. `Task.WhenAll` = I/O-bound, concurrent async tasks.
- **"`SemaphoreSlim` vs `lock`?"** — `lock` is synchronous (blocks the thread). `SemaphoreSlim.WaitAsync()` is async — it frees the thread to do other work while waiting.
