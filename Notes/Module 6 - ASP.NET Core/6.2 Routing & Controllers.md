# 6.2 Routing & Controllers

## Minimal APIs vs Controllers

**.NET 6+** introduced **Minimal APIs** — a lightweight way to define endpoints without controllers.

```csharp
// Minimal API — good for small services, microservices, simple CRUD
app.MapGet("/api/tournaments", async (AppDbContext db) =>
    await db.Tournaments.ToListAsync());

app.MapPost("/api/tournaments", async (Tournament t, AppDbContext db) =>
{
    db.Tournaments.Add(t);
    await db.SaveChangesAsync();
    return Results.Created($"/api/tournaments/{t.Id}", t);
});
```

| Aspect | Controllers | Minimal APIs |
|---|---|---|
| Best for | Large apps, complex logic | Small services, prototypes |
| Filters | ✅ Full support | ✅ Endpoint filters (.NET 7+) |
| Model binding | ✅ `[FromBody]`, `[FromQuery]` | ✅ Same, auto-inferred |
| Grouping | Controller classes | `MapGroup()` (.NET 7+) |
| Swagger | ✅ Built-in | ✅ Built-in |

## Attribute Routing

Routing maps incoming HTTP requests to controller actions based on URL and HTTP method.

```csharp
[Route("api/[controller]")]
[ApiController]
public class TournamentsController : ControllerBase
{
    [HttpGet("{id:int}")]              // GET api/tournaments/5
    public IActionResult Get(int id) => Ok(...);

    [HttpGet("search/{name:alpha}")]   // GET api/tournaments/search/cup
    public IActionResult Search(string name) => Ok(...);

    [HttpPost]
    public async Task<IActionResult> Create(CreateTournamentRequest request)
    {
        // With [ApiController]: model validation is automatic — no need for ModelState.IsValid
        var tournament = await _service.CreateAsync(request);
        return CreatedAtAction(nameof(Get), new { id = tournament.Id }, tournament);
    }
}
```

**Route constraints:** `:int`, `:guid`, `:alpha`, `:minlength(5)`, `:range(1,100)`.

> **`[ApiController]` annotation:** Enables automatic model validation (returns 400 on invalid `ModelState`), infers `[FromBody]` for complex types, and formats validation errors as `ProblemDetails`. Without it you must check `ModelState.IsValid` manually.

## MVC Pattern

**Model → View → Controller** — server-side rendering pattern.

- **Model** — the data / domain objects
- **View** — Razor `.cshtml` templates that render HTML
- **Controller** — receives requests, calls services, returns views

For **Web APIs** (JSON-returning), controllers inherit from `ControllerBase` (no view support). For **Razor MVC** (HTML-returning), controllers inherit from `Controller` (adds `View()`, `PartialView()`, etc.).

In modern .NET shops, the typical flow is: API controller → service layer → repository → database. Views are handled by a separate frontend (React/Vue).

## Common `IActionResult` return types

| Method | Status Code | When to use |
|---|---|---|
| `Ok(data)` | 200 | Successful GET / general success |
| `Created(uri, data)` | 201 | Resource created (POST) |
| `CreatedAtAction(...)` | 201 | Created, with Location pointing to GET endpoint |
| `NoContent()` | 204 | Success, no body (DELETE / PUT) |
| `BadRequest(...)` | 400 | Validation failure |
| `Unauthorized()` | 401 | Not authenticated |
| `Forbid()` | 403 | Authenticated but not authorized |
| `NotFound()` | 404 | Resource doesn't exist |
| `Conflict()` | 409 | State conflict |

```csharp
// Typed result — better than IActionResult for Swagger generation
public async Task<ActionResult<TournamentDto>> GetTournament(long id)
{
    var tournament = await _service.GetAsync(id);
    return tournament is null ? NotFound() : Ok(tournament);
}
```

## IHostedService / BackgroundService

Code that runs alongside your app but is **not triggered by HTTP requests**. It starts when the app starts, stops when the app shuts down.

Real uses: message queue consumers (Kafka, RabbitMQ), scheduled cleanup jobs, cache warming on startup.

`IHostedService` is the raw interface — you implement `StartAsync` and `StopAsync` manually. In practice you always use `BackgroundService` instead, which gives you a single `ExecuteAsync` method with a loop.

```csharp
public class KafkaConsumerService : BackgroundService
{
    private readonly ILogger<KafkaConsumerService> _logger;

    public KafkaConsumerService(ILogger<KafkaConsumerService> logger)
        => _logger = logger;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            // poll Kafka, process message
            _logger.LogInformation("Consuming...");
            await Task.Delay(1000, stoppingToken);
        }
    }
}

// Register in Program.cs
builder.Services.AddHostedService<KafkaConsumerService>();
```

**The one trap you must know:** `BackgroundService` is registered as a **singleton**. If you inject `DbContext` or any scoped service directly into the constructor, you get a lifetime mismatch — the scoped service never gets disposed. Fix: inject `IServiceScopeFactory` and create a scope per unit of work.

```csharp
public class KafkaConsumerService : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            using var scope = _scopeFactory.CreateScope();
            var repo = scope.ServiceProvider.GetRequiredService<ITournamentRepository>();
            // use repo, scope disposes at end of using block
        }
    }
}
```

## Interview Traps

- **"Controller vs ControllerBase?"** — `Controller` inherits `ControllerBase` and adds Razor view support (`View()`, `PartialView()`). For Web APIs returning JSON, use `ControllerBase`.
- **"What does `[ApiController]` do?"** — Automatic model validation (400 on invalid input), infers `[FromBody]` for complex parameter types, formats validation errors as `ProblemDetails`.
- **"Minimal API vs Controller — when to choose?"** — Minimal API for simple/small services. Controllers for complex apps where you need the full filter pipeline, action filters, or separation of concerns across controller classes.
- **"What are route constraints?"** — Inline type validation on route parameters: `{id:int}` only matches integers, `{name:alpha}` only letters. Returns 404 automatically if the constraint fails.
- "Can you inject DbContext directly into a BackgroundService?"_ — No. It's a singleton, DbContext is scoped. Use `IServiceScopeFactory` and create a new scope per iteration.
