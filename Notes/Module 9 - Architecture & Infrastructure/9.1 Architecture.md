# 9.1 — Architecture

## Layered / N-Tier Architecture

The classic split: divide the application into horizontal layers where each layer only talks to the one directly below it.

```
Presentation  (Controllers, API endpoints)
     ↓
Application   (Services, use-case orchestration)
     ↓
Domain        (Entities, business rules)
     ↓
Data Access   (Repositories, DbContext)
```

Simple, predictable, easy to reason about. Works well for most line-of-business apps. The risk: over time, layers become a formality and logic leaks everywhere — controllers calling repos directly, business logic in SQL, etc.

---

## Clean Architecture

Clean Architecture enforces the same layers but adds one critical rule: **dependency direction is strictly inward**. The Domain layer knows nothing about the layers around it — not EF Core, not ASP.NET, not JSON. Everything depends on Domain; Domain depends on nothing.

```
        ┌─────────────────────┐
        │   Infrastructure    │  EF Core, HTTP clients, file system
        │   ┌─────────────┐   │
        │   │  Application│   │  Services, use cases, interfaces
        │   │  ┌───────┐  │   │
        │   │  │Domain │  │   │  Entities, business rules, domain interfaces
        │   │  └───────┘  │   │
        │   └─────────────┘   │
        └─────────────────────┘
              ↑
        Presentation           Controllers, minimal APIs — depends on Application
```

The Domain defines interfaces (`ITournamentRepository`). Infrastructure implements them (`TournamentRepository : ITournamentRepository`). The Domain never imports EF Core — it just declares what it needs.

**Why it matters:** you can swap out EF Core for Dapper, replace SQL Server with PostgreSQL, or test the Domain without a database at all.

**In your TO2 app:** your Clean Architecture layers, generic repository, and interface-driven services are exactly this pattern. If an interviewer asks "what is Clean Architecture?" — walk them through your own project.

**Interview trap:**

- _"Clean Architecture vs N-Tier — what's the difference?"_ — N-Tier allows dependencies to flow down through all layers. Clean Architecture enforces that the inner rings (Domain, Application) never reference outer rings (Infrastructure, Presentation). The direction of dependency is the key distinction, not the number of layers.

## Monolith vs Microservices

**Monolith** — the entire application is one deployable unit. One codebase, one database, one process. Most applications start here.

**Microservices** — the application is split into independently deployable services, each owning its own database and communicating over HTTP or a message broker.

||Monolith|Microservices|
|---|---|---|
|Deployment|Single unit|Independent per service|
|Scaling|Scale everything|Scale individual services|
|Complexity|Low (ops)|High (networking, consistency, observability)|
|Team size|Small to medium|Multiple independent teams|
|DB|Shared|One per service|

**When NOT to use microservices** — this is the real junior question:

- Small team (coordination cost kills you)
- Early stage product (boundaries aren't clear yet, you'll get them wrong)
- Simple domain (you add distributed systems complexity for zero benefit)
- No ops maturity (microservices need CI/CD, container orchestration, distributed tracing)

> Netflix and Amazon have microservices because they have thousands of engineers. A team of 5 building a tournament platform does not. Starting with a monolith and extracting services later (when you actually understand the boundaries) is almost always the right call.

**Interview trap:**

- _"Why is your TO2 a monolith?"_ — Because the domain is well-understood, the team is small, and there's no scaling problem that justifies the operational overhead of distributed services. Microservices solve org-scale problems, not code problems.

## Modular Monolith

The middle ground: a single deployable unit internally divided into **strongly bounded modules** with explicit interfaces between them. Modules can't reach into each other's internals — only through defined contracts.

```
Single Process
├── TournamentModule    (owns Tournament, Standing entities)
├── MatchModule         (owns Match, GameResult entities)
└── IdentityModule      (owns User, Tenant entities)
```

Each module could become a microservice later — the boundaries are already drawn. You get the simplicity of a monolith without the big ball of mud.

Awareness level: know what it is, know why it's often a better starting point than jumping straight to microservices.

## CQRS and MediatR

**CQRS** (Command Query Responsibility Segregation) — separate read operations from write operations. A Command changes state and returns nothing (or just an ID). A Query reads state and has no side effects.

```
Command — RegisterTeamCommand   → changes DB, returns TeamId
Query   — GetTournamentQuery    → reads DB, returns TournamentDto
```

**Why:** read and write paths have different needs. Queries can use `AsNoTracking()`, projections, and caching. Commands go through validation and domain logic. Mixing them in one service produces bloated methods.

**MediatR** — an in-process mediator library. Controllers send a command or query object; MediatR routes it to the right handler. Controllers don't reference services directly.

```csharp
// Controller — sends a command, knows nothing about the handler
[HttpPost]
public async Task<IActionResult> Register(RegisterTeamRequest req, IMediator mediator)
{
    var id = await mediator.Send(new RegisterTeamCommand(req.Name, req.TenantId));
    return CreatedAtAction(nameof(Get), new { id }, null);
}

// Handler — owns the logic
public class RegisterTeamHandler : IRequestHandler<RegisterTeamCommand, long>
{
    public async Task<long> Handle(RegisterTeamCommand cmd, CancellationToken ct)
    {
        var team = new Team(cmd.Name, cmd.TenantId);
        await _repo.AddAsync(team, ct);
        return team.Id;
    }
}
```

**Pipeline behaviours** — MediatR lets you wrap every handler with cross-cutting logic (validation, logging, caching) without touching the handlers themselves. Same idea as middleware, but for your application layer.

```csharp
// Runs before every command/query handler automatically
public class ValidationBehaviour<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
{
    public async Task<TResponse> Handle(TRequest request,
        RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        // validate request — throw if invalid
        return await next(); // continue to handler
    }
}
```

**Interview traps:**

- _"What is CQRS?"_ — Separation of reads and writes into distinct models/handlers. Not necessarily different databases — even in-process separation is CQRS.
- _"Do you need MediatR for CQRS?"_ — No. MediatR is a convenience. You can implement CQRS with plain service classes. MediatR adds the mediator pattern and pipeline behaviour infrastructure on top.
- _"What problem does MediatR solve?"_ — Decouples controllers from service classes, eliminates large service interfaces, and gives you a single place to plug in cross-cutting concerns via pipeline behaviours.

## Event-Driven Architecture (Awareness)

Instead of services calling each other directly, they emit events. Other services subscribe and react asynchronously.

```
MatchService publishes  → MatchCompletedEvent
                                ↓
StandingService subscribes → updates standings
NotificationService subscribes → sends push notification
```

**Why:** loose coupling — the publisher doesn't know who's listening. Adding a new reaction (e.g. a stats service) means zero changes to the publisher.

**In your TO2:** you used Kafka for this. `MatchCompletedEvent` on a topic, consumers handling the downstream effects. That's event-driven in practice.

**The tradeoff:** you give up immediate consistency. The standings update happens eventually, not in the same transaction as the match result. This is **eventual consistency** — accepted in exchange for decoupling and resilience.

Awareness level: know what it is, know the tradeoff (loose coupling vs eventual consistency), and be able to reference your Kafka experience.
