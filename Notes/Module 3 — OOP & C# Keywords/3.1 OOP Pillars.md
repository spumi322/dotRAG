# 3.1 OOP Pillars

OOP is a paradigm centred around **objects** that bundle state (data) and behaviour (methods). Four core pillars define it.

---

## Encapsulation

Bundling data and the methods that operate on it into a single unit, while **controlling who can access** what. State is hidden; behaviour is exposed through a deliberate interface.

```csharp
public class Tournament
{
    // State is private — nobody touches it directly
    private int _maxTeams;
    private readonly List<TournamentTeam> _teams = new();

    // Controlled access — validation lives here, not scattered across callers
    public int MaxTeams
    {
        get => _maxTeams;
        private set
        {
            if (value < 2) throw new ArgumentException("Need at least 2 teams.");
            _maxTeams = value;
        }
    }

    // Expose read-only view — callers can iterate, not mutate
    public IReadOnlyList<TournamentTeam> Teams => _teams;

    public void AddTeam(TournamentTeam team)
    {
        if (_teams.Count >= _maxTeams)
            throw new InvalidOperationException("Tournament is full.");
        _teams.Add(team); // the only valid way to add a team
    }
}
```

**Access modifiers:**

| Modifier | Accessible from |
|---|---|
| `private` | Declaring class only |
| `protected` | Declaring class + derived classes |
| `internal` | Same assembly |
| `protected internal` | Same assembly OR derived classes |
| `private protected` | Declaring class + derived classes in same assembly |
| `public` | Anywhere |

---

## Inheritance

A class acquires members (fields, properties, methods) of another class. C# supports **single class inheritance** only — one base class. Multiple **interface** implementation is allowed.

```csharp
public abstract class EntityBase
{
    public long Id { get; protected set; }
    public DateTime CreatedAt { get; private set; } = DateTime.UtcNow;
    public long TenantId { get; protected set; }
}

public class Tournament : EntityBase
{
    public string Name { get; private set; }

    public Tournament(string name, long tenantId)
    {
        Name = name;
        TenantId = tenantId; // protected setter — accessible from derived class
    }
}
```

`base` keyword calls the parent constructor or method:

```csharp
public class SpecialTournament(string name, long tenantId, int maxRounds)
    : Tournament(name, tenantId)       // calls Tournament's constructor
{
    public int MaxRounds { get; } = maxRounds;

    public override string ToString()
        => $"{base.ToString()} — {MaxRounds} rounds";
}
```

## Polymorphism

Letting different types be treated as the same type through a shared interface or base class. The correct behaviour is resolved at **runtime** based on the actual object type.

```csharp
public abstract class Standing
{
    public abstract string GetSummary();
}

public class GroupStanding : Standing
{
    public override string GetSummary() => "Group stage standings";
}

public class BracketStanding : Standing
{
    public override string GetSummary() => "Bracket standings";
}

// Caller works against the base type
Standing s = new GroupStanding();
s.GetSummary(); // "Group stage standings" — dispatch at runtime
```

## Abstraction

Hiding **how** something works, exposing only **what** it does. Achieved through abstract classes and interfaces — callers depend on the contract, not the implementation.

```csharp
// Interface — the what
public interface IRepository<T>
{
    Task<T?> GetByIdAsync(long id);
    Task<IReadOnlyList<T>> GetAllAsync();
}

// Caller depends on the interface, not the concrete class
public class TournamentService(IRepository<Tournament> repo)
{
    public async Task<Tournament?> Get(long id) => await repo.GetByIdAsync(id);
}
```

See [[3.2 Abstraction Mechanisms]] for abstract class vs interface decision.

## Composition over Inheritance

**Inheritance — "is-a":** use only when the derived type genuinely is a specialisation of the base.

**Composition — "has-a":** use when a class _uses_ another class but is not a specialisation of it.

```csharp
// ✅ Tournament IS an EntityBase — inheritance is correct
public class Tournament : EntityBase { }

// ✅ Tournament HAS a PrizePool — composition is correct
public class Tournament : EntityBase
{
    public PrizePool Prize { get; private set; }
    public Format Format { get; private set; }
}

// ❌ Don't inherit just to reuse code
public class Tournament : PrizePool { }  // Tournament is NOT a PrizePool
```

**Rule of thumb:** if you can't say "A is a B" naturally in plain English, use composition. Deep inheritance hierarchies (3+ levels) are fragile — a base class change breaks all derived classes.

---

## Interview Traps

- **"Four pillars of OOP?"** — Encapsulation, Inheritance, Polymorphism, Abstraction. Know a one-liner and a code example for each.
- **"C# has multiple inheritance?"** — No, single class inheritance only. Multiple **interface** implementation is not the same thing.
- **"When does inheritance break?"** — When the base class changes and all derived classes are affected. Deep hierarchies create tight coupling.
- **"OOP vs Functional?"** — OOP = objects with state and behaviour. Functional = pure functions, no side effects, immutable data. C# supports both — LINQ is functional style, classes are OOP.
- **"Inheritance vs Composition?"** — Prefer composition. Inheritance is for genuine is-a relationships. Composition is more flexible and produces loosely coupled code.
