# 3.2 Abstraction Mechanisms

## Abstract Class

Blueprint. Defines **how** derived classes must look and provides shared state or default implementations. Cannot be instantiated directly — only inherited.

```csharp
public abstract class EntityBase
{
    // Fields — interfaces can't have these
    private readonly DateTime _createdAt = DateTime.UtcNow;

    // Concrete property — inherited as-is
    public long Id { get; protected set; }

    // Abstract property — derived class MUST implement, no body here
    public abstract string DisplayName { get; }

    // Constructor — called via base() from the derived class
    // Protected so only derived classes can reach it
    protected EntityBase(long id) => Id = id;

    // Virtual method — has a body, derived class CAN override
    public virtual string Describe() => $"Entity #{Id}";

    // Abstract method — no body, derived class MUST override
    public abstract bool IsValid();
}

public class Tournament : EntityBase
{
    public string Name { get; }

    // base(id) calls EntityBase's constructor — required when
    // the base class has no parameterless constructor
    public Tournament(long id, string name) : base(id)
        => Name = name;

    public override string DisplayName => Name;                      // required
    public override bool IsValid() => !string.IsNullOrEmpty(Name);  // required
    public override string Describe() => $"Tournament: {Name}";     // optional
}
```

## Interface

Contract. Defines **what** a class can do, not how. Stateless — no fields, no instance state.

```csharp
public interface IRepository<T>
{
    // METHOD — public abstract by default, no body
    Task<T?> GetByIdAsync(long id);

    // PROPERTY — public, no implementation
    string EntityName { get; }

    // EVENT
    event EventHandler OnSaved;

    // DEFAULT METHOD (C# 8+) — has a body, implementing class inherits it
    // unless it overrides. Useful for adding methods without breaking
    // all existing implementations.
    void Log(string msg) => Console.WriteLine($"[{EntityName}] {msg}");

    // STATIC MEMBER — belongs to the interface type itself
    static string Version => "1.0";
}
```

---
Same named method implementation
```csharp
interface IRunner { void Run(); }
interface ILogger { void Run(); }

class Agent : IRunner, ILogger
{
    void IRunner.Run() => Console.WriteLine("Running");
    void ILogger.Run() => Console.WriteLine("Logging");
}

var agent = new Agent();
agent.Run();              // ❌ compile error — ambiguous
((IRunner)agent).Run();   // ✅ "Running"
((ILogger)agent).Run();   // ✅ "Logging"
```
## C# 8 Default Interface Methods

Before C# 8, adding a method to an interface was a **breaking change** — every implementing class had to add it. Default interface methods solve this by providing a default body.

```csharp
public interface INotifier
{
    void Send(string message);

    // Default implementation — existing implementors inherit this
    // unless they override it
    void SendBatch(IEnumerable<string> messages)
    {
        foreach (var msg in messages)
            Send(msg);
    }
}

// Old implementation — does NOT need to change
public class EmailNotifier : INotifier
{
    public void Send(string message) => Console.WriteLine($"Email: {message}");
    // SendBatch is inherited from the interface — no change required
}

// New implementation — can override for efficiency
public class SmsNotifier : INotifier
{
    public void Send(string message) => Console.WriteLine($"SMS: {message}");

    public void SendBatch(IEnumerable<string> messages)
    {
        // Override for bulk SMS optimisation
        var batch = string.Join(",", messages);
        Console.WriteLine($"SMS batch: {batch}");
    }
}
```

> Default interface methods are accessed through the **interface type**, not the implementing type. If you hold a reference typed as `EmailNotifier`, you cannot call `SendBatch` unless `EmailNotifier` explicitly implements it.

```csharp
INotifier n = new EmailNotifier();
n.SendBatch(messages); // ✅ accessed through interface — uses default

EmailNotifier e = new EmailNotifier();
e.SendBatch(messages); // ❌ compile error — not on concrete type
```

## Abstract Class vs Interface — When to Use Which

| Criteria | Abstract Class | Interface |
|---|---|---|
| State (fields) | ✅ Yes | ❌ No |
| Constructors | ✅ Yes | ❌ No |
| Multiple inheritance | ❌ Single only | ✅ Multiple |
| Default implementation | ✅ Always could | ✅ Since C# 8 |
| Static abstract members | ❌ No | ✅ Since C# 11 |
| Access modifiers on members | ✅ Any | ✅ Since C# 8 (limited) |
| **Use when** | **Shared state/behaviour among related types** | **Defining a capability/contract across unrelated types** |

**Decision rule:**
- Related types that share state and implementation → **abstract class**
- Unrelated types that share a capability → **interface**
- Need multiple inheritance of contracts → **interface**

```csharp
// Abstract class — GroupStanding and BracketStanding share Standing state
public abstract class Standing
{
    public int Id { get; }
    public abstract string GetSummary();
}

// Interface — Tournament and Match are unrelated, but both can be described
public interface IDescribable
{
    string Describe();
}
```

## Interview Traps

- **"Can an interface have state?"** — No (in standard C#). Interfaces cannot have instance fields. They can have static fields and default method implementations since C# 8.
- **"Can a class implement multiple abstract classes?"** — C# disallows multiple class inheritance specifically to avoid the diamond problem: if class D inherits from both B and C, and both inherit from A and override the same method, the compiler has no way to decide which version D should use.
- **"What problem do default interface methods solve?"** — They allow adding new methods to an interface without breaking all existing implementors. Useful for library authors who can't force every consumer to update.
- **"When would you use an abstract class instead of an interface?"** — When you need shared state (fields) or a protected constructor. Abstract classes model "is-a" relationships with shared structure; interfaces model "can-do" capabilities.
