# 4.3 C# Keywords & Polymorphism

---

## Keywords

|Keyword|Applies to|Meaning|
|---|---|---|
|`static`|Class, member|Belongs to type, not instance|
|`abstract`|Class, method/property|Must be inherited / overridden, no body|
|`virtual`|Method, property|Can be overridden in derived class|
|`override`|Method, property|Replaces virtual/abstract base member|
|`new`|Member|Hides base member — non-polymorphic|
|`sealed`|Class, override member|Prevents further inheritance / overriding|

```csharp
// static — belongs to type, no instance needed
public static class MathHelper
{
    public static int Add(int a, int b) => a + b; // MathHelper.Add()
}

// abstract — cannot instantiate, forces derived to implement
public abstract class EntityBase
{
    public abstract string DisplayName { get; } // no body — MUST override
    public virtual string Describe() => $"Entity";  // has body — CAN override
}

// sealed — string is sealed, you cannot inherit from it
// sealed override — prevents further overriding down the chain
public class Middle : EntityBase
{
    public override string DisplayName => "Middle";
    public sealed override string Describe() => "Middle entity"; // locked here
}
```

## Overloading — compile-time polymorphism

Same method name, **different parameters**. Resolved at **compile time** by the compiler based on argument types and count.

```csharp
public class MatchService
{
    public Task<Match> GetAsync(int matchId) => ...;
    public Task<Match> GetAsync(int matchId, long tenantId) => ...;
    public Task<IReadOnlyList<Match>> GetAsync(int[] matchIds) => ...;
}
```

> You cannot overload on return type alone — the compiler won't know which to call.

---

## Overriding — runtime polymorphism

Base declares `virtual` or `abstract`, derived uses `override`. The correct method is resolved at **runtime** based on the **actual object type**.

```csharp
public abstract class Standing
{
    public abstract string GetSummary();
}

public class GroupStanding : Standing
{
    public override string GetSummary() => "Group stage standings";
}

public class BracketStanding : Standing
{
    public override string GetSummary() => "Bracket standings";
}

Standing s = new GroupStanding();
s.GetSummary(); // "Group stage standings" — runtime dispatch
```

---

## Hiding — `new` keyword

`new` hides the base member instead of overriding it. Resolved at **compile time** based on the **variable type**, not the object type. Almost always a design smell.

```csharp
public class Base
{
    public virtual string GetName() => "Base";
}

public class Derived : Base
{
    public new string GetName() => "Derived"; // hides, not overrides
}

Base obj = new Derived();
obj.GetName();    // "Base"    ← variable type wins (compile-time)

Derived obj2 = new Derived();
obj2.GetName();   // "Derived" ← variable type is Derived
```

## Upcasting & Downcasting

**Upcasting** — derived → base. Always implicit, always safe.

```csharp
GroupStanding group = new GroupStanding();
Standing s = group;       // implicit upcast
s.GetSummary();           // still calls GroupStanding.GetSummary() — polymorphism works
```

**Downcasting** — base → derived. Requires explicit cast, can throw `InvalidCastException`. Use `as` + null check or `is` pattern instead.

```csharp
Standing s = new GroupStanding();

// ❌ Risky — throws if wrong type
GroupStanding g = (GroupStanding)s;

// ✅ Safe — returns null if wrong type
GroupStanding? g = s as GroupStanding;
if (g != null) { ... }

// ✅ Preferred — pattern matching
if (s is GroupStanding gs)
{
    Console.WriteLine(gs.GetSummary());
}
```

## Interview Traps

- **"Runtime vs compile-time polymorphism?"** — `override` = runtime (actual object type decides). Overloading = compile-time (parameter signature decides).
- **"What's the difference between `override` and `new`?"** — `override` is polymorphic, dispatch based on object type. `new` hides, dispatch based on variable type.
- **"Can you overload on return type alone?"** — No. C# doesn't consider return type in overload resolution.
- **"What is upcasting?"** — Treating a derived type as its base. Always implicit and safe.
- **"What is downcasting?"** — Treating a base type as derived. Explicit cast required, can throw. Use `as` or `is` pattern.
- **"When would you use `sealed`?"** — To prevent inheritance (security, correctness) or to hint the JIT to devirtualize calls (minor perf gain). `string` is sealed.
- **"Is `new` on a member ever valid?"** — Rarely. Intentional hiding in versioning scenarios (library adds a method with the same name). In application code it's almost always a design mistake.
