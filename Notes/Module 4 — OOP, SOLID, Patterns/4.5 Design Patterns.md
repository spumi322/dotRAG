# 4.5 Design Patterns

Reusable solutions to common design problems. Grouped into three categories.

---

## Creational — how objects are created

### Singleton

Ensures only **one instance** of a class exists. Provides a global access point.

```csharp
// Thread-safe Singleton using Lazy<T>
public sealed class Singleton
{
    private static readonly Lazy<Singleton> _instance =
        new(() => new Singleton());

    private Singleton() { }

    public static Singleton Instance => _instance.Value;
}
```

> In .NET, the idiomatic way is `AddSingleton<ITEntity, TEntity>()` in DI — the container manages the single instance. You rarely implement the pattern manually in modern apps.

**Interview traps:**
- "Thread-safe Singleton without lock?" — Use `Lazy<T>` or a static readonly field (both are thread-safe by CLR guarantees)
- "Problems with Singleton?" — Hidden global state, hard to test, tight coupling. Prefer DI singleton lifetime instead

### Factory

Creates objects without specifying the exact class. The caller depends on the interface, not the concrete type.

```csharp
public interface INotificationSender
{
    void Send(string recipient, string message);
}

public class EmailSender : INotificationSender
{
    public void Send(string recipient, string message)
        => Console.WriteLine($"Email → {recipient}: {message}");
}

public class SmsSender : INotificationSender
{
    public void Send(string recipient, string message)
        => Console.WriteLine($"SMS → {recipient}: {message}");
}

public static class NotificationSenderFactory
{
    public static INotificationSender Create(string type) => type switch
    {
        "email"   => new EmailSender(),
        "sms"     => new SmsSender(),
        _ => throw new ArgumentException($"Unknown type: {type}")
    };
}

var sender = NotificationSenderFactory.Create("email");
sender.Send("player@example.com", "Match starts in 10 minutes.");
```

**Interview traps:**
- "Factory vs `new`?" — Factory hides which concrete type is created; caller only knows the interface. Enables OCP — add a new type by adding a class, not editing the caller
- "Factory vs Abstract Factory?" — Factory creates one product. Abstract Factory creates families of related products (e.g. cross-platform UI controls)

### Builder

Constructs a complex object step by step via a fluent chain. **Same type every time** — solves "too many constructor parameters", not "which type to create" (that's Factory).

```csharp
public class Tournament
{
    public string Name     { get; init; }
    public int MaxTeams    { get; init; }
    public string Format   { get; init; }
    public bool IsPublic   { get; init; }
    public int BestOf      { get; init; }

    private Tournament() { }

    public class Builder
    {
        private readonly Tournament _t = new();

        public Builder WithName(string name)       { _t.Name = name;      return this; }
        public Builder WithMaxTeams(int max)       { _t.MaxTeams = max;   return this; }
        public Builder WithFormat(string format)   { _t.Format = format;  return this; }
        public Builder IsPublicTournament(bool pub){ _t.IsPublic = pub;   return this; }
        public Builder WithBestOf(int n)           { _t.BestOf = n;       return this; }

        public Tournament Build()
        {
            if (string.IsNullOrEmpty(_t.Name)) throw new InvalidOperationException("Name required.");
            if (_t.MaxTeams < 2)               throw new InvalidOperationException("Need ≥2 teams.");
            return _t;
        }
    }
}

var tournament = new Tournament.Builder()
    .WithName("Spring Open 2026")
    .WithMaxTeams(16)
    .WithFormat("SingleElim")
    .WithBestOf(3)
    .IsPublicTournament(true)
    .Build();
```

> `WebApplication.CreateBuilder(args).Services.AddControllers()` is Builder in the BCL.

### Prototype (awareness)

Creates new objects by cloning an existing one. Useful when creation is expensive and you need many similar objects. In C#: `ICloneable`, `record with { }` expressions, or manual `Clone()` methods.

### Abstract Factory (awareness)

Creates **families** of related objects without specifying concrete classes. Example: a UI factory that returns platform-specific buttons/checkboxes for Windows vs macOS.

## Structural — how classes and objects are composed

### Adapter

Wraps an incompatible interface so it fits what your code expects. Common when integrating third-party SDKs.

```csharp
// Your expected interface
public interface IPaymentGateway
{
    Task<bool> ChargeAsync(decimal amount);
}

// Third-party SDK — incompatible signature
public class StripeClient
{
    public Task<StripeResult> CreateCharge(int cents) { ... }
}

// Adapter bridges the gap
public class StripeAdapter(StripeClient stripe) : IPaymentGateway
{
    public async Task<bool> ChargeAsync(decimal amount)
    {
        var result = await stripe.CreateCharge((int)(amount * 100));
        return result.Status == "succeeded";
    }
}
```

Your code only knows `IPaymentGateway` — the Stripe dependency is isolated behind the adapter.

### Decorator

Wraps an object to add behaviour **without modifying the original class**. Stackable at runtime.

```csharp
public interface IMatchService
{
    Task<MatchDto> GetAsync(int id);
}

public class MatchService(IMatchRepository repo) : IMatchService
{
    public async Task<MatchDto> GetAsync(int id) => await repo.GetAsync(id);
}

// Decorator — wraps IMatchService, adds logging transparently
public class LoggingMatchService(IMatchService inner, ILogger<LoggingMatchService> log) : IMatchService
{
    public async Task<MatchDto> GetAsync(int id)
    {
        log.LogInformation("Fetching match {Id}", id);
        var result = await inner.GetAsync(id);
        log.LogInformation("Match {Id} fetched", id);
        return result;
    }
}
```

> **ASP.NET Core middleware IS Decorator** — each middleware wraps the next, adding behaviour (auth, logging, exception handling) without modifying what's inside.

**Interview traps:**
- "Decorator vs Inheritance?" — Inheritance is compile-time, static. Decorator is runtime, composable — you can stack multiple decorators
- "Decorator vs Proxy?" — Decorator adds behaviour. Proxy controls access (auth, caching, lazy load). Intent differs; structure is similar

### Repository (Structural / Architectural)

Abstracts data access behind an interface. Application layer talks to the interface — not to EF Core directly. Enables swapping the ORM and mocking in tests.

```csharp
public interface ITournamentRepository
{
    Task<Tournament?> GetByIdAsync(long id, long tenantId);
    Task AddAsync(Tournament tournament);
}

public class TournamentRepository(AppDbContext ctx) : ITournamentRepository
{
    public async Task<Tournament?> GetByIdAsync(long id, long tenantId)
        => await ctx.Tournaments.FirstOrDefaultAsync(t => t.Id == id && t.TenantId == tenantId);

    public async Task AddAsync(Tournament t)
    {
        ctx.Tournaments.Add(t);
        await ctx.SaveChangesAsync();
    }
}

// Service never imports EF Core
public class TournamentService(ITournamentRepository repo) { ... }
```

## Behavioral — how objects communicate

### Strategy

Defines a family of algorithms behind a common interface. Swap the algorithm at runtime without changing the caller.

```csharp
public interface IGameResultPipelineStep
{
    Task ExecuteAsync(GameResultContext ctx);
}

public class ScoreGameStep : IGameResultPipelineStep
{
    public async Task ExecuteAsync(GameResultContext ctx) { ... }
}

public class UpdateGroupStatsStep : IGameResultPipelineStep
{
    public async Task ExecuteAsync(GameResultContext ctx) { ... }
}

// Caller iterates the interface — doesn't know which steps exist
public class GameResultPipeline(IEnumerable<IGameResultPipelineStep> steps)
{
    public async Task RunAsync(GameResultContext ctx)
    {
        foreach (var step in steps)
            await step.ExecuteAsync(ctx);
    }
}
```

Adding a new step = create the class, register it in DI. Zero changes to the pipeline. Strategy + OCP working together.

**Interview traps:**
- "Strategy vs if/else?" — if/else bakes the algorithm into the caller, violates OCP. Strategy externalises it
- "Strategy vs State?" — Strategy swaps algorithm from outside (caller decides). State swaps behaviour from inside (object decides based on its own state)
- "`IComparer<T>` passed to `Array.Sort()` — which pattern?" — Strategy

### Observer

One publisher notifies many subscribers when state changes. Subscribers don't know about each other. C# `event`/`delegate` is the built-in Observer implementation.

```csharp
public class TournamentService
{
    public event EventHandler<MatchFinishedEventArgs>? MatchFinished;

    public void FinishMatch(Match match)
    {
        // update state...
        MatchFinished?.Invoke(this, new MatchFinishedEventArgs(match));
    }
}

// Subscribers register independently
tournamentService.MatchFinished += async (_, e)
    => await _hubContext.Clients.All.SendAsync("MatchFinished", e.Match);
```

**Interview traps:**
- "Memory leak risk?" — subscribing with `+=` and never unsubscribing keeps the subscriber alive (GC can't collect it). Always unsubscribe with `-=` or use weak references
- "Observer vs Pub/Sub?" — Observer: publisher knows subscribers directly. Pub/Sub: message broker in between (e.g. RabbitMQ, Azure Service Bus) — fully decoupled

## Interview Traps (Cross-pattern)

- **"Factory vs Builder?"** — Factory decides which type to create. Builder constructs the same type step by step.
- **"Decorator vs Proxy?"** — Same structure, different intent. Decorator adds behaviour. Proxy controls access.
- **"Strategy vs State?"** — Strategy: external swap (caller decides which algorithm). State: internal swap (object changes its own behaviour based on its own state).
- **"Where is Observer in .NET?"** — `event`/`delegate`, `IObservable<T>`/`IObserver<T>`, Rx.NET, SignalR hubs, ASP.NET Core `IHostedService` + pub/sub.
- **"Where is Builder in the BCL?"** — `WebApplication.CreateBuilder()`, `StringBuilder`, `HttpClient` with `HttpClientBuilder`.
