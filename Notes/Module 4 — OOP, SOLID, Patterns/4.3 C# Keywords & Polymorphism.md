# 4.3 C# Keywords & Polymorphism

## `static` — belongs to the type, not an instance

```csharp
public class MathHelper
{
    public static double Pi = 3.14159;             // shared across all code
    public static int Add(int a, int b) => a + b;  // called via MathHelper.Add()
}

// Static classes cannot be instantiated or inherited
public static class StringExtensions
{
    public static bool IsNullOrEmpty(this string s) => string.IsNullOrEmpty(s);
}
```

No instance needed. Static members live for the app lifetime. Static classes can only contain static members.

---

## `abstract` — must be overridden, no implementation

```csharp
public abstract class Shape
{
    public abstract double Area();         // no body — derived classes MUST implement
    public virtual string Color => "Red";  // has body — derived classes CAN override
}

public class Circle : Shape
{
    private double _r;
    public Circle(double r) => _r = r;
    public override double Area() => Math.PI * _r * _r; // required

    // var s = new Shape(); // ❌ Cannot instantiate abstract class
}
```

---

## `virtual` + `override` — runtime polymorphism

`virtual` = "this method can be overridden." `override` = "I'm replacing the base implementation." The correct method is resolved **at runtime** based on the actual object type.

```csharp
public class Animal
{
    public virtual string Speak() => "...";
}

public class Dog : Animal
{
    public override string Speak() => "Bark";
}

Animal a = new Dog();
a.Speak(); // "Bark" — runtime dispatch, variable type is Animal but object is Dog
```

---

## `new` (member hiding) — NOT polymorphic

`new` hides the base member instead of overriding it. Method dispatch is based on the **variable type**, not the object type. Almost always a design smell.

```csharp
class Base
{
    public string GetName() => "Base"; // NOT virtual
}

class Derived : Base
{
    public new string GetName() => "Derived"; // hides, does NOT override
}

Base obj = new Derived();
obj.GetName();    // "Base"    ← variable type is Base, Base.GetName() runs

Derived obj2 = new Derived();
obj2.GetName();   // "Derived" ← variable type is Derived
```

---

## `override` vs `new` — the #1 interview trap

```csharp
public class Base
{
    public virtual string Identify() => "Base";
}

public class Child : Base
{
    public override string Identify() => "Child (override)";
}

public class Sneaky : Base
{
    public new string Identify() => "Sneaky (new)";
}

Base a = new Child();
a.Identify();  // "Child (override)" — override is polymorphic, runtime dispatch

Base b = new Sneaky();
b.Identify();  // "Base" — new hides, compile-time dispatch, Base.Identify() wins
```

> **Rule:** Use `override` when you want the subclass behaviour to apply regardless of what type the variable is declared as. `new` is almost never what you want when working with polymorphism.

---

## `sealed` — prevent inheritance or further overriding

```csharp
public sealed class Singleton { } // cannot be inherited

public class Base
{
    public virtual void Execute() { }
}

public class Middle : Base
{
    public sealed override void Execute() { } // subclasses of Middle cannot override
}
```

`string` is a sealed class — you cannot inherit from it. Sealing also enables JIT optimisations (devirtualisation of method calls).

## Polymorphism: runtime vs compile-time

| Type | Keyword | Resolution | Mechanism |
|---|---|---|---|
| **Runtime** (dynamic) | `virtual` + `override` | Object type at runtime | Virtual dispatch table |
| **Compile-time** (static) | Method overloading | Parameter signature | Compiler picks overload |
| **Non-polymorphic hiding** | `new` | Variable type at compile time | Hides base member |

---

## Upcasting and Downcasting

**Upcasting** — treating a derived type as its base. Implicit, always safe.

```csharp
Dog dog = new Dog();
Animal a = dog;         // upcast — implicit
a.Speak();              // "Bark" — still polymorphic
```

**Downcasting** — treating a base type as a derived type. Requires explicit cast, can throw `InvalidCastException`.

```csharp
Animal a = new Dog();

// ❌ Risky — throws if a is not actually a Dog
Dog d = (Dog)a;

// ✅ Safe — use 'as' + null check
Dog? d = a as Dog;
if (d != null) d.Fetch();

// ✅ Preferred in modern C# — pattern matching
if (a is Dog dog)
    dog.Fetch();
```

---

## Keyword Summary

| Keyword | Applies to | Meaning |
|---|---|---|
| `static` | Class, member | Belongs to type, not instance |
| `abstract` | Class, method | Must be inherited/overridden, no body |
| `virtual` | Method, property | Can be overridden in derived class |
| `override` | Method, property | Replaces virtual/abstract base member (polymorphic) |
| `new` | Member | Hides base member — not polymorphic |
| `sealed` | Class, override | Prevents further inheritance or overriding |

---

## Interview Traps

- **"What does `override` vs `new` do?"** — `override` is polymorphic: method dispatch uses the object's actual type. `new` hides: dispatch uses the variable's declared type.
- **"Can you override a non-virtual method?"** — No, compile error. The base method must be marked `virtual` or `abstract` to be overridable.
- **"Can you overload on return type alone?"** — No. C# ignores return type in overload resolution.
- **"What is upcasting?"** — Treating a derived type as its base. Implicit and safe. Downcasting (base → derived) requires explicit cast and can throw — prefer `is` pattern or `as` + null check.
- **"Why is `string` sealed?"** — To prevent subclassing, which would undermine string interning and other optimisations. Sealing also allows the JIT to devirtualise calls.
