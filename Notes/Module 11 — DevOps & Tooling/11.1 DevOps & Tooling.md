# 11.1 DevOps & Tooling

## Git

### Merge vs Rebase

Both integrate changes from one branch into another — different strategies:

```
Feature branch (F1, F2, F3)
Main          (M1, M2)

Merge result:
  M1 → M2 → M3 (merge commit joining both histories)
       ↗
  F1 → F2 → F3

Rebase result:
  M1 → M2 → F1' → F2' → F3' (linear history, commits replayed)
```

| | Merge | Rebase |
|---|---|---|
| History | Preserves full branching history | Linear, cleaner |
| Safety | Non-destructive | Rewrites commit hashes |
| Conflicts | Resolved once | Resolved per replayed commit |
| Use for | Feature merging (PR), preserving context | Local cleanup, syncing with main before PR |

**Rule:** Never rebase shared/public branches — it rewrites history and breaks other developers' work. Rebase locally before opening a PR; merge when closing it.

```bash
# Sync feature branch with latest main (before opening PR)
git checkout feature/my-feature
git rebase main

# Merge PR (GitHub/Azure DevOps does this)
git checkout main
git merge --no-ff feature/my-feature  # --no-ff preserves branch in history
```

### Feature branch / PR workflow

```
main ──────────────────────────────────→ (production-ready, protected)
         ↑                    ↑
  feature/add-login    fix/null-check

1. Branch from main:    git checkout -b feature/add-login
2. Make commits
3. Push:                git push -u origin feature/add-login
4. Open Pull Request — code review, CI checks must pass
5. Squash and merge (or merge) into main
6. Delete feature branch
```

**Key practices:**
- Short-lived branches (days, not weeks)
- Small PRs — easier to review
- Require at least 1 approval + CI green before merge
- Protect `main`: no direct pushes, no force pushes

### Common commands

```bash
git log --oneline --graph        # visual branch history
git stash / git stash pop        # temporarily shelve changes
git cherry-pick <sha>            # apply a single commit to current branch
git bisect start/bad/good        # binary search for regression
git reset --soft HEAD~1          # undo last commit, keep changes staged
git reset --hard HEAD~1          # undo last commit, discard changes (⚠️ destructive)
```

## Docker

### What Docker is

Docker packages an application and all its dependencies (runtime, libraries, config) into a **container** — a lightweight, isolated, reproducible environment that runs the same everywhere.

```
Without Docker:
  "Works on my machine" → deploy → "Crashes on server" (different OS, Node version, env vars)

With Docker:
  Container = your app + exact OS + exact runtime + exact config → identical everywhere
```

### Dockerfile

Defines how to build a Docker image for your app:

```dockerfile
# .NET API — multi-stage build
FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build
WORKDIR /src
COPY ["dotRAG.API/dotRAG.API.csproj", "dotRAG.API/"]
RUN dotnet restore "dotRAG.API/dotRAG.API.csproj"
COPY . .
RUN dotnet publish "dotRAG.API/dotRAG.API.csproj" -c Release -o /app/publish

FROM mcr.microsoft.com/dotnet/aspnet:9.0 AS final
WORKDIR /app
COPY --from=build /app/publish .
EXPOSE 8080
ENTRYPOINT ["dotnet", "dotRAG.API.dll"]
```

**Multi-stage builds** — the build stage uses the full SDK image; the final stage uses only the lightweight runtime image. Result: smaller production image.

### docker-compose

Defines and runs multi-container applications (API + DB + cache):

```yaml
# docker-compose.yml
services:
  api:
    build: .
    ports:
      - "8080:8080"
    environment:
      - ConnectionStrings__Default=Server=db;Database=dotRAG;...
    depends_on:
      - db

  db:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      - ACCEPT_EULA=Y
      - SA_PASSWORD=YourStrong!Password
    ports:
      - "1433:1433"
```

```bash
docker-compose up -d        # start all services in background
docker-compose down         # stop and remove containers
docker-compose logs -f api  # tail logs for a service
```

## CI/CD

### What it is

**CI (Continuous Integration)** — automatically build and test every commit/PR.
**CD (Continuous Delivery/Deployment)** — automatically deploy to staging/production after CI passes.

### Pipeline stages (typical)

```
Push to PR → CI triggers:
  1. Build          — dotnet build / npm install
  2. Unit tests     — dotnet test
  3. Static analysis — linting, code quality checks
  4. Docker build   — build and tag image

Merge to main → CD triggers:
  5. Integration tests  — run against staging environment
  6. Deploy to staging  — push image, run migrations
  7. (Manual gate or auto) Deploy to production
```

### Triggers

```yaml
# GitHub Actions example
on:
  push:
    branches: [main]        # CD: deploy on merge to main
  pull_request:
    branches: [main]        # CI: test every PR against main
```

### GitHub Actions example

```yaml
name: CI
on: [push, pull_request]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '9.x'
      - run: dotnet restore
      - run: dotnet build --no-restore
      - run: dotnet test --no-build --verbosity normal
```

## Caching strategies

### In-memory cache

Stores data in the process's memory. Fast, but local to one server instance — not suitable for horizontally scaled apps.

```csharp
// Registration
builder.Services.AddMemoryCache();

// Usage
public class TournamentService(IMemoryCache cache, ITournamentRepository repo)
{
    public async Task<Tournament?> GetAsync(int id)
    {
        return await cache.GetOrCreateAsync($"tournament:{id}", async entry =>
        {
            entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
            return await repo.GetByIdAsync(id);
        });
    }
}
```

### Distributed cache (Redis)

Stores data outside the process — shared across all app instances. Required for load-balanced / multi-instance deployments.

```csharp
builder.Services.AddStackExchangeRedisCache(o =>
    o.Configuration = builder.Configuration["Redis:ConnectionString"]);

// Usage
await _cache.SetStringAsync("key", value, new DistributedCacheEntryOptions
{
    AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5)
});
var value = await _cache.GetStringAsync("key");
```

### Cache-aside pattern

The application is responsible for loading data into the cache on cache miss, and invalidating it on update.

```
Read:
  1. Check cache → hit? Return cached value.
  2. Cache miss → read from DB → store in cache → return value.

Write:
  1. Update DB.
  2. Invalidate (remove) the cached entry.
     (Next read will fetch fresh data and re-cache it.)
```

```csharp
// Read (cache-aside)
var tournament = await _cache.GetAsync(key);
if (tournament is null)
{
    tournament = await _repo.GetByIdAsync(id);
    await _cache.SetAsync(key, tournament, options);
}

// Write — invalidate after update
await _repo.UpdateAsync(tournament);
await _cache.RemoveAsync(key);
```

> **Cache invalidation** is the hard part. Common strategies: time-based expiration (TTL), explicit removal on write, write-through (update cache and DB together).

## Performance metrics

| Metric | What it measures | Tool |
|---|---|---|
| **Latency** | Time for a single request (p50, p95, p99) | Application Insights, Datadog |
| **Throughput** | Requests per second the system handles | Load tests (k6, NBomber) |
| **Error rate** | % of failed requests | Dashboards, alerts |
| **CPU/Memory** | Server resource utilization | Azure Monitor, Prometheus |
| **DB query time** | Slow queries (N+1, missing indexes) | EF Core logging, SQL profiler |

**p95 latency** = 95% of requests complete within this time. More actionable than average (which hides outliers).

**Key .NET performance tools:**
- `dotnet-trace` — CPU profiling
- `dotnet-counters` — live runtime metrics (GC, thread pool, request rate)
- Application Insights — distributed tracing, dependency tracking, live metrics

## Interview Traps

- **"Merge vs Rebase?"** — Both integrate branches. Merge preserves full history with a merge commit. Rebase replays commits onto the target branch for linear history. Never rebase shared branches — it rewrites commit hashes.
- **"What is Docker?"** — Packages an app + its dependencies into a portable container that runs identically everywhere. Solves "works on my machine" problems.
- **"What is CI/CD?"** — CI automatically builds and tests every commit. CD automatically deploys to staging/production after CI passes. Together they make releases fast and reliable.
- **"In-memory vs distributed cache?"** — In-memory is per-process (fast, lost on restart, not shared). Distributed (Redis) is shared across instances — required for horizontally scaled apps.
- **"What is cache-aside?"** — App checks cache first; on miss, reads from DB and populates cache. On write, invalidates the cached entry. The application owns cache management.
- **"What is p99 latency?"** — 99% of requests complete within this time. More meaningful than average because it captures worst-case user experience.
