# 2.1 Generics

## What generics are and why they're used

Generic types allow you to define classes, interfaces, and methods with **type placeholders**. The concrete type is supplied by the caller, giving you reusable, type-safe code without duplication or boxing.

```csharp
// Works with any type that implements IComparable<T>
public T GetMax<T>(T a, T b) where T : IComparable<T>
{
    return a.CompareTo(b) > 0 ? a : b;
}

GetMax(3, 7);      // 7
GetMax("a", "z");  // "z"
```

### Generic classes and interfaces

```csharp
// Generic interface — T is the entity type
public interface IRepository<T> where T : EntityBase
{
    Task<T?> GetByIdAsync(long id);
    Task<IReadOnlyList<T>> GetAllAsync();
}

// One implementation works for every entity
public class Repository<T> : IRepository<T> where T : EntityBase { ... }
```

## Constraints (`where T : ...`)

Constraints let you require that a type parameter satisfies certain conditions.

| Constraint | Meaning |
|---|---|
| `where T : class` | T must be a reference type |
| `where T : struct` | T must be a value type |
| `where T : new()` | T must have a parameterless constructor |
| `where T : SomeClass` | T must inherit from SomeClass |
| `where T : ISomeInterface` | T must implement the interface |
| `where T : notnull` | T must be non-nullable |
| `where T : unmanaged` | T must be an unmanaged type (no references) |

```csharp
// Factory method — needs new()
public T Create<T>() where T : new() => new T();

// Work with entities — needs specific base class
public void Save<T>(T entity) where T : EntityBase, new() { ... }
```

## Func\<\>, Action\<\>, Predicate\<T\>

Built-in generic delegates. The foundation of LINQ and callbacks in .NET.

### Func\<T, TResult\>

Represents a method that **returns a value**. The last type parameter is always the return type.

```csharp
Func<int, int, int> add = (a, b) => a + b;
int result = add(3, 4); // 7

// Common in LINQ — Select takes Func<T, TResult>
var names = tournaments.Select(t => t.Name); // Func<Tournament, string>
```

### Action\<T\>

Represents a method that **returns void**. Zero or more input parameters.

```csharp
Action<string> log = message => Console.WriteLine(message);
log("Tournament started");

// Common in callbacks and event handlers
```

### Predicate\<T\>

Represents a method returning `bool`. Takes exactly one input.

```csharp
Predicate<int> isEven = n => n % 2 == 0;
Console.WriteLine(isEven(4)); // True
```

**Predicate\<T\> vs Func\<T, bool\>:** Signature-identical. `Predicate<T>` is older (pre-LINQ) and appears in older APIs like `List<T>.FindAll()`. Modern code uses `Func<T, bool>` everywhere.

```csharp
var numbers = new List<int> { 1, 2, 3, 4 };

// Old style
var evens1 = numbers.FindAll(n => n % 2 == 0); // Predicate<int>

// Modern style
var evens2 = numbers.Where(n => n % 2 == 0).ToList(); // Func<int, bool>
```

> **Interview answer:** "`Predicate<T>` is the same as `Func<T, bool>` but predates LINQ. Modern code prefers `Func`."

## Covariance and Contravariance

These describe how generic types relate to each other when type parameters have an inheritance relationship.

### Covariance (`out`) — use a more derived type

Applies to interfaces and delegates where T is only **returned** (produced), not written. Declared with `out`.

```csharp
IEnumerable<out T>       // read T out
IReadOnlyList<out T>     // read T out
Func<out TResult>        // returns TResult
```

```csharp
// TournamentDetailsDto inherits from TournamentDto
public IReadOnlyList<TournamentDto> GetTournaments()
{
    var details = new List<TournamentDetailsDto> { ... };
    return details; // ✅ Works — IReadOnlyList<out T> is covariant
}
```

**Why `List<T>` is NOT covariant:** `List<T>` allows writes (`.Add()`, `.Remove()`). If it were covariant, you could add a `TournamentDto` to a `List<TournamentDetailsDto>`, breaking type safety.

Covariance hierarchy (all read-only → all covariant):

```
IEnumerable<out T>
    ↑
IReadOnlyCollection<out T>
    ↑
IReadOnlyList<out T>
```

### Contravariance (`in`) — use a less derived type

Applies where T is only **consumed** (written in), not returned. Declared with `in`.

```csharp
IComparer<in T>          // takes T for comparison
Action<in T>             // takes T as parameter
Func<in TInput, out TResult>
```

```csharp
// A handler that accepts any Exception can be used where a
// more specific exception type is expected — contravariance
Action<Exception> logger = ex => Console.WriteLine(ex.Message);

Action<InvalidOperationException> logInvalidOp = logger; // ✅
Action<ArgumentException> logArgument = logger;          // ✅
```

### Covariance vs Contravariance summary

| | Direction | Keyword | Example |
|---|---|---|---|
| Covariance | Return / produce | `out` | `IEnumerable<out T>` |
| Contravariance | Accept / consume | `in` | `Action<in T>` |

---

## Interview Traps

- **"Why can't `List<T>` be covariant?"** — Because it allows writes. Covariance only works on read-only (output-only) interfaces.
- **"What's the difference between `Func<T, bool>` and `Predicate<T>`?"** — Same signature. `Predicate<T>` predates LINQ and is found in older APIs. Modern code uses `Func<T, bool>`.
- **"When would you add a constraint `where T : new()`?"** — When your generic code needs to instantiate a T (e.g., a factory). Without the constraint, `new T()` won't compile because the compiler can't guarantee T has a parameterless constructor.
