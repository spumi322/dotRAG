# 7.4 EF Core

## What is an ORM?

ORM (Object-Relational Mapper) bridges the gap between relational databases and object-oriented code. Instead of writing raw SQL, you work with C# classes and the ORM generates SQL.

```csharp
// Without ORM — manual, verbose, error-prone
var cmd = new SqlCommand("SELECT * FROM Tournaments WHERE Id = @id", conn);
cmd.Parameters.AddWithValue("@id", id);
var reader = cmd.ExecuteReader();
// manually map every column to a property...

// With EF Core — one line
var tournament = await _context.Tournaments.FindAsync(id);
```

**Advantages:** Productivity, LINQ support, migrations, change tracking, parameterized queries by default.
**Disadvantages:** Performance overhead on complex queries, less control than raw SQL for advanced scenarios.

## DbContext

`DbContext` is the central class in EF Core. It represents a **session with the database** and is responsible for:

- Managing the database connection
- Exposing `DbSet<T>` collections (one per entity)
- Tracking changes to entities
- Translating LINQ queries to SQL
- Executing queries and saving data via `SaveChanges()`
- Managing transactions (implicit per `SaveChanges()`)

```csharp
public class AppDbContext : DbContext
{
    public DbSet<Tournament> Tournaments { get; set; }
    public DbSet<Match> Matches { get; set; }

    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // Fluent API configuration — relationships, constraints, indexes
        modelBuilder.Entity<Tournament>()
            .HasMany(t => t.Matches)
            .WithOne(m => m.Tournament)
            .HasForeignKey(m => m.TournamentId);
    }
}
```

> `DbContext` must be registered as **Scoped** in ASP.NET Core — one instance per HTTP request. Never register it as Singleton (captive dependency problem — see [[6.3 Dependency Injection]]).

## Code-First Migrations

Migrations evolve the database schema incrementally as your model changes, without losing data. EF Core compares your current model to a stored snapshot and generates migration files.

```bash
dotnet ef migrations add AddTournamentFormat   # create migration file
dotnet ef database update                     # apply pending migrations to DB
dotnet ef migrations remove                   # undo last migration (if not applied to DB)
dotnet ef migrations script --idempotent      # generate SQL script for production deployment
```

Each migration has `Up()` (apply) and `Down()` (rollback):

```csharp
public partial class AddTournamentFormat : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.AddColumn<string>(name: "Format", table: "Tournaments", nullable: true);
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropColumn(name: "Format", table: "Tournaments");
    }
}
```

`__EFMigrationsHistory` table tracks which migrations have been applied.

**Best practices:**
- Always review generated migration code before applying
- Use `--idempotent` for production deployments (safe to re-run)
- Never edit the model snapshot manually
- Name migrations descriptively: `AddTournamentFormat`, not `Migration3`

## Change Tracking

EF Core tracks every entity it loads so it knows what to persist on `SaveChanges()`.

**How it works (Snapshot tracking — default):**
1. Entity is loaded → EF takes a snapshot of original values
2. You modify properties
3. `SaveChanges()` → EF compares current values to snapshot → generates INSERT/UPDATE/DELETE

```csharp
var tournament = await _context.Tournaments.FindAsync(id); // tracked, snapshot taken
tournament.MaxTeams = 32;                                  // EF detects the change
await _context.SaveChangesAsync();                         // UPDATE Tournaments SET MaxTeams = 32 WHERE Id = @id
```

**Entity states:**

| State | Meaning |
|---|---|
| `Added` | New entity, will INSERT |
| `Modified` | Existing entity with changes, will UPDATE |
| `Deleted` | Marked for removal, will DELETE |
| `Unchanged` | Loaded, no changes |
| `Detached` | Not tracked by this context |

### AsNoTracking — read-only performance

For read-only queries, disable tracking — no snapshot is taken, less memory and CPU:

```csharp
// ✅ Use for GET endpoints, reports, any query you won't modify
var tournaments = await _context.Tournaments
    .AsNoTracking()
    .Where(t => t.Status == TournamentStatus.Live)
    .ToListAsync();
```

## Loading Related Data

### Eager loading — preferred

Loads related data in one query with a JOIN:

```csharp
var tournaments = await _context.Tournaments
    .Include(t => t.Matches)
    .ThenInclude(m => m.Games)
    .ToListAsync();
```

### Explicit loading — on demand

Load a specific navigation property for an already-loaded entity:

```csharp
await _context.Entry(tournament).Collection(t => t.Matches).LoadAsync();
```

### Lazy loading — avoid in APIs

Automatically loads navigation properties on first access. Requires proxy configuration.

```csharp
// ⚠️ Avoid — causes N+1 when accessed in a loop
var matches = await _context.Matches.ToListAsync();
foreach (var m in matches)
    Console.WriteLine(m.Games.Count); // triggers a separate query per match
```

---

## N+1 Query Problem

The most common EF Core performance mistake: loading a collection, then accessing a navigation property inside a loop.

```csharp
// ❌ N+1: 1 query for matches + N queries for games (one per match)
var matches = await _context.Matches.ToListAsync();
foreach (var m in matches)
    Console.WriteLine(m.Games.Count); // each access hits the database

// ✅ Fixed: single query with JOIN via Include
var matches = await _context.Matches
    .Include(m => m.Games)
    .ToListAsync();
```

## Projections — Select Only What You Need

Projections load only the columns needed for the response, reducing data transfer and memory:

```csharp
// ❌ Loads all columns into memory
var tournaments = await _context.Tournaments.ToListAsync();

// ✅ Loads only Id, Name, MaxTeams — generates SELECT Id, Name, MaxTeams FROM Tournaments
var dtos = await _context.Tournaments
    .Where(t => t.Status != TournamentStatus.Cancelled)
    .Select(t => new TournamentDto(t.Id, t.Name, t.MaxTeams))
    .ToListAsync();
```

> Projections with `.Select()` also bypass change tracking automatically — you don't need `.AsNoTracking()`.

## Transactions in EF Core

`SaveChangesAsync()` wraps all pending changes in a **single implicit transaction** automatically.

```csharp
// Implicit transaction — covers all pending changes in one SaveChanges call
_context.Matches.Add(match);
_context.Standings.Add(standing);
await _context.SaveChangesAsync(); // both are committed or both rolled back
```

For multi-step operations requiring multiple `SaveChanges` calls in one atomic unit:

```csharp
// Explicit transaction
await using var tx = await _context.Database.BeginTransactionAsync();
try
{
    _context.Matches.Add(match);
    await _context.SaveChangesAsync();

    _context.Games.Add(game);
    await _context.SaveChangesAsync();

    await tx.CommitAsync();
}
catch
{
    await tx.RollbackAsync();
    throw;
}
```

## Optimistic Concurrency

**The problem:** two users load the same entity, both modify it, last write wins and silently overwrites the first change.

**The solution:** attach a `RowVersion` timestamp to the entity. EF includes it in every `UPDATE WHERE` clause — if the row changed since you loaded it, the update affects 0 rows and EF throws `DbUpdateConcurrencyException`.

```csharp
public class Tournament
{
    public long Id { get; set; }
    public string Name { get; set; }

    [Timestamp]
    public byte[] RowVersion { get; set; }  // auto-incremented by SQL Server on every update
}
```

EF generates:
```sql
UPDATE Tournaments SET Name = @name
WHERE Id = @id AND RowVersion = @originalRowVersion  -- fails if someone else updated it
```

```csharp
try
{
    await _context.SaveChangesAsync();
}
catch (DbUpdateConcurrencyException)
{
    throw new ConflictException("The record was modified by another user. Please retry.");
}
```

> "Optimistic concurrency assumes conflicts are rare — it doesn't lock the row, but detects conflicts at save time using a rowversion token. If a conflict occurs, EF throws `DbUpdateConcurrencyException`."

## Interview Traps

- **"What is DbContext?"** — Represents a session with the database: manages connection, tracks entity changes, translates LINQ to SQL, wraps SaveChanges in a transaction.
- **"What is change tracking?"** — EF snapshots entity values on load. On `SaveChanges()`, it diffs current vs snapshot to generate the minimal INSERT/UPDATE/DELETE. Use `AsNoTracking()` to skip it for read-only queries.
- **"What is the N+1 problem?"** — Loading a collection, then accessing a navigation property in a loop triggers one query per item. Fix: use `Include()` to load it in a single JOIN query.
- **"Eager vs Lazy loading?"** — Eager (`Include`) loads related data upfront in one query. Lazy loads on property access — bad in APIs, causes N+1. Prefer eager loading.
- **"When do you use an explicit transaction?"** — When you need multiple `SaveChanges` calls to succeed or fail together as one atomic unit.
- **"What is optimistic concurrency?"** — No row locking. A rowversion stamp is added to the UPDATE WHERE clause — if it doesn't match, the update fails and `DbUpdateConcurrencyException` is thrown. Assumes conflicts are rare.
