# 4.2 Design Patterns

Reusable solutions to common design problems. You don't invent these — you recognize the situation and apply the pattern. 

---

## Creational — how objects are created

### Singleton

One instance, global access point. The class manages its own lifecycle.

```csharp
// Thread-safe via Lazy<T>
public sealed class Singleton
{
    private static readonly Lazy<Singleton> _instance = new(() => new Singleton());
    private Singleton() { }
    public static Singleton Instance => _instance.Value;
}
```

> In ASP.NET Core you almost never implement this manually. `services.AddSingleton<T>()` does it — the DI container owns the lifetime. Knowing the pattern matters; hand-rolling it in production code is a smell.

**Interview traps:**

- "Thread-safe Singleton without `lock`?" — `Lazy<T>` or a `static readonly` field. Both are guaranteed thread-safe by the CLR.
- "Problems with Singleton?" — Hidden global state, hard to test, tight coupling. Prefer DI singleton lifetime.

### Factory

Creates objects without the caller knowing which concrete type gets instantiated. Caller depends on the interface, factory owns the `new`.

```csharp
public interface INotificationSender
{
    Task SendAsync(string recipient, string message);
}

public class EmailSender : INotificationSender { ... }
public class SmsSender  : INotificationSender { ... }

public static class NotificationSenderFactory
{
    public static INotificationSender Create(string channel) => channel switch
    {
        "email" => new EmailSender(),
        "sms"   => new SmsSender(),
        _       => throw new ArgumentException($"Unknown channel: {channel}")
    };
}
```

Adding a new channel = new class + one new switch arm. The caller never changes.

**Interview traps:**

- "Factory vs `new`?" — `new` couples the caller to a concrete type. Factory hides that decision behind an interface — enabling OCP and testability.
- "Factory vs Abstract Factory?" — Factory creates one product. Abstract Factory creates families of related products (e.g. cross-platform UI: Windows buttons + checkboxes vs macOS buttons + checkboxes).

### Builder

Constructs a complex object step by step via a fluent chain. Solves "constructor with 8 parameters" — not "which type to create" (that's Factory).

```csharp
public class Tournament
{
    public string Name   { get; private set; }
    public int MaxTeams  { get; private set; }
    public string Format { get; private set; }
    public bool IsPublic { get; private set; }

    private Tournament() { }

    public class Builder
    {
        private readonly Tournament _t = new();

        public Builder WithName(string n)     { _t.Name = n;      return this; }
        public Builder WithMaxTeams(int m)    { _t.MaxTeams = m;  return this; }
        public Builder WithFormat(string f)   { _t.Format = f;    return this; }
        public Builder Public(bool p = true)  { _t.IsPublic = p;  return this; }

        public Tournament Build()
        {
            if (string.IsNullOrEmpty(_t.Name)) throw new InvalidOperationException("Name required.");
            if (_t.MaxTeams < 2)               throw new InvalidOperationException("Need ≥2 teams.");
            return _t;
        }
    }
}

var t = new Tournament.Builder()
    .WithName("Spring Open 2026")
    .WithMaxTeams(16)
    .WithFormat("SingleElim")
    .Public()
    .Build();
```

> `WebApplication.CreateBuilder(args)` and `StringBuilder` are Builder in the BCL. The fluent chain and a terminal method (`Build()`, `ToString()`) are the tell.

**Interview trap:** "Builder vs Factory?" — Factory decides _which type_. Builder builds the _same type_ step by step with validation at the end.

## Structural — how classes and objects are composed

### Adapter

Wraps an incompatible interface so it fits what your code expects. The go-to pattern when integrating third-party SDKs.

```csharp
// Your contract
public interface IPaymentGateway
{
    Task<bool> ChargeAsync(decimal amount, string currency);
}

// Third-party SDK — incompatible signature
public class StripeClient
{
    public Task<StripeResult> CreateCharge(int cents, string currency) { ... }
}

// Adapter — translates between the two
public class StripeAdapter(StripeClient stripe) : IPaymentGateway
{
    public async Task<bool> ChargeAsync(decimal amount, string currency)
    {
        var result = await stripe.CreateCharge((int)(amount * 100), currency);
        return result.Status == "succeeded";
    }
}
```

Your application only knows `IPaymentGateway` — switching from Stripe to another provider means a new adapter, zero changes to business logic.

### Decorator

Wraps an object to add behavior without modifying the original class. Stackable at runtime.

```csharp
public interface IMatchService
{
    Task<MatchDto> GetAsync(int id);
}

// Core implementation
public class MatchService(IMatchRepository repo) : IMatchService
{
    public async Task<MatchDto> GetAsync(int id) => await repo.GetAsync(id);
}

// Decorator — adds caching transparently
public class CachedMatchService(IMatchService inner, IMemoryCache cache) : IMatchService
{
    public async Task<MatchDto> GetAsync(int id)
    {
        if (cache.TryGetValue($"match:{id}", out MatchDto dto)) return dto;
        dto = await inner.GetAsync(id);
        cache.Set($"match:{id}", dto, TimeSpan.FromMinutes(5));
        return dto;
    }
}

// Stack another decorator — adds logging on top of caching
public class LoggedMatchService(IMatchService inner, ILogger<LoggedMatchService> log) : IMatchService
{
    public async Task<MatchDto> GetAsync(int id)
    {
        log.LogInformation("Fetching match {Id}", id);
        return await inner.GetAsync(id);
    }
}
```

> **ASP.NET Core middleware IS Decorator** — each middleware wraps the next, adding behaviour (auth, logging, exception handling) without touching what's inside.

**Interview traps:**

- "Decorator vs Inheritance?" — Inheritance is compile-time and static. Decorator is runtime and composable — you can stack decorators without creating a class explosion.
- "Decorator vs Proxy?" — Same structure, different intent. Decorator _adds_ behavior. Proxy _controls access_ (see below).

---

### Proxy

Same interface as the real object, but intercepts calls to control access, add lazy loading, or cache results. Structurally identical to Decorator — the distinction is intent.

```csharp
public interface ITournamentRepository
{
    Task<Tournament?> GetByIdAsync(long id);
}

// Proxy — adds a cache layer in front of the real repository
public class CachingTournamentProxy(
    ITournamentRepository inner,
    IMemoryCache cache) : ITournamentRepository
{
    public async Task<Tournament?> GetByIdAsync(long id)
    {
        var key = $"tournament:{id}";
        if (cache.TryGetValue(key, out Tournament? t)) return t;

        t = await inner.GetByIdAsync(id);
        if (t is not null) cache.Set(key, t, TimeSpan.FromMinutes(10));
        return t;
    }
}
```

**Decorator vs Proxy in one sentence:** Decorator wraps to _enrich_; Proxy wraps to _guard or optimise_.

Real-world proxies: EF Core lazy-loading proxies, `Castle.DynamicProxy` (used by Moq internally), API gateways.

### Facade

Hides a complex subsystem behind a single, simple entry point. Callers don't need to know about the internals — they just call the facade.

```csharp
// Subsystem — multiple services, complex orchestration
public class TournamentFacade(
    ITournamentService tournaments,
    ITeamService teams,
    INotificationService notifications)
{
    // One method hides the multi-step orchestration
    public async Task<TournamentDto> CreateWithTeamsAsync(CreateTournamentRequest req)
    {
        var tournament = await tournaments.CreateAsync(req);
        await teams.RegisterBulkAsync(tournament.Id, req.TeamIds);
        await notifications.SendInvitesAsync(req.TeamIds, tournament);
        return tournament;
    }
}
```

> Facade is about convenience, not access control. If you find your controller orchestrating 4+ services directly, a Facade (or a dedicated command handler) is usually the fix.

### Repository (Structural / Architectural)

Abstracts data access behind an interface. The application layer talks to the contract — never to EF Core directly.

```csharp
public interface ITournamentRepository
{
    Task<Tournament?> GetByIdAsync(long id, long tenantId);
    Task AddAsync(Tournament tournament);
    Task<IReadOnlyList<Tournament>> GetActiveAsync(long tenantId);
}

public class TournamentRepository(AppDbContext ctx) : ITournamentRepository
{
    public Task<Tournament?> GetByIdAsync(long id, long tenantId)
        => ctx.Tournaments.FirstOrDefaultAsync(t => t.Id == id && t.TenantId == tenantId);

    public async Task AddAsync(Tournament t)
        => ctx.Tournaments.Add(t);

    public Task<IReadOnlyList<Tournament>> GetActiveAsync(long tenantId)
        => ctx.Tournaments
               .Where(t => t.TenantId == tenantId && t.Status == TournamentStatus.Live)
               .ToListAsync()
               .ContinueWith(r => (IReadOnlyList<Tournament>)r.Result);
}

// Service has zero EF Core imports
public class TournamentService(ITournamentRepository repo) { ... }
```

Benefits: swap ORM without touching the service layer, mock the interface in unit tests with no real DB.

## Behavioral — how objects communicate

### Strategy

Defines a family of algorithms behind a common interface. The caller doesn't care which one runs — it just calls the interface.

```csharp
public interface IGameResultPipelineStep
{
    Task ExecuteAsync(GameResultContext ctx);
}

public class ScoreGameStep         : IGameResultPipelineStep { ... }
public class UpdateGroupStatsStep  : IGameResultPipelineStep { ... }
public class AdvanceBracketStep    : IGameResultPipelineStep { ... }

public class GameResultPipeline(IEnumerable<IGameResultPipelineStep> steps)
{
    public async Task RunAsync(GameResultContext ctx)
    {
        foreach (var step in steps)
            await step.ExecuteAsync(ctx);
    }
}
```

Adding a new step = new class + register in DI. Zero changes to the pipeline. Strategy + OCP working together.

**Interview traps:**

- "Strategy vs if/else?" — if/else bakes the algorithm into the caller, violates OCP. Strategy externalises it — each algorithm is a separate class.
- "Strategy vs State?" — Strategy: the _caller_ swaps the algorithm from outside. State: the _object_ swaps its own behaviour internally based on its own state.
- "`IComparer<T>` passed to `Array.Sort()` — which pattern?" — Strategy.

---

### Observer

One publisher notifies many subscribers when something happens. Subscribers don't know about each other. C# `event`/`delegate` is the built-in Observer.

```csharp
public class TournamentService
{
    public event EventHandler<MatchFinishedEventArgs>? MatchFinished;

    public void FinishMatch(Match match)
    {
        // update state...
        MatchFinished?.Invoke(this, new MatchFinishedEventArgs(match));
    }
}

// Subscribers register independently — service doesn't know them
_tournamentService.MatchFinished += async (_, e)
    => await _hubContext.Clients.All.SendAsync("MatchFinished", e.Match);

_tournamentService.MatchFinished += async (_, e)
    => await _notificationService.NotifyAsync(e.Match);
```

**Interview traps:**

- "Memory leak risk?" — Subscribing with `+=` and never unsubscribing keeps the subscriber alive. The publisher holds a reference, so the GC can't collect the subscriber. Always `−=` when done, or use weak references.
- "Observer vs Pub/Sub?" — Observer: publisher knows its subscribers directly. Pub/Sub: a message broker sits in between (RabbitMQ, Azure Service Bus) — publisher and subscriber are fully decoupled and don't know each other exist.

### Command

Encapsulates a request as an object. Decouples the sender (who asks) from the handler (who does the work). In modern .NET the pattern is almost always implemented via **MediatR**.

```csharp
// Without MediatR — raw Command pattern
public interface ICommand<TResult> { }
public interface ICommandHandler<TCommand, TResult>
{
    Task<TResult> HandleAsync(TCommand command);
}

public record CreateTournamentCommand(string Name, int MaxTeams, long TenantId)
    : ICommand<TournamentDto>;

public class CreateTournamentHandler(ITournamentRepository repo)
    : ICommandHandler<CreateTournamentCommand, TournamentDto>
{
    public async Task<TournamentDto> HandleAsync(CreateTournamentCommand cmd)
    {
        var t = new Tournament(cmd.Name, cmd.MaxTeams, cmd.TenantId);
        await repo.AddAsync(t);
        return new TournamentDto(t.Id, t.Name);
    }
}
```

**With MediatR — how it looks in practice:**

```csharp
// Command (request)
public record CreateTournamentCommand(string Name, int MaxTeams, long TenantId)
    : IRequest<TournamentDto>;

// Handler — MediatR finds and invokes it automatically
public class CreateTournamentHandler(ITournamentRepository repo)
    : IRequestHandler<CreateTournamentCommand, TournamentDto>
{
    public async Task<TournamentDto> Handle(
        CreateTournamentCommand cmd, CancellationToken ct)
    {
        var t = new Tournament(cmd.Name, cmd.MaxTeams, cmd.TenantId);
        await repo.AddAsync(t);
        return new TournamentDto(t.Id, t.Name);
    }
}

// Controller — sends the command, never calls the handler directly
[HttpPost]
public async Task<IActionResult> Create(
    CreateTournamentCommand cmd,
    IMediator mediator)
{
    var dto = await mediator.Send(cmd);
    return CreatedAtAction(nameof(Get), new { id = dto.Id }, dto);
}
```

MediatR also supports **pipeline behaviors** — cross-cutting concerns (logging, validation, auth) that wrap every command/query handler, similar to middleware.

```csharp
public class LoggingBehaviour<TRequest, TResponse>(ILogger<LoggingBehaviour<TRequest, TResponse>> log)
    : IPipelineBehavior<TRequest, TResponse>
{
    public async Task<TResponse> Handle(
        TRequest request, RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        log.LogInformation("Handling {Request}", typeof(TRequest).Name);
        var response = await next();
        log.LogInformation("Handled {Request}", typeof(TRequest).Name);
        return response;
    }
}
```

**Interview traps:**

- "What problem does Command/MediatR solve?" — Decouples controllers from service classes. Controllers send commands; they don't know which class handles them. Also enables CQRS — Commands mutate state, Queries read it, each with its own handler.
- "Command vs Strategy?" — Strategy swaps _how_ to do something. Command encapsulates _what_ to do and _who_ asked.
- "MediatR pipeline behavior vs middleware?" — Both are Decorator chains. Middleware runs per HTTP request. MediatR behaviors run per command/query — more granular, no HTTP dependency.

## Interview Traps (Cross-pattern)

- **"Factory vs Builder?"** — Factory decides _which type_ to create. Builder constructs the _same type_ step by step.
- **"Decorator vs Proxy?"** — Identical structure, different intent. Decorator adds behavior. Proxy controls access or optimizes (caching, lazy load, auth).
- **"Strategy vs State?"** — Strategy: external swap by the caller. State: internal swap by the object itself.
- **"Where is Observer in .NET?"** — `event`/`delegate`, `IObservable<T>`/`IObserver<T>`, SignalR, MediatR notifications.
- **"Where is Builder in the BCL?"** — `WebApplication.CreateBuilder()`, `StringBuilder`, `HttpClientBuilder`, `IServiceCollection` fluent registration.
- **"Where is Decorator in ASP.NET Core?"** — Middleware pipeline. Each middleware wraps the next.
- **"Can you name a pattern you've used in your own project?"** — TO2: Strategy in the game result pipeline, Repository for data access, Command via MediatR if implemented, Decorator in middleware.
