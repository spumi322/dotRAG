# 3.1 Delegates & Events

## Delegates

A delegate is a **type-safe function pointer** — a reference to a method with a specific signature. It lets you pass methods as arguments, store them in variables, and invoke them later.

```csharp
// Delegate type declaration
public delegate int MathOperation(int a, int b);

// Method that matches the signature
public int Add(int a, int b) => a + b;

// Assign and invoke
MathOperation op = Add;
int result = op(3, 4); // 7
```

In practice, you rarely declare custom delegate types. Use `Func<>`, `Action<>`, and `Predicate<T>` from the BCL instead.

---

## Multicast delegates

A delegate can hold references to **more than one method**. When invoked, it calls all methods in the order they were added.

```csharp
Action<string> notify = null;
notify += SendEmail;
notify += SendSms;
notify += LogToDb;

notify("Tournament started"); // calls all three in order
```

**Removing a method:**

```csharp
notify -= SendSms;
notify("Round 2"); // now calls SendEmail and LogToDb only
```

> If a multicast delegate has a return type, only the **last** method's return value is used. This is why multicast delegates are typically `void` (`Action<T>`).

---

## Anonymous methods

Inline method defined with the `delegate` keyword. Older syntax — superseded by lambda expressions.

```csharp
Action<string> log = delegate(string msg)
{
    Console.WriteLine($"[LOG] {msg}");
};

log("Tournament created");
```

## Events

An event is a **restricted delegate** — it can only be subscribed to (`+=`) or unsubscribed from (`-=`) from outside the declaring class. Only the class itself can invoke (raise) it.

```csharp
public class TournamentService
{
    // Event declaration — uses EventHandler<T> by convention
    public event EventHandler<string> TournamentStarted;

    public void Start(string name)
    {
        // Invoke the event (only possible inside the class)
        TournamentStarted?.Invoke(this, name);
    }
}

// Subscriber
var service = new TournamentService();
service.TournamentStarted += (sender, name) =>
    Console.WriteLine($"Tournament '{name}' started!");

service.Start("ESL Pro League"); // triggers the event
```

### Event vs delegate — the distinction

| | Delegate | Event |
|---|---|---|
| Invocation | Anyone who has the reference | Only the declaring class |
| Assignment (`=`) | ✅ anyone can reassign | ❌ only the class (external code can only `+=`/`-=`) |
| Usage | Callbacks, LINQ, strategy | Observer pattern, pub/sub |

```csharp
// ❌ External code cannot invoke or reassign an event
service.TournamentStarted("ESL"); // compile error — can't invoke
service.TournamentStarted = null; // compile error — can't assign
```

> Use `event` when you want to prevent external callers from clearing all subscribers with `= null` or from invoking the delegate directly. Events enforce the **observer pattern** at compile time.

### EventHandler convention

```csharp
// Standard signature: (object sender, TEventArgs e)
public event EventHandler TournamentEnded;            // no data
public event EventHandler<MatchResult> MatchCompleted; // with data

// Custom EventArgs
public class MatchResult : EventArgs
{
    public int HomeScore { get; set; }
    public int AwayScore { get; set; }
}
```

---

## Interview Traps

- **"What's the difference between a delegate and an event?"** — A delegate is a type that holds method references. An event is a delegate field with restricted access: external code can only subscribe/unsubscribe, not invoke or reassign.
- **"Can a multicast delegate have a return value?"** — Technically yes, but only the last method's return value is accessible. That's why multicast delegates are typically `void`.
- **"Why use `EventHandler<T>` instead of a custom delegate?"** — Convention. `EventHandler<T>` standardises the `(sender, args)` signature used throughout .NET, making events predictable and compatible with framework tooling.
