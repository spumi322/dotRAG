# 2.3 Delegates, Lambdas & Closures

## Delegates

A delegate is a **type-safe function pointer** — a variable that holds a reference to a method. It lets you pass behaviour (a method) around like data.

In modern C# you almost never declare a custom delegate type. The BCL gives you three built-in ones that cover everything:

|Type|Signature|Use case|
|---|---|---|
|`Func<T, TResult>`|Takes inputs, returns a value|LINQ `Select`, `Where`, transformations|
|`Action<T>`|Takes inputs, returns void|Callbacks, logging, side effects|
|`Predicate<T>`|Takes one input, returns bool|Legacy APIs (`List.FindAll`) — use `Func<T, bool>` in modern code|

```csharp
Func<int, int, int>  add  = (a, b) => a + b;
Action<string>       log  = msg   => Console.WriteLine(msg);
Func<Tournament, bool> isLive = t => t.Status == TournamentStatus.Live;
```

A delegate can hold more than one method (multicast) — when invoked, it calls all of them in order. This is how events are built under the hood.

## Events

An event is a **delegate with restricted access**. External code can only subscribe (`+=`) or unsubscribe (`-=`). Only the class that declares the event can fire it.

```csharp
public class TournamentService
{
    public event EventHandler<string> TournamentStarted;

    public void Start(string name)
    {
        TournamentStarted?.Invoke(this, name); // only callable from inside
    }
}

var svc = new TournamentService();
svc.TournamentStarted += (_, name) => Console.WriteLine($"{name} started");

// ❌ svc.TournamentStarted("Summer Cup"); — compile error, can't invoke from outside
// ❌ svc.TournamentStarted = null; — compile error, can't reassign from outside
```

||Delegate|Event|
|---|---|---|
|Who can invoke|Anyone with the reference|Only the declaring class|
|External assignment (`=`)|✅|❌|
|Use case|Callbacks, LINQ, strategy pattern|Observer / pub-sub|

> In practice: if you're building a publisher-subscriber relationship between classes, use `event`. For passing a callback into a method, use `Func` or `Action`.

## Lambda expressions

A lambda is an **inline anonymous function**. It's the syntax you write every time you use LINQ or EF Core.

```csharp
// Expression lambda — single expression, no braces, no return keyword
t => t.Name

// Statement lambda — multiple lines, needs braces and return
t =>
{
    var prefix = t.IsActive ? "[LIVE]" : "[DONE]";
    return $"{prefix} {t.Name}";
}
```

You use lambdas daily — you just don't always think of them as lambdas:

```csharp
// EF Core — lambda tells EF what SQL to generate
var active = await _context.Tournaments
    .Where(t => t.Status == TournamentStatus.Live)       // Func<Tournament, bool>
    .Include(t => t.Teams)                               // navigation property
    .OrderBy(t => t.StartDate)                           // sort key
    .Select(t => new TournamentDto(t.Id, t.Name))        // projection
    .ToListAsync();
```

Each argument inside `.Where(...)`, `.Select(...)` etc. is a lambda. EF Core's `IQueryable` machinery converts them into SQL — that's only possible because they're passed as **expression trees**, not just compiled code.

## Closures

A closure is what happens when a lambda **captures a variable** from the surrounding scope. The variable stays alive as long as the lambda exists — even after the outer method has returned.

```csharp
var threshold = 16;

// 'threshold' is captured — the lambda closes over it
var largeTournaments = tournaments.Where(t => t.MaxTeams > threshold);
```

This is usually what you want. But there's a classic trap with loops.

### The for-loop capture trap

```csharp
var actions = new List<Action>();

for (int i = 0; i < 3; i++)
    actions.Add(() => Console.WriteLine(i)); // captures the variable i, not its value

actions.ForEach(a => a());
// Output: 3, 3, 3  ← NOT 0, 1, 2
```

By the time the lambdas run, the loop is done and `i` is 3. All three lambdas share the **same variable**.

**Fix — capture the value by copying it:**

```csharp
for (int i = 0; i < 3; i++)
{
    int copy = i;                              // new variable per iteration
    actions.Add(() => Console.WriteLine(copy));
}
// Output: 0, 1, 2 ✅
```

With `foreach` and modern `for` using `let` semantics this is less of an issue — but it still bites people with `Task.Run` inside loops:

```csharp
// ❌ Common real-world version of the same bug
for (int i = 0; i < 5; i++)
    Task.Run(() => ProcessItem(i)); // all tasks may see i = 5
```

## Interview Traps

- **"What's the difference between a delegate and an event?"** — A delegate is a type that holds method references — anyone with access can invoke or reassign it. An event wraps a delegate with access restrictions: external code can only subscribe/unsubscribe, the class alone fires it.
- **"What is a lambda?"** — An inline anonymous function. Under the hood it compiles to either a delegate or an expression tree depending on context. In EF Core queries, lambdas become expression trees so the ORM can translate them to SQL.
- **"What is a closure?"** — When a lambda captures a variable from its outer scope. The variable's lifetime is extended for as long as the lambda exists.
- **"What does this print?"**
    
    ```csharp
    for (int i = 0; i < 3; i++)    actions.Add(() => Console.WriteLine(i));actions.ForEach(a => a());
    ```
    
    — `3, 3, 3`. The lambda captures the variable `i`, not its value at time of capture. By execution time, `i` is 3.
