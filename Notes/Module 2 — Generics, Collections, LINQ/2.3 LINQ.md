# 2.3 LINQ

## What LINQ is

LINQ (Language-Integrated Query) provides a unified syntax for querying different data sources — in-memory collections, databases, XML — directly in C#. Type-safe, compile-time checked, IntelliSense support.

**Two syntaxes (method syntax preferred):**

```csharp
// Query syntax (SQL-like)
var result = from u in users where u.Age >= 18 select u;

// Method syntax (preferred — more composable)
var result = users.Where(u => u.Age >= 18);
```

---

## Deferred execution

LINQ queries are **lazy** — they don't execute when you write them. They execute when you enumerate the results.

```csharp
var numbers = new List<int> { 1, 2, 3 };
var query = numbers.Where(n => n > 1); // NOT executed yet — just a query object

numbers.Add(4); // modify source AFTER defining query

foreach (var n in query) // executes NOW — sees 2, 3, 4
    Console.WriteLine(n);
```

**What triggers execution:**
- Iteration: `foreach`, `.ToList()`, `.ToArray()`, `.ToDictionary()`
- Aggregation: `.Count()`, `.Sum()`, `.Any()`, `.First()`, `.Single()`

**When to materialise with `.ToList()`:**
- You'll enumerate multiple times
- You need count AND items
- You want a snapshot ignoring future source changes
- You're returning from a method and want to stop deferred execution

## Core operators

### Filtering

```csharp
var adults = users.Where(u => u.Age >= 18);
var active = products.Where(p => p.IsActive && p.Stock > 0);
```

### Projection

```csharp
// Select — transform each element (one-to-one)
var names = users.Select(u => u.Name);
var dtos  = users.Select(u => new UserDto(u.Id, u.Name));

// SelectMany — flatten nested collections (one-to-many)
var allOrders = customers.SelectMany(c => c.Orders);
// Input:  [Customer1[Order1, Order2], Customer2[Order3]]
// Output: [Order1, Order2, Order3]
```

### Sorting

```csharp
var sorted = products.OrderBy(p => p.Price);
var desc   = products.OrderByDescending(p => p.Price);
var multi  = products.OrderBy(p => p.Category).ThenBy(p => p.Name);
```

### Grouping

```csharp
var byCategory = products.GroupBy(p => p.Category);

foreach (var group in byCategory)
{
    Console.WriteLine($"Category: {group.Key}");
    foreach (var p in group)
        Console.WriteLine($"  - {p.Name}");
}

// With aggregation
var counts = products
    .GroupBy(p => p.Category)
    .Select(g => new { Category = g.Key, Count = g.Count() });
```

### Joining

```csharp
// Inner join
var result = users.Join(
    orders,
    user  => user.Id,
    order => order.UserId,
    (user, order) => new { user.Name, order.Total }
);
```

### Element operators

```csharp
var first           = users.First();                     // throws if empty
var firstOrDefault  = users.FirstOrDefault();            // null/default if empty

var single          = users.Single();                    // throws if 0 or >1
var singleOrDefault = users.SingleOrDefault();           // throws if >1

// With predicate
var admin   = users.First(u => u.Role == "Admin");
var premium = users.SingleOrDefault(u => u.IsPremium);  // expects 0 or 1
```

### Existence checks

```csharp
bool hasAdults = users.Any(u => u.Age >= 18); // at least one
bool allActive = users.All(u => u.IsActive);  // every one
```

### Pagination

```csharp
var page = products
    .OrderBy(p => p.Id)
    .Skip(pageSize * pageNumber)
    .Take(pageSize);
```

## Common pitfalls

### 1. Multiple enumeration — query runs twice

```csharp
var query = dbContext.Products.Where(p => p.Price > 100);

var count = query.Count();   // ⚠️ executes SQL
var list  = query.ToList();  // ⚠️ executes SQL AGAIN — two round trips

// ✅ Fix: materialise once
var list  = query.ToList();
var count = list.Count;
```

### 2. Modifying collection during iteration

```csharp
var numbers = new List<int> { 1, 2, 3 };
foreach (var n in numbers)
    numbers.Add(n * 2); // ❌ InvalidOperationException: collection modified

// ✅ Fix: iterate a copy
foreach (var n in numbers.ToList())
    numbers.Add(n * 2);
```

### 3. Forgetting deferred execution with captured variables

```csharp
var threshold = 100;
var query = products.Where(p => p.Price > threshold);
threshold = 200; // changed AFTER query definition
var list = query.ToList(); // uses threshold = 200, not 100
```

## IEnumerable\<T\> vs IQueryable\<T\>

Both represent sequences of elements but execute in different places.

### IEnumerable\<T\> — in-memory execution

- Defined in `System.Collections.Generic`
- Evaluates queries **in memory** (client-side, in C#)
- Uses delegates (`Func<T, bool>`)
- Once data is pulled from the database, all further operations happen in C#

```csharp
// ❌ Loads entire Products table, then filters in C#
IEnumerable<Product> all = _context.Products; // loads ALL rows
var expensive = all.Where(p => p.Price > 100); // filtered in memory
```

### IQueryable\<T\> — database execution

- Defined in `System.Linq`, extends `IEnumerable<T>`
- Builds an **expression tree** that EF Core translates to SQL
- Uses expressions (`Expression<Func<T, bool>>`)

```csharp
// ✅ Database does the filtering — only matching rows transferred
IQueryable<Product> query = _context.Products; // no query yet
var expensive = query.Where(p => p.Price > 100); // → SQL WHERE Price > 100
var list = await expensive.ToListAsync();         // SQL executes NOW
```

### Comparison

| | `IEnumerable<T>` | `IQueryable<T>` |
|---|---|---|
| Execution | In memory (C#) | Database (SQL) |
| Filter type | `Func<T, bool>` | `Expression<Func<T, bool>>` |
| Deferred execution | ✅ | ✅ |
| Best for | In-memory collections | EF Core, database queries |
| Performance risk | Loads all data first | Only fetches matching rows |

> **Interview trap:** Calling `.AsEnumerable()` on an `IQueryable` switches to in-memory evaluation for everything that follows — subsequent `.Where()` calls will filter in C# after fetching all rows.

## Interview Traps

- **"What's deferred execution?"** — LINQ queries don't run when written. They run when enumerated (`foreach`, `.ToList()`, `.Count()`). This allows composition and reflects changes to the source.
- **"Difference between `First()` and `Single()`?"** — `First()` returns the first element, throws if empty. `Single()` expects exactly one element, throws if zero OR multiple. Use `Single()` when uniqueness is expected (e.g., "get user by email").
- **"`Select()` vs `SelectMany()`?"** — `Select()` is one-to-one transform. `SelectMany()` flattens one-to-many (customers → all their orders in a flat sequence).
- **"Problem with `var count = q.Count(); var list = q.ToList();`?"** — The query runs twice. For a database query that means two round trips. Fix: materialise first, then use `.Count` on the list.
- **"When does `IQueryable` become `IEnumerable`?"** — When you call `.ToList()`, `.ToArray()`, `.AsEnumerable()`, or iterate with `foreach`. At that point, SQL executes and data moves to memory.
