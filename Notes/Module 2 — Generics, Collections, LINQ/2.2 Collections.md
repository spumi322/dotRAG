# 2.2 Collections

## Collection interfaces

### IEnumerable\<T\>

The most basic interface — "you can loop over this." Any type implementing it works with `foreach` and LINQ.

- `GetEnumerator()` — returns an `IEnumerator<T>`

### IEnumerator\<T\>

The cursor that moves through the collection. `foreach` calls `GetEnumerator()`, then repeatedly calls `MoveNext()` and reads `Current` until `MoveNext()` returns false.

- `MoveNext()` — advance, returns false when done
- `Current` — element at current position
- `Reset()` — go back to start (rarely used)

### ICollection\<T\>

Extends `IEnumerable<T>` — adds mutation and count.

- `Add(T)`, `Remove(T)`, `Clear()`, `Contains(T)`
- `Count`, `IsReadOnly`

### IList\<T\>

Extends `ICollection<T>` — adds index-based access.

- `this[int index]`
- `IndexOf(T)`, `Insert(int, T)`, `RemoveAt(int)`

`List<T>` implements `IList<T>`.

### IDictionary\<TKey, TValue\>

Key-value collection — look up values by key.

- `Add(TKey, TValue)`, `Remove(TKey)`, `ContainsKey(TKey)`
- `TryGetValue(TKey, out TValue)` — safe lookup, no exception on missing key
- `Keys`, `Values`

## Interface hierarchy

```
IEnumerable<T>
├── ICollection<T>
│   ├── IList<T>              → List<T>, arrays
│   └── IDictionary<K,V>     → Dictionary<K,V>
├── IReadOnlyCollection<T>
│   ├── IReadOnlyList<T>
│   └── IReadOnlyDictionary<K,V>
└── (Concurrent)
    ├── ConcurrentDictionary<K,V>
    ├── ConcurrentQueue<T>
    └── ConcurrentBag<T>
```

## Read-only interfaces

### IReadOnlyList\<T\>

Read-only indexed access — `Count` and `this[int index]`, no mutation methods. Use as **return type** to signal callers cannot modify the collection.

```csharp
public class TournamentService
{
    private readonly List<Tournament> _tournaments = new();

    public IReadOnlyList<Tournament> GetAll() => _tournaments.AsReadOnly();
}
```

> Returning `IReadOnlyList<T>` instead of `List<T>` is encapsulation: you expose what callers need (read access) without leaking internal state.

### IReadOnlyDictionary\<TKey, TValue\>

Read-only key-value access. Exposes `TryGetValue`, `ContainsKey`, `Keys`, `Values` — no `Add`/`Remove`.

Both `IReadOnlyList<T>` and `IReadOnlyDictionary<K,V>` are **covariant** (`out T`) — you can return a `List<DerivedType>` where `IReadOnlyList<BaseType>` is expected.

## Thread-safe collections (`System.Collections.Concurrent`)

### ConcurrentDictionary\<TKey, TValue\>

Thread-safe dictionary — no explicit locking needed for reads and writes.

Key methods:

- `TryAdd(TKey, TValue)` — adds if key absent
- `TryRemove(TKey, out TValue)` — removes and returns value
- `TryGetValue(TKey, out TValue)` — safe lookup
- `GetOrAdd(TKey, Func<TKey, TValue>)` — atomic get-or-create
- `AddOrUpdate(TKey, TValue, Func<TKey,TValue,TValue>)` — atomic add-or-update

```csharp
var cache = new ConcurrentDictionary<string, TournamentDto>();

// Atomic: calls LoadFromDb only if the key doesn't exist
var dto = cache.GetOrAdd(tournamentId, id => LoadFromDb(id));
```

**When to use:** ASP.NET Core in-memory caching, background services, parallel processing pipelines. For single-threaded code, `Dictionary<K,V>` is faster.

Other concurrent collections (awareness level):

- `ConcurrentQueue<T>` — thread-safe FIFO
- `ConcurrentStack<T>` — thread-safe LIFO
- `ConcurrentBag<T>` — thread-safe unordered
- `BlockingCollection<T>` — producer-consumer wrapper

## Concrete data structures

| Type | Ordered | Duplicates | Access | Notes |
|---|---|---|---|---|
| `List<T>` | ✅ insertion | ✅ | By index O(1) | General purpose, array-backed |
| `Dictionary<K,V>` | ❌ | Keys: ❌ | By key O(1) | Hash-based |
| `HashSet<T>` | ❌ | ❌ | — | Unique elements, set operations |
| `Queue<T>` | ✅ FIFO | ✅ | — | `Enqueue`/`Dequeue` |
| `Stack<T>` | ✅ LIFO | ✅ | — | `Push`/`Pop` |
| `LinkedList<T>` | ✅ | ✅ | — | O(1) insert/remove at known node |
| `SortedDictionary<K,V>` | ✅ by key | Keys: ❌ | By key O(log n) | Red-black tree |

## Interview Traps

- **"Difference between `IEnumerable` and `ICollection`?"** — `IEnumerable` is read-only iteration only. `ICollection` adds mutation (`Add`/`Remove`) and `Count`.
- **"Why return `IReadOnlyList<T>` instead of `List<T>` from a service?"** — Encapsulation. Callers get no ability to mutate internal state. Intent is explicit at the interface level.
- **"How does `foreach` work under the hood?"** — Calls `GetEnumerator()`, then loops `MoveNext()` / `Current`. `yield return` lets the compiler build an enumerator state machine for you.
- **"When would you use `ConcurrentDictionary` over `Dictionary`?"** — Any shared mutable state accessed from multiple threads. In ASP.NET Core, `IMemoryCache` handles most of this for you — reaching for `ConcurrentDictionary` directly usually means you're building a custom cache or registry.
- **"Dictionary vs HashSet?"** — Both O(1) hash-based lookup. `HashSet<T>` stores values only; use when you care about existence, not association. Good for deduplication or set intersection/union.
