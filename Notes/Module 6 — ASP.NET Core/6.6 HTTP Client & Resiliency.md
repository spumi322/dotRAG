## 6.6 HTTP Client & Resiliency

## Why `new HttpClient()` is broken

`HttpClient` looks disposable so developers wrap it in `using` — or create a new instance per request. Both are wrong.

Two actual problems:

**Socket exhaustion** — each `new HttpClient()` opens a socket. Disposing it doesn't release the port immediately (TCP TIME_WAIT). Under load you run out of ports and requests start failing.

**Stale DNS** — a long-lived static `HttpClient` caches DNS resolution forever. If the target service changes its IP (failover, deployment), your client keeps hitting the old address.


```csharp
// ❌ Classic mistake — looks safe, causes socket exhaustion under load
public async Task<string> GetData()
{
    using var client = new HttpClient();
    return await client.GetStringAsync("https://api.example.com/data");
}

// ❌ Also wrong — static solves exhaustion but breaks DNS rotation
private static readonly HttpClient _client = new();
```

---

## `IHttpClientFactory` — the fix

Introduced in .NET Core 2.1. Manages a pool of `HttpMessageHandler` instances under the hood — reuses connections (no exhaustion) and recycles handlers on a timer (no stale DNS).


```csharp
// Register once in Program.cs
builder.Services.AddHttpClient();

// Inject and use — factory hands you a pre-configured, pooled client
public class PaymentService(IHttpClientFactory factory)
{
    public async Task<string> GetRatesAsync()
    {
        var client = factory.CreateClient();
        return await client.GetStringAsync("https://api.rates.com/latest");
    }
}
```

## Named vs Typed clients

**Named client** — pre-configure a client by string key. Simple but string-based, easy to typo.


```csharp
// Registration
builder.Services.AddHttpClient("payments", c =>
{
    c.BaseAddress = new Uri("https://api.payments.com");
    c.DefaultRequestHeaders.Add("X-Api-Key", "secret");
});

// Usage
var client = factory.CreateClient("payments");
```

**Typed client** — wrap `HttpClient` in a dedicated service class. Cleaner, DI-friendly, testable. Prefer this.


```csharp
// Registration
builder.Services.AddHttpClient<PaymentGatewayClient>(c =>
    c.BaseAddress = new Uri("https://api.payments.com"));

// The class receives HttpClient via constructor — factory manages it
public class PaymentGatewayClient(HttpClient client)
{
    public Task<PaymentResult> ChargeAsync(PaymentRequest req)
        => client.PostAsJsonAsync<PaymentResult>("/charge", req);
}
```

## Polly — awareness only

Polly is a resilience library. Attach retry and circuit breaker policies to your HTTP clients so transient failures don't crash your app.


```csharp
// Retry 3 times on transient HTTP errors — that's all you need to know at junior level
builder.Services.AddHttpClient<PaymentGatewayClient>()
    .AddStandardResilienceHandler(); // .NET 8 built-in, wraps Polly
```

You don't need to configure Polly pipelines for a junior interview. Know it exists, know what problem it solves.

## Interview Traps

- **"Why not `new HttpClient()` per request?"** — Socket exhaustion under load, and disposing doesn't release the port immediately due to TCP TIME_WAIT.
- **"Why not a static `HttpClient`?"** — Solves exhaustion but caches DNS forever. If the target service changes IP, you're stuck hitting the old one.
- **"Named vs Typed client — which do you prefer?"** — Typed. It's a proper class, injectable, mockable in tests, no magic strings.
- **"What does `IHttpClientFactory` actually pool?"** — Not `HttpClient` itself — it pools the underlying `HttpMessageHandler`. Each `CreateClient()` call returns a new `HttpClient`, but the handler (and its connection pool) is reused.
