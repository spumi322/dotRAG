# 6.5 Serialization & Error Handling

## Serialization

Serialization converts objects to JSON (or XML) for HTTP transmission. In ASP.NET Core controllers, this is automatic — you return an object, the framework serializes it.

**`System.Text.Json`** — the default serializer since ASP.NET Core 3.0 (replacing `Newtonsoft.Json`):

```csharp
// Direct usage (outside of controllers)
var json = JsonSerializer.Serialize(tournament);
var tournament = JsonSerializer.Deserialize<Tournament>(json);

// With options
var json = JsonSerializer.Serialize(dto, new JsonSerializerOptions
{
    PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
    WriteIndented = true
});
```

### Content negotiation and the `Accept` header

The client sends an `Accept` header to indicate what format it wants:
- `Accept: application/json` — JSON
- `Accept: application/xml` — XML

ASP.NET Core controllers respect this via `AddControllers().AddXmlSerializerFormatters()`. Without an XML formatter registered, XML requests fall back to JSON.

```csharp
// Global serializer options in Program.cs
builder.Services.AddControllers()
    .AddJsonOptions(o =>
    {
        o.JsonSerializerOptions.PropertyNamingPolicy = JsonNamingPolicy.CamelCase;
        o.JsonSerializerOptions.DefaultIgnoreCondition = JsonIgnoreCondition.WhenWritingNull;
    });
```

## Error handling

### Global exception handler middleware

`UseExceptionHandler()` catches all unhandled exceptions at the pipeline level. Combined with `AddProblemDetails()`, it returns structured `ProblemDetails` JSON automatically.

```csharp
builder.Services.AddProblemDetails(); // register ProblemDetails service

app.UseExceptionHandler();            // catch unhandled exceptions → ProblemDetails response
app.UseStatusCodePages();             // catch empty error responses (404, etc.) → ProblemDetails
```

### Custom exception handler

For domain-specific error mapping (e.g., `NotFoundException` → 404, `ValidationException` → 422):

```csharp
app.UseExceptionHandler(errApp => errApp.Run(async context =>
{
    var ex = context.Features.Get<IExceptionHandlerFeature>()?.Error;
    var (status, title) = ex switch
    {
        NotFoundException    => (StatusCodes.Status404NotFound, "Resource not found"),
        ValidationException  => (StatusCodes.Status422UnprocessableEntity, "Validation failed"),
        _                    => (StatusCodes.Status500InternalServerError, "An error occurred")
    };

    context.Response.StatusCode = status;
    await context.Response.WriteAsJsonAsync(new ProblemDetails
    {
        Status = status,
        Title  = title,
        Detail = ex?.Message
    });
}));
```

### Exception filters

`IExceptionFilter` / `ExceptionFilterAttribute` for controller-scoped exception handling:

```csharp
public class DomainExceptionFilter : IExceptionFilter
{
    public void OnException(ExceptionContext context)
    {
        if (context.Exception is NotFoundException ex)
        {
            context.Result = new NotFoundObjectResult(new ProblemDetails
            {
                Status = 404,
                Detail = ex.Message
            });
            context.ExceptionHandled = true;
        }
    }
}

// Register globally
builder.Services.AddControllers(o => o.Filters.Add<DomainExceptionFilter>());
```

### MVC filter pipeline execution order

```
Authorization → Resource → Action → Result → Exception
```

Exception filters only catch exceptions from controller actions and action filters — **not** from middleware or other pipeline components.

## Pagination

Never return unbounded collections. Use query parameters for pagination:

`GET /api/tournaments?page=1&pageSize=20`

Response should include metadata:

```json
{
  "data": [...],
  "page": 1,
  "pageSize": 20,
  "totalCount": 84,
  "hasNextPage": true
}
```

```csharp
[HttpGet]
public async Task<ActionResult<PagedResult<TournamentDto>>> GetAll(
    [FromQuery] int page = 1,
    [FromQuery] int pageSize = 20,
    CancellationToken cancellationToken = default)
{
    var result = await _service.GetPagedAsync(page, pageSize, cancellationToken);
    return Ok(result);
}
```

**Offset-based pagination** (page/pageSize) is simplest and sufficient for most APIs. For high-performance or infinite-scroll scenarios, **cursor-based pagination** (using an opaque cursor token) avoids the "shifting page" problem when items are inserted between pages.

## Interview Traps

- **"`System.Text.Json` vs `Newtonsoft.Json`?"** — `System.Text.Json` is built-in, faster, and allocation-efficient. `Newtonsoft.Json` is more feature-rich (better polymorphism, custom converters). Default in ASP.NET Core is `System.Text.Json`; add Newtonsoft via NuGet when advanced features are needed.
- **"How does global error handling work?"** — `UseExceptionHandler()` catches unhandled exceptions pipeline-wide. Combined with `AddProblemDetails()`, it returns RFC 9457-compliant JSON. Exception filters only handle exceptions within the MVC/controller layer.
- **"What is ProblemDetails?"** — RFC 9457 standardized error format with `type`, `title`, `status`, `detail` fields. ASP.NET Core 7+ returns this automatically via `AddProblemDetails()`.
- **"Why paginate?"** — Unbounded queries can return thousands of rows, causing memory pressure, slow responses, and poor UX. Pagination limits the result set per request and returns metadata (total count, hasNextPage) for the client to navigate.
- **"What is the `Accept` header?"** — Client tells the server its preferred response format (`application/json`, `application/xml`). The server performs content negotiation to pick the matching formatter.
