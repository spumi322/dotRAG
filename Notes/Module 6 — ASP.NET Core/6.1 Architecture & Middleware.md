# 7.1 Architecture & Middleware

## What ASP.NET Core is

ASP.NET Core is a cross-platform, high-performance framework for building modern web APIs and applications. It is a complete redesign of the legacy ASP.NET Framework.

Key characteristics:
- **Cross-platform** — runs on Windows, macOS, Linux
- **Modular** — built on NuGet packages
- **DI-first** — built-in IoC container (`IServiceProvider`), first-class citizen
- **Middleware-based** — request pipeline is fully composable
- **Kestrel** — default cross-platform web server; IIS/Nginx/Apache used as reverse proxies in production

---

## Minimal Hosting Model (.NET 6+)

Since .NET 6, everything lives in a single `Program.cs` with top-level statements. The legacy `Startup.cs` split between `ConfigureServices()` and `Configure()` is gone.

```csharp
var builder = WebApplication.CreateBuilder(args);

// 1. Register services
builder.Services.AddControllers();
builder.Services.AddScoped<IProductService, ProductService>();
builder.Services.AddDbContext<AppDbContext>(o => o.UseSqlite("..."));
builder.Services.AddProblemDetails();

var app = builder.Build();

// 2. Configure middleware pipeline
app.UseExceptionHandler();
app.UseHttpsRedirection();
app.UseAuthentication();
app.UseAuthorization();

// 3. Map endpoints
app.MapControllers();
app.MapGet("/ping", () => "pong");

app.Run();
```

> **Legacy note:** Before .NET 6, the same logic was split across `Startup.cs` (`ConfigureServices` + `Configure`) and `Program.cs` (`CreateHostBuilder`). You may see this in older codebases.

## Middleware pipeline

Middleware components form a pipeline. Each component can inspect or modify the request, pass it to the next component, and then process the response on the way back. **Order matters.**

```
Exception Handling → HTTPS Redirect → Static Files → Routing
→ CORS → Authentication → Authorization → Endpoint Execution
```

Each middleware either:
- Calls `await next(context)` to pass control to the next component, or
- Short-circuits the pipeline and writes a response directly

Custom middleware:
```csharp
app.Use(async (context, next) =>
{
    // Before — request going in
    var sw = Stopwatch.StartNew();
    await next(context);
    // After — response coming back
    sw.Stop();
    context.Response.Headers["X-Elapsed-Ms"] = sw.ElapsedMilliseconds.ToString();
});
```

## `app.Use()` vs `app.Run()` vs `app.Map()`

| Method | Passes to next? | Use for |
|---|---|---|
| `app.Use()` | ✅ Yes (calls `next`) | Most middleware — inspect/modify and continue |
| `app.Run()` | ❌ No (terminal) | End of pipeline — always generates a response |
| `app.Map()` | Branches pipeline | Separate pipeline for a specific path prefix |

```csharp
// Use — passes to next middleware
app.Use(async (context, next) =>
{
    Console.WriteLine("Before");
    await next(context);
    Console.WriteLine("After");
});

// Run — terminal, nothing after this runs
app.Run(async context =>
{
    await context.Response.WriteAsync("Final response");
});

// Map — branches for /health path only
app.Map("/health", branch =>
{
    branch.Run(async ctx => await ctx.Response.WriteAsync("Healthy"));
});
```

> **Interview trap:** If you call `app.Run()` before other middleware, everything after it is **dead code** — it never executes because `Run` short-circuits the pipeline.

---

## Interview Traps

- **"What is middleware?"** — Components in a pipeline that process HTTP requests and responses in sequence. Order matters: each can inspect/modify the request going in and the response coming back.
- **"What's the difference between `app.Use()` and `app.Run()`?"** — `Use` calls `next` and continues the pipeline. `Run` is terminal — it never calls `next`. Putting `Run` before other middleware means those others never execute.
- **"What changed in .NET 6?"** — The minimal hosting model consolidated `Startup.cs` into `Program.cs`. `ConfigureServices` and `Configure` are gone; you register services and configure middleware in a single file with top-level statements.
- **"What is Kestrel?"** — The default cross-platform web server embedded in ASP.NET Core. In production, it typically sits behind a reverse proxy (IIS, Nginx) which handles SSL termination, load balancing, and static files.
