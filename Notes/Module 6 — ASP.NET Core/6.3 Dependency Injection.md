# 6.3 Dependency Injection

## What DI is

Dependency Injection is a technique where objects receive their dependencies from the outside rather than creating them internally. In ASP.NET Core, the built-in IoC container manages object creation and lifetime.

```csharp
// ❌ Without DI — tight coupling, hard to test
public class TournamentService
{
    private readonly AppDbContext _db = new AppDbContext(); // creates its own dependency
}

// ✅ With DI — loose coupling, injectable, testable
public class TournamentService
{
    private readonly AppDbContext _db;

    public TournamentService(AppDbContext db) // container injects this
    {
        _db = db;
    }
}
```

Register services in `Program.cs`, inject via constructor:

```csharp
// Registration
builder.Services.AddScoped<ITournamentService, TournamentService>();
builder.Services.AddDbContext<AppDbContext>(o => o.UseSqlite("..."));

// Injection — container resolves automatically
public class TournamentsController(ITournamentService service) : ControllerBase
{
    [HttpGet("{id}")]
    public async Task<IActionResult> Get(long id)
        => Ok(await service.GetAsync(id));
}
```

## Service lifetimes

| Lifetime | Instance created | Use for |
|---|---|---|
| **Transient** | Every time requested | Lightweight, stateless services |
| **Scoped** | Once per HTTP request | `DbContext`, Unit of Work, anything request-specific |
| **Singleton** | Once for app lifetime | Caches, `HttpClient` factories, config wrappers |

```csharp
builder.Services.AddTransient<IEmailSender, SmtpEmailSender>();
builder.Services.AddScoped<ITournamentService, TournamentService>();
builder.Services.AddSingleton<ICacheService, MemoryCacheService>();
```

### Captive dependency problem

Never inject a **Scoped** service into a **Singleton**. The scoped instance gets captured at singleton creation time and lives for the entire app lifetime — causing stale data, connection leaks, and bugs.

```csharp
// ❌ Captive dependency — DbContext (Scoped) captured inside Singleton
public class MyCacheService // registered as Singleton
{
    public MyCacheService(AppDbContext db) { ... } // ❌ db will never be released
}

// ✅ Use IServiceScopeFactory to resolve scoped services inside a singleton
public class MyCacheService(IServiceScopeFactory scopeFactory)
{
    public async Task RefreshAsync()
    {
        using var scope = scopeFactory.CreateScope();
        var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        // db is properly scoped and disposed
    }
}
```

> The DI container throws at startup (in Development) if `ValidateScopes` is enabled — which is the default. This catches captive dependency bugs early.

## IOptions pattern

The standard way to bind `appsettings.json` sections to strongly-typed classes:

```csharp
// appsettings.json: { "JwtSettings": { "Issuer": "TO2", "ExpiryMinutes": 60 } }
public class JwtSettings
{
    public string Issuer { get; set; } = string.Empty;
    public int ExpiryMinutes { get; set; }
}

// Registration
builder.Services.Configure<JwtSettings>(builder.Configuration.GetSection("JwtSettings"));

// Injection
public class AuthService(IOptions<JwtSettings> options)
{
    private readonly JwtSettings _jwt = options.Value;
}
```

## Interview Traps

- **"What is DI and why use it?"** — Objects declare dependencies via constructor; the container injects them. Benefits: loose coupling, testability (swap real for mock), single place to configure lifetimes.
- **"Service lifetimes — when to use Scoped?"** — For anything that should live for one HTTP request. `DbContext` is the canonical example — it tracks changes across a request but must not be shared across requests.
- **"What is the captive dependency problem?"** — A Singleton holds a reference to a Scoped service. The scoped instance is never released (captured for the app lifetime). Fix: inject `IServiceScopeFactory` and create a scope manually.
- **"`IOptions` vs `IOptionsSnapshot`?"** — `IOptions` reads config once at startup (singleton). `IOptionsSnapshot` re-reads per request (scoped) — supports hot-reload scenarios.
