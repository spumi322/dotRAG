# 9.2 — Design Principles & Infrastructure

## Separation of Concerns (SoC)

Each component should do one thing and own it fully. Routing logic in controllers, business rules in services, data access in repositories — each layer knows its job and nothing else.

Violation looks like: a controller that validates input, queries the database directly, sends an email, and formats the response. When something breaks, you have no idea where to look.

Directly related to SRP (Module 4) — SoC is the architectural expression of the same idea applied across layers rather than within a single class.

---

## Loose Coupling / High Cohesion

**Loose coupling** — components depend on abstractions, not concretions. `TournamentService` depends on `ITournamentRepository`, not `TournamentRepository`. You can swap, mock, or extend without touching the caller.

**High cohesion** — things that change together, live together. A `MatchService` that handles match creation, scoring, and standing updates is cohesive. A `UtilityService` that handles email, PDF generation, and date formatting is not.

The goal: modules that are easy to change in isolation without ripple effects across the codebase.

**Interview trap:**

- _"Coupling vs cohesion — aren't they the same thing?"_ — No. Cohesion is about what's inside a module (related things together). Coupling is about between modules (how much they depend on each other). You want high cohesion and low coupling simultaneously.

---

## God Class Anti-Pattern

A class that knows too much and does too much. Grows over time as the path of least resistance — every new feature gets added to the existing big class instead of a new focused one.

Signs: 1000+ line service class, method names like `ProcessEverything()`, dozens of injected dependencies in the constructor.

Fix: split by responsibility. A `TournamentService` that handles registration, scheduling, standings, notifications, and billing should be five services.

**Interview trap:**

- _"How do you spot a God class in code review?"_ — Constructor with 8+ dependencies is the fastest signal. Each dependency usually represents a responsibility. If you can't name them all without reading the code, it's a God class.

---

## Technical Debt

The cost of shortcuts taken now that you'll pay later — with interest. Not always wrong: a deliberate shortcut to hit a deadline is acceptable if you track and address it. Accidental debt (sloppy code, no tests, wrong abstractions) compounds silently.

Common sources: no tests (refactoring becomes scary), wrong abstractions chosen early (hard to undo), copy-pasted code (same bug in five places), ignored warnings.

Managing it: name it explicitly in code (`// TODO: extract to service`), track it in the backlog, address it in dedicated refactor sprints rather than letting it accumulate.

## Anemic vs Rich Domain Model

**Anemic domain model** — entities are plain data bags (just properties, no behaviour). All business logic lives in service classes.

```csharp
// Anemic — entity has no behaviour
public class Tournament { public TournamentStatus Status { get; set; } }

// Logic scattered in a service
public void StartTournament(Tournament t) {
    if (t.Status != TournamentStatus.Draft) throw new InvalidOperationException();
    t.Status = TournamentStatus.Active;
}
```

**Rich domain model** — entities own and enforce their own rules. The object knows how to transition itself.

```csharp
// Rich — entity protects its own invariants
public class Tournament {
    public TournamentStatus Status { get; private set; }

    public void Start() {
        if (Status != TournamentStatus.Draft)
            throw new InvalidOperationException("Only draft tournaments can be started.");
        Status = TournamentStatus.Active;
    }
}
```

**In practice:** most .NET shops use anemic models, especially with EF Core (which prefers simple entities it can track). Neither is universally right — the honest answer is that anemic is simpler and more common, rich is more expressive for complex domains.

**Interview trap:**

- _"Is anemic domain model an anti-pattern?"_ — Technically yes (Fowler called it that), but it's the default in most .NET EF Core projects for practical reasons. The right answer is: know the tradeoff, don't be dogmatic.

## DDD Awareness

Domain-Driven Design is a philosophy for modelling complex business domains. You don't need to implement full DDD as a junior, but the vocabulary is everywhere in senior conversations.

**Entity** — an object with a unique identity that persists over time. A `Tournament` with an `Id` is an entity — it's still "the same tournament" even if its name changes.

**Value Object** — an object defined entirely by its values, no identity. An `Address` or `Money(100, "USD")` — two Money objects with the same amount and currency are equal. Immutable.

**Aggregate** — a cluster of entities and value objects treated as a single unit for data changes. One root entity (the Aggregate Root) is the only entry point. Example: `Tournament` is the root, `TournamentTeam` registrations are part of the aggregate — you always access teams through the tournament, never directly.

**Ubiquitous Language** — use the same terms in code as the business uses. If the business says "registration closes", your method is `CloseRegistration()`, not `UpdateStatusToRegistrationClosed()`.

Awareness level: know these four terms, be able to define them if asked. You don't need to have implemented full DDD.

## SQL vs NoSQL

**Relational (SQL)** — structured schema, strong consistency, ACID transactions, powerful joins. Use when your data has clear relationships and integrity matters.

**NoSQL** — flexible schema, horizontal scaling, various models (document, key-value, graph, column). Use when your data is unstructured, you need massive scale, or your access patterns are simple and fast.

||SQL (e.g. SQL Server, PostgreSQL)|NoSQL (e.g. MongoDB, Redis)|
|---|---|---|
|Schema|Fixed, enforced|Flexible|
|Transactions|Full ACID|Limited (varies)|
|Joins|Native|Not supported (denormalize instead)|
|Scale|Vertical (mostly)|Horizontal|
|Best for|Relational data, complex queries|High-volume, simple lookups, caching|

**Interview trap:**

- _"When would you use NoSQL over SQL?"_ — When you need extreme horizontal scale, your data is document-like with no strong relational structure, or for specific use cases like caching (Redis), session storage, or real-time leaderboards. Not because NoSQL is "faster" — that's a myth without context.

## Caching

Caching stores results of expensive operations so repeat requests are served without recomputing.

**Cache-aside pattern** — the application manages the cache manually:

1. Check cache → hit? Return it.
2. Miss? Fetch from DB, write to cache, return.

```csharp
var tournament = await _cache.GetAsync<TournamentDto>($"tournament:{id}");
if (tournament is null)
{
    tournament = await _repo.GetByIdAsync(id);
    await _cache.SetAsync($"tournament:{id}", tournament, TimeSpan.FromMinutes(5));
}
return tournament;
```

**In-memory cache** (`IMemoryCache`) — lives in the process. Fast, but lost on restart and not shared across multiple app instances.

**Distributed cache** (`IDistributedCache`, Redis) — external store shared across all instances. Slower than in-memory but consistent across a scaled-out app.

**Interview trap:**

- _"What is cache invalidation?"_ — The hard part of caching. When source data changes, the cache entry becomes stale. Strategies: TTL expiry (cache expires after N minutes), explicit invalidation (delete the key when data changes), or write-through (update cache and DB together). There's no perfect solution — it's a tradeoff between freshness and performance.

## Message Brokers (Awareness)

A message broker (Kafka, RabbitMQ, Azure Service Bus) decouples services by passing messages asynchronously through a queue or topic.

Producer publishes a message → broker holds it → consumer reads and processes it at its own pace.

**Why:** the producer doesn't wait for the consumer. If the consumer is slow or down, messages queue up. You get resilience, decoupling, and the ability to scale consumers independently.

**In your TO2:** your Kafka consumer (`BackgroundService`) reads match result events from a topic and processes standings updates asynchronously. The match service doesn't wait for standings to update — it just fires the event and moves on.

Awareness level: know what a broker does, know the producer/consumer model, be able to reference your Kafka usage.

## Azure Basics (Awareness)

|Service|What it is|Use for|
|---|---|---|
|**App Service**|Managed hosting for web apps|Deploy ASP.NET Core APIs without managing VMs|
|**Azure SQL**|Managed SQL Server|Production relational database|
|**Blob Storage**|Object/file storage|Images, documents, exports, backups|
|**Key Vault**|Secrets management|Store connection strings, API keys — never in appsettings|
|**Application Insights**|Monitoring + logging|Distributed tracing, performance metrics, error tracking|

**Interview trap:**

- _"Where do you store secrets in production?"_ — Azure Key Vault (or equivalent). Never in `appsettings.json`, never in source control. The app reads secrets from the vault at runtime via managed identity — no credentials needed in the app config at all.
